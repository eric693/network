Section 01

IPv4 位址結構

IPv4 位址是一個 32 位元（4 bytes）的二進位數字， 代表網路中某個介面的邏輯識別碼。 為了方便人類閱讀，我們將這 32 個位元拆成四組各 8 位元（稱為八位元組 / Octet）， 每組以十進位表示，組間用點分隔，稱為點十進位表示法（Dotted Decimal Notation）。

每個八位元組的值域是 0–255（2⁸ = 256 種狀態）。 32 位元的位址空間共可表示 2³² = 4,294,967,296 個位址， 約 43 億個，而實際可用數量因各種保留位址而更少。

IPv4 位址本質上由兩部分組成： 網路部分（Network ID）識別所屬網路， 主機部分（Host ID）識別網路中的特定設備。 哪些位元屬於網路部分、哪些屬於主機部分，由子網路遮罩決定。

Section 02

位址分類（Classful Addressing）

早期（1981–1993 年）網際網路使用有類別位址（Classful Addressing）， 根據 IP 位址的最高幾個位元自動決定網路與主機的分界。 雖然今日已被 CIDR 取代，理解分類制度對讀懂歷史文件與設備設定仍有幫助。

為何 127.x.x.x 不在 Class A？

127.0.0.0/8 整個區塊被保留為迴路位址（Loopback）， 最常用的是 127.0.0.1（也稱為 localhost）。 送到這個位址的封包不會離開主機，直接被作業系統的網路堆疊回送給發送者， 常用於本機服務測試。

有類別位址的致命缺陷是位址浪費： 一間只有 300 台電腦的公司申請 Class B 網路， 拿到 65,534 個位址卻只用 300 個， 超過 99% 的位址空白浪費。這是推動 CIDR 的根本原因。

計算網路位址的方式是對 IP 位址與子網路遮罩進行 位元 AND 運算（Bitwise AND）： 兩個位元都是 1 時結果為 1，否則為 0。 AND 運算的結果就是這個 IP 所屬的網路位址（Network Address）。

Section 04

CIDR：無類別網域間路由

1993 年，IETF 提出 CIDR（Classless Inter-Domain Routing，RFC 4632）， 以更靈活的方式分配位址。CIDR 廢除了 Class A/B/C 的固定邊界， 改以前綴長度（Prefix Length）自由指定網路位元數， 用斜線後接數字的方式表達，稱為CIDR 表示法， 如 192.168.1.0/24，其中 /24 表示網路部分有 24 個位元。

CIDR 帶來的兩大革命

路由聚合（Route Aggregation / Supernetting）： 多個連續的小網路可以合併成一條路由條目，大幅縮減骨幹路由器的路由表大小。 例如 200.1.0.0/24、200.1.1.0/24、 200.1.2.0/24、200.1.3.0/24 可以聚合為單條 200.1.0.0/22。 精準分配：按需求分配剛好大小的位址塊， 而非強制使用 Class A/B/C 的固定大小，大幅減少浪費。

Section 06

特殊用途 IPv4 位址

IANA 保留了一些位址範圍用於特定目的，這些位址不會在公用網際網路上路由。 在設計網路或排除故障時，遇到這些位址需要特別注意。

位址範圍 CIDR 用途 RFC
0.0.0.0 /8 「本網路」預設路由或未指定來源（如 DHCP 請求時的來源） RFC 1122
10.0.0.0 /8 私有位址（Class A 私有），企業內網常用 RFC 1918
100.64.0.0 /10 共享位址空間（CGN / Carrier-Grade NAT），ISP 內部使用 RFC 6598
127.0.0.0 /8 本地迴路（Loopback），127.0.0.1 = localhost RFC 1122
169.254.0.0 /16 APIPA（Automatic Private IP Addressing），DHCP 失敗時自動指派 RFC 3927
172.16.0.0 /12 私有位址（Class B 私有），172.16.0.0 – 172.31.255.255 RFC 1918
192.0.2.0 /24 文件與範例用，永不在實際網路中路由（TEST-NET-1） RFC 5737
192.168.0.0 /16 私有位址（Class C 私有），家用路由器最常見 RFC 1918
198.18.0.0 /15 效能測試專用位址，不可路由 RFC 2544
224.0.0.0 /4 多播（Multicast）位址範圍 RFC 5771
255.255.255.255 /32 有限廣播位址，只在本地子網路廣播，路由器不轉發 RFC 919

169.254.x.x 的診斷意義

當你在 Windows 上執行 ipconfig 看到 169.254.x.x， 表示該電腦無法從 DHCP 伺服器獲取位址，改用 APIPA 自動配置。 這幾乎總是意味著：網路線未插好、Wi-Fi 未連接，或 DHCP 伺服器出問題。

Section 07

私有位址與 NAT

RFC 1918 定義了三個私有位址範圍，這些位址在公用網際網路上不可路由， 任何組織都可以在內部網路自由使用而無需向 IANA 申請。

10.0.0.0 – 10.255.255.255
10.0.0.0 /8
單一大型私有網路，提供超過 1600 萬個位址。 適用於大型企業或雲端服務內部 VPC 規劃。
172.16.0.0 – 172.31.255.255
172.16.0.0 /12
中型私有網路，提供超過 100 萬個位址。 Docker 預設的橋接網路即使用此範圍。
192.168.0.0 – 192.168.255.255
192.168.0.0 /16
最廣為人知的私有範圍，提供 65,536 個位址。 家用路由器幾乎全部預設使用 192.168.0.x 或 192.168.1.x。

NAT（Network Address Translation）

由於私有位址無法直接在網際網路上路由，設備上網時需要 NAT（網路位址轉換）將私有位址轉換為公有位址。 這個工作通常由家用路由器或企業防火牆執行。

最常見的是 NAPT（Network Address and Port Translation）， 又稱為 PAT（Port Address Translation） 或俗稱「IP 偽裝（Masquerading）」。 它讓多台私有位址的設備共用一個公有 IP 位址上網， 透過不同的來源連接埠號加以區分。

NAPT 運作示意（多台設備共用一個公有 IP）
內部網路（私有位址）              NAT 路由器               網際網路
192.168.1.10:54321  ──────────>  公有 IP：203.0.113.1     目標：8.8.8.8:53
192.168.1.11:54322  ──────────>  NAT 轉換表：
192.168.1.12:54323  ──────────>
                                  私有 IP:Port         ↔  公有 IP:Port
                                  192.168.1.10:54321  ↔  203.0.113.1:40001
                                  192.168.1.11:54322  ↔  203.0.113.1:40002
                                  192.168.1.12:54323  ↔  203.0.113.1:40003
                                            ↓
                                  送往 8.8.8.8 的封包來源為 203.0.113.1
                                  回應封包依連接埠號轉發給正確的內部設備

NAT 的缺點

NAT 破壞了 IP 設計中「端對端可定址性」的原則：外部設備無法主動連線到 NAT 後的設備， 必須使用連接埠轉發（Port Forwarding）才能讓外部連入。 這讓 P2P 應用、VoIP、遊戲伺服器的設定複雜化，也是 IPv6 普及後設計上試圖淘汰 NAT 的原因。

Section 08

IPv6 位址詳解

IPv6 使用 128 位元的位址空間，以冒號十六進位表示， 每 16 位元一組，共 8 組，組間以冒號 : 分隔。

完整 IPv6 位址
2001:0db8:85a3:0000:0000:8a2e:0370:7334

IPv6 縮寫規則

完整的 IPv6 位址很長，實際使用時遵循兩條縮寫規則使其更易讀：

規則 說明 範例
省略前導零 每組的前導零可省略（至少保留一個數字） 0db8 → db8，0000 → 0
雙冒號縮寫 一段（或多段）連續的全零組可用 :: 取代，全址只能用一次 ::0000:0000:: 只能寫一個 ::
縮寫過程
完整：2001:0db8:85a3:0000:0000:8a2e:0370:7334
省略前導零：2001:db8:85a3:0:0:8a2e:370:7334
雙冒號縮寫：2001:db8:85a3::8a2e:370:7334

IPv6 位址類型

類型 前綴 說明
全球單播（GUA） 2000::/3 類似 IPv4 公有位址，可在網際網路上路由，目前主要分配 2001::/32 至 2620::/23
連結本地（LLA） fe80::/10 只在本地連結有效，不可路由。介面啟用後自動配置，用於鄰居發現（NDP）
唯一本地（ULA） fc00::/7 類似 IPv4 私有位址，僅限組織內部，不可在公網路由
多播（Multicast） ff00::/8 取代 IPv4 的廣播，所有廣播功能由多播實現，如 ff02::1 = 所有節點
迴路（Loopback） ::1/128 等同 IPv4 的 127.0.0.1，共 128 個位元，全為 0 只除最後一個 1
未指定 ::/128 全為零，等同 IPv4 的 0.0.0.0，表示「任意位址」

SLAAC：IPv6 的自動位址配置

IPv6 無需 DHCP 也能自動配置位址，機制稱為 SLAAC（Stateless Address Autoconfiguration，RFC 4862）。 主機透過路由器通告（Router Advertisement）取得網路前綴， 再結合自身的介面識別碼（從 MAC 位址衍生的 EUI-64 或隨機值）生成完整位址。 這大幅簡化了大規模網路的位址管理。

Section 09

VLSM：可變長度子網路遮罩

固定長度的子網路切割（所有子網路大小相同）在實際網路規劃中非常浪費。 VLSM（Variable Length Subnet Mask）允許同一個主要網路 被切割成大小不同的子網路，讓每個區段拿到剛好夠用的位址數量。

VLSM 規劃範例：公司總部網路 172.16.0.0/16
需求：
  - 研發部：500 台主機
  - 業務部：100 台主機
  - 行政部：50 台主機
  - IT 部：25 台主機
  - 路由器點對點連線：4 條（各需 2 台）

VLSM 分配（由大到小分配，避免浪費）：

  研發部    172.16.0.0  /23   → 可用 510 台  ✓ (2⁹−2 = 510)
  業務部    172.16.2.0  /25   → 可用 126 台  ✓ (2⁷−2 = 126)
  行政部    172.16.2.128/26   → 可用  62 台  ✓ (2⁶−2 = 62)
  IT 部     172.16.2.192/27   → 可用  30 台  ✓ (2⁵−2 = 30)
  P2P 連線1 172.16.2.224/30   → 可用   2 台  ✓
  P2P 連線2 172.16.2.228/30   → 可用   2 台  ✓
  P2P 連線3 172.16.2.232/30   → 可用   2 台  ✓
  P2P 連線4 172.16.2.236/30   → 可用   2 台  ✓

  剩餘空間：172.16.2.240/28 及後續，供未來擴充

VLSM 規劃原則

始終從最大需求的子網路開始分配，依序遞減。 每個子網路的起始位址必須是其區塊大小的整數倍（稱為位元邊界對齊）， 否則計算會出錯。例如 /26（64 個位址）的網路位址必須是 64 的倍數： 0、64、128、192。

Day 4 重點回顧

今天你學到了什麼

  • IPv4 是 32 位元，點十進位表示，分為網路部分與主機部分
  • Classful 按首位元自動分界（A/B/C），因位址浪費嚴重於 1993 年被 CIDR 取代
  • 子網路遮罩 AND 運算取網路位址；主機部分全 1 為廣播位址；可用數 = 2ⁿ − 2
  • CIDR 用 /前綴長度 靈活劃分，支援路由聚合，大幅縮減路由表
  • 三個 RFC 1918 私有範圍：10/8、172.16/12、192.168/16，不可在公網路由
  • NAPT 讓多台設備共用一個公有 IP，透過來源連接埠區分連線
  • IPv6 128 位元，冒號十六進位；:: 縮寫連續零組；無廣播，改用多播
  • VLSM 允許同一網路切成不同大小子網路；規劃時由大到小、注意位元邊界對齊
