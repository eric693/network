Section 01

傳統網路架構的挑戰

在傳統網路架構中，每台路由器和交換器都是一個自治的封閉系統—— 硬體、作業系統、轉發邏輯和管理介面全部整合在一起， 由各家廠商（Cisco、Juniper、Arista）各自實作、各自封閉。 這帶來了幾個根本性的挑戰：

挑戰說明
垂直整合，廠商鎖定硬體與軟體由同一廠商捆綁銷售，切換廠商成本極高，創新受制於廠商的產品週期
分散式管理，難以協調每台設備獨立執行路由協定（OSPF/BGP），網路策略需逐台設備手動配置，人工操作容易出錯
靜態配置，無法快速響應流量工程（Traffic Engineering）需要提前規劃，無法根據即時流量動態調整轉發路徑
難以程式化CLI 或 SNMP 是主要管理介面，缺乏標準的程式化 API，自動化部署困難
資料中心擴展瓶頸雲端資料中心需要數萬個租戶的網路隔離，傳統 VLAN 只有 4,094 個 ID，完全不夠用

2008 年，Stanford 大學的 Nick McKeown 和 Scott Shenker 在論文《OpenFlow: Enabling Innovation in Campus Networks》中 提出了「將控制平面與資料平面分離」的思想， 奠定了 SDN 的理論基礎，掀起了網路架構的革命。

Section 02

SDN 三層架構

SDN（Software-Defined Networking，軟體定義網路） 的核心思想是將控制平面（Control Plane）從資料平面（Data Plane）中分離出來， 由集中式的軟體控制器統一管理所有設備的轉發決策。

▲
應用層（Application Plane）
網路應用程式透過北向 API（Northbound API）告訴控制器需要什麼網路行為。應用程式只需描述意圖，不需知道底層如何實現。
負載均衡器、防火牆策略、流量工程應用、網路監控應用
●
控制層（Control Plane）
SDN 控制器（Controller）是整個網路的「大腦」。維護全域網路拓撲視圖，計算最優轉發路徑，將規則下發到各交換器。透過南向 API（如 OpenFlow）控制硬體。
OpenDaylight、ONOS、NOX、Floodlight、Ryu
▼
資料層（Data Plane）
實體或虛擬的交換器只需按照控制器下發的流表（Flow Table）轉發封包，不做任何路由決策。設備可以是白牌交換器（White-box Switch），大幅降低成本。
OpenFlow 交換器、Open vSwitch（OVS）、P4 可程式化 ASIC
傳統網路 vs SDN 架構對比

北向 API 與南向 API

API 方向連接對象代表協定/介面
南向 API（Southbound） 控制器 ↔ 網路設備（交換器、路由器） OpenFlow、NETCONF/YANG、gRPC/gNMI、OVSDB、P4Runtime
北向 API（Northbound） 控制器 ↔ 應用程式 REST API、Python SDK、Intent-based API（如 OpenDaylight MD-SAL）
東西向 API（East-West） 控制器 ↔ 控制器（多控制器叢集） OpenDaylight Clustering、ONOS Cluster Consensus
Section 03

OpenFlow 協定

OpenFlow（目前最新版本 1.5）是 SDN 南向 API 的開創性標準， 定義了控制器如何向交換器下達轉發規則。 OpenFlow 的核心概念是流表（Flow Table）—— 一張「如果封包符合這些條件，就執行這些動作」的規則表。

流表條目結構

OpenFlow 流表條目（Flow Table Entry）
  ┌─────────────────┬──────────────┬──────────────────────────────────────┐
  │  Match Fields   │   Priority   │              Actions                  │
  │  （匹配欄位）    │  （優先順序）  │             （執行動作）               │
  ├─────────────────┼──────────────┼──────────────────────────────────────┤
  │ 可匹配的欄位：  │              │ 可執行的動作：                         │
  │ • 入埠號        │  數字越大     │ • OUTPUT(port)  → 從指定埠轉發         │
  │ • 來源 MAC      │  優先級越高   │ • DROP          → 丟棄封包             │
  │ • 目的 MAC      │              │ • FLOOD         → 洪泛到所有埠         │
  │ • VLAN ID       │  範圍：      │ • PUSH/POP VLAN → 新增/移除 VLAN 標籤  │
  │ • EtherType     │  0–65535     │ • SET_FIELD      → 修改封包欄位         │
  │ • 來源/目的 IP  │              │ • GOTO_TABLE     → 跳到另一張流表       │
  │ • Protocol      │              │ • METER          → 速率限制             │
  │ • TCP/UDP Port  │              │ 若無匹配：送往控制器（Packet-In）        │
  └─────────────────┴──────────────┴──────────────────────────────────────┘

  範例：將所有 HTTP 流量轉發到特定的 IDS 監控埠
  Match: tcp.dst_port == 80  → Action: OUTPUT(monitor_port), OUTPUT(dst_port)

Open vSwitch（OVS）

Open vSwitch（OVS）是最廣泛使用的開源虛擬交換器， 實作了 OpenFlow 協定，運行在 Linux 核心中， 是 OpenStack、KVM、VMware NSX 等雲端平台的核心網路元件。 在單一實體主機上，OVS 可以連接數百個虛擬機的虛擬網路介面（vNIC）， 提供 VLAN 分隔、隧道（VXLAN/GRE）、QoS、流量鏡像等功能。

OVS 常用指令Open vSwitch
# 查看 OVS 交換器和埠
$ ovs-vsctl show

# 新增交換器和埠
$ ovs-vsctl add-br br0
$ ovs-vsctl add-port br0 eth0

# 查看流表
$ ovs-ofctl dump-flows br0

# 手動添加流表規則
$ ovs-ofctl add-flow br0 "priority=100,tcp,tp_dst=80,actions=output:2"

# 新增 VXLAN 隧道埠（連接另一台主機）
$ ovs-vsctl add-port br0 vxlan0 -- set interface vxlan0 \
    type=vxlan options:remote_ip=10.0.0.2 options:key=100
Section 04

NFV：網路功能虛擬化

NFV（Network Functions Virtualization，網路功能虛擬化） 與 SDN 是互補但不同的概念： SDN 分離控制平面與資料平面； NFV 則是將傳統需要專有硬體的網路功能（防火牆、負載均衡器、WAN 加速器） 以軟體形式（VNF，Virtual Network Function）運行在通用伺服器上。

傳統硬體設備 vs NFV 虛擬化
傳統方式
專有硬體 + 封閉軟體
購買 Cisco ASA（防火牆）、F5 BIG-IP（負載均衡）、Riverbed（WAN 加速）等專用硬體設備，各自需要獨立部署、維護、升級。採購週期長（數週到數月），擴展需要購買新設備。
NFV 方式
軟體 VNF 運行在通用 x86 伺服器
同樣的功能以虛擬機或容器形式運行：Palo Alto VM-Series（虛擬防火牆）、HAProxy/Nginx（軟體負載均衡）、Viptela vEdge（虛擬路由器）。分鐘級部署，彈性擴縮容，降低 CAPEX。
NFV 元件角色代表產品
VNF（Virtual Network Function） 虛擬化的網路功能本身，如虛擬防火牆、虛擬路由器 Fortinet FortiGate-VM、Cisco CSR 1000v、Juniper vSRX
NFVI（Infrastructure） 運行 VNF 的基礎設施層，包含計算、儲存、網路資源 OpenStack、VMware vSphere、KVM + OVS
MANO（Management & Orchestration） VNF 的生命週期管理：部署、擴縮容、故障恢復、計費 OpenStack Tacker、ONAP、OSM（Open Source MANO）

SDN 與 NFV 的關係

SDN 和 NFV 各自獨立，但常常協同部署： NFV 讓網路功能成為軟體，可彈性部署； SDN 讓控制這些功能的網路也可程式化。 兩者結合是電信商 5G 核心網（5G Core）和雲端資料中心的基礎架構。 不過 NFV 不一定需要 SDN（軟體防火牆可以在傳統網路中運行）， SDN 也不一定需要 NFV（白牌交換器就是硬體設備）。

Section 05

Overlay 網路技術

Overlay 網路是在現有（Underlay）網路之上 建立的虛擬網路層，透過封裝（Encapsulation）技術 在實體網路上「疊加」一層邏輯網路。 虛擬機或容器可以橫跨多台實體主機， 彷彿在同一個二層網路中，而實體網路只需支援 IP 路由即可。

VXLAN
Virtual eXtensible LAN — RFC 7348
將 L2 訊框封裝在 UDP 中傳輸，支援 16M（2²⁴）個虛擬網路 ID（VNI）。雲端資料中心標準，Kubernetes、OpenStack、VMware NSX 廣泛使用
GENEVE
Generic Network Virtualization Encapsulation
VXLAN 的後繼者（RFC 8926），支援可擴展的 Option 欄位，允許攜帶額外 Metadata。OVN（Open Virtual Network）、AWS 內部使用
GRE
Generic Routing Encapsulation — RFC 2784
最通用的封裝協定，可封裝任何 L3 協定。Protocol 47，無 UDP 開銷。傳統企業 VPN 和點對點隧道的標準選擇
STT
Stateless Transport Tunneling
VMware Nicira 提出，利用 TCP 標頭格式讓 NIC 的 TSO（TCP Segmentation Offload）硬體加速隧道封包分割，提高效能。已被 GENEVE 取代趨勢
WireGuard
現代 VPN / Overlay
雖主要是 VPN 協定，也廣泛用於構建安全的點對點 Overlay 網路，尤其在 Kubernetes（Flannel WireGuard 模式）和邊緣計算中
MPLS
Multi-Protocol Label Switching
電信骨幹網的標準 Overlay，以標籤（Label）代替 IP 路由決策，實現確定性轉發、QoS 和流量工程。MPLS VPN 是企業廣域網的主流
Section 06

VXLAN 深度解析

VXLAN（Virtual eXtensible LAN）是當前雲端資料中心最重要的 Overlay 技術。 傳統 VLAN 只有 12 bits（4,094 個網段）， 遠不夠大型多租戶雲端平台使用； VXLAN 用 24 bits 的 VNI（VXLAN Network Identifier） 支援超過 1,600 萬個虛擬網段，徹底解決了擴展瓶頸。

VXLAN 封裝格式

VXLAN 封包結構（共新增 50 bytes 額外開銷）
  原始 VM 封包（L2 訊框）：
  ┌──────────┬──────────┬──────────────────────────┐
  │ Dst MAC  │ Src MAC  │  IP + TCP + Application  │
  └──────────┴──────────┴──────────────────────────┘

  VXLAN 封裝後在實體網路傳輸：
  ┌────────────┬───────────────────────────────────────────────────────────┐
  │外部 IP Hdr │ 外部 UDP Hdr │ VXLAN Header  │    原始 L2 訊框（加密封裝）  │
  │Src: VTEP_A │ Dst Port:   │ ┌───────────┐ │ ┌──────┬──────┬──────────┐ │
  │Dst: VTEP_B │ 4789（UDP） │ │  VNI(24b) │ │ │DstMAC│SrcMAC│IP+TCP+Data│ │
  │(實體 IP)   │             │ │  Flags(8b)│ │ └──────┴──────┴──────────┘ │
  │            │             │ └───────────┘ │                             │
  └────────────┴─────────────┴───────────────┴─────────────────────────────┘
  ← 外部 IP Header（20b）→← UDP（8b）→← VXLAN（8b）→← 原始 L2 訊框 →

  VTEP（VXLAN Tunnel End Point）：負責封裝和解封裝的端點，可以是主機的軟體（OVS）或支援 VXLAN 的硬體交換器

VTEP 學習機制

VXLAN 需要知道「哪個 VM 的 MAC 位址在哪個 VTEP 後面」， 類似傳統交換器的 MAC 學習，但跨越了實體網路邊界。 有兩種主要學習方式：

方式說明適用場景
Data Plane Learning（洪泛+學習） 未知目的 MAC 的訊框透過 Multicast 群組洪泛到所有 VTEP，VTEP 透過回應封包學習 MAC-to-VTEP 映射。類似傳統交換器學習，不需要集中控制 小型部署、不想引入控制平面複雜性的場景
Control Plane Learning（集中分發） VM 的 MAC/IP 資訊由集中式系統（如 Kubernetes CNI、OpenStack Neutron、VMware NSX Controller）透過 BGP EVPN 或自訂協定分發給所有 VTEP。消除洪泛 大型雲端平台，Kubernetes、OpenStack 標準做法
Section 07

容器網路（CNI）

容器的快速啟停和大量部署對網路帶來了獨特的挑戰： 每個 Pod（Kubernetes 的基本調度單位）都需要一個 IP， Pod 之間需要互通，同時需要服務發現與負載均衡。 CNI（Container Network Interface）是 Kubernetes 定義的容器網路插件標準，允許選用不同的網路實作。

Kubernetes 網路模型四原則

原則說明
Pod 間不用 NAT 通訊每個 Pod 有唯一的 IP，Pod 之間直接用 IP 通訊，不需要網路地址轉換
Node 到 Pod 不用 NATNode 可以直接用 Pod IP 存取任何 Pod，反之亦然
Pod 看到的 IP = 外部看到的 IPPod 的 IP 不因 NAT 而改變，簡化應用程式的網路邏輯
Service 提供穩定端點Pod IP 會隨重啟改變，Service 提供穩定的虛擬 IP（ClusterIP）和 DNS 名稱

eBPF：下一代網路加速技術

eBPF（extended Berkeley Packet Filter） 允許在 Linux 核心中安全地執行自訂程式，無需修改核心程式碼或載入核心模組。 在網路領域，eBPF 可以在封包進入 / 離開網路介面時攔截並處理， 完全繞過傳統的 iptables 和 netfilter 路徑。

Cilium 和 Pixie 等工具使用 eBPF 實現： 傳統 iptables 需要 O(n) 線性掃描規則， 而 eBPF 的 Map 結構是 O(1) 雜湊查找—— 在有數萬條規則的大型叢集中，eBPF 的效能優勢可達到 10 倍以上。 eBPF 也是現代可觀測性（如 Hubble）和安全策略執行的基礎。

Section 08

服務網格（Service Mesh）

微服務架構中，服務間通訊變得複雜： 服務發現、負載均衡、熔斷（Circuit Breaking）、 重試、逾時、mTLS 加密、分散式追蹤—— 每個服務都需要實作這些能力，造成大量重複程式碼。 服務網格（Service Mesh）將這些網路關注點 從應用程式中抽離，下沉到基礎設施層。

服務網格 Sidecar 代理模式
  傳統微服務：每個服務自己處理網路邏輯
  ┌────────────────┐         ┌────────────────┐
  │  Service A     │ ──────> │  Service B     │
  │  (含 LB/重試/  │         │  (含 LB/重試/  │
  │   mTLS/追蹤)   │         │   mTLS/追蹤)   │
  └────────────────┘         └────────────────┘

  Service Mesh：Sidecar 代理接管所有網路流量
  ┌────────────────────────────────────────────────────────────┐
  │  Pod A                         Pod B                       │
  │  ┌──────────────┐             ┌──────────────┐             │
  │  │  Service A   │             │  Service B   │             │
  │  │  (只寫業務)  │             │  (只寫業務)  │             │
  │  └──────┬───────┘             └──────┬───────┘             │
  │         │ localhost                  │ localhost            │
  │  ┌──────┴───────┐  mTLS/HTTP2 ┌─────┴────────┐            │
  │  │ Envoy Proxy  │ ──────────> │ Envoy Proxy  │            │
  │  │  (Sidecar)   │             │  (Sidecar)   │            │
  │  └──────────────┘             └──────────────┘            │
  └────────────────────────────────────────────────────────────┘
              ↑                          ↑
         所有流量由 Sidecar 代理，應用程式感知不到

  Control Plane（如 Istio istiod）：
    • 分發服務發現資訊（Pilot）
    • 分發 mTLS 憑證（Citadel）
    • 收集遙測數據（Telemetry）
Service Mesh 功能說明
mTLS（雙向 TLS）所有服務間通訊自動加密並相互認證，實現零信任網路，無需改應用程式程式碼
流量管理精細的流量路由（Canary Release、A/B Testing）、重試、逾時、熔斷器、故障注入
可觀測性自動收集所有服務間流量的 Metrics、Logs、Traces，無需應用程式埋點
服務發現自動發現叢集中的服務端點，維護健康的端點列表，做客戶端負載均衡

主流 Service Mesh：Istio（最完整，以 Envoy 為 Sidecar，Google/IBM/Lyft 開發）、 Linkerd（輕量，Rust 實作的 Sidecar，效能開銷低）、 Consul Connect（HashiCorp，整合服務發現與 KV Store）、 Cilium Service Mesh（eBPF 實作，無需 Sidecar，效能最佳）。

Section 09

SD-WAN：廣域網路的 SDN

傳統企業廣域網路（WAN）依賴 MPLS 電路—— 費用高昂、部署週期長（數週到數月）、缺乏彈性。 SD-WAN（Software-Defined WAN）將 SDN 的理念應用到廣域網路， 透過集中化的控制平面，在低成本的寬頻網路（Internet VPN）上 實現智慧流量路由和動態路徑選擇，同時保留 MPLS 的品質保障。

SD-WAN 的安全整合：SASE

SASE（Secure Access Service Edge）由 Gartner 於 2019 年提出， 將 SD-WAN 與安全服務（CASB、ZTNA、SWG、FWaaS）整合到雲端邊緣節點， 讓使用者的流量在最近的雲端節點就完成安全檢查和路由決策， 不需要再回傳到公司總部的安全閘道。 Cloudflare One、Zscaler、Palo Alto Prisma Access 是主要 SASE 提供商。

Day 13 重點回顧

今天你學到了什麼

  • 傳統網路的問題：垂直整合、分散管理、靜態配置、VLAN 數量不足
  • SDN 三層：應用層（業務意圖）→ 控制層（集中決策）→ 資料層（轉發執行），北向/南向/東西向 API
  • OpenFlow 流表：Match Fields + Priority + Actions，封包無匹配時送往控制器（Packet-In）
  • OVS（Open vSwitch）：最廣泛的虛擬交換器，支援 OpenFlow、VXLAN、GRE，是 OpenStack/KVM 基礎
  • NFV 將網路功能從專有硬體解放到通用伺服器上，VNF + NFVI + MANO 三元件
  • VXLAN 用 24 bits VNI 支援 16M 虛擬網段，解決 4,094 VLAN 的限制；VTEP 負責封裝/解封裝
  • Kubernetes CNI 四原則：Pod 間不用 NAT；Calico（BGP）/ Cilium（eBPF）是生產主流
  • eBPF 讓核心可程式化，Cilium 用 O(1) Map 取代 iptables O(n) 線性掃描，效能提升 10×
  • Service Mesh（Istio/Linkerd）：Sidecar 代理接管服務間流量，提供 mTLS/流量管理/可觀測性
  • SD-WAN 集中控制多條 WAN 路徑（MPLS + 寬頻 + 4G），動態選路、Zero-touch 部署
