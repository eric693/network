Phase 02 — 網路層與傳輸層 — Day 08 / 16

UDP 與多播技術
為速度而生的輕量傳輸

並非所有應用都需要 TCP 的可靠性保證——有時，速度與低延遲比完整性更重要。 UDP 以極簡設計換取極低開銷，是 DNS、影音串流、線上遊戲的基石。 多播技術則讓一份資料同時高效送達多個接收者，是 IPTV 與路由協定的核心。

Section 01

UDP 的設計哲學

UDP（User Datagram Protocol）定義於 RFC 768（1980 年）， 整份規格文件只有 3 頁。它的核心哲學是 「只做必要的事，其他都交給應用層」： 透過連接埠號實現多工、提供可選的 Checksum，其餘的可靠性、 排序、流量控制、壅塞控制全部捨棄。

優勢說明
極低延遲無需三向交握，第一個封包就是資料，對 DNS 等毫秒級操作至關重要
極低開銷Header 僅 8 bytes（TCP 是 20 bytes），大量小封包的傳輸效率顯著更高
支援廣播 / 多播TCP 是連線導向的一對一協定，不支援一對多；UDP 天然支援廣播與多播
無壅塞控制限制應用程式可自訂傳送速率，適合固定碼率（CBR）的媒體串流
無狀態伺服器無需維護連線狀態，可輕鬆擴展至數百萬並發（如 DNS、DHCP）

UDP 不等於「不可靠」

UDP 本身不提供可靠性，但應用程式可以在 UDP 之上自行實作所需的可靠性機制，只包含自己需要的部分，而不是 TCP 的全套機制。QUIC 在 UDP 上實作了選擇性重傳卻保留 0-RTT 握手；遊戲引擎可能只實作「最新狀態覆蓋舊狀態」的低延遲模式。這種彈性是 UDP 超越 TCP 限制的核心價值。

Section 02

UDP 標頭結構

UDP 標頭固定 8 bytes，四個欄位各 16 bits，是傳輸層協定中最精簡的結構。

UDP Header（固定 8 bytes = 4 × 16 bits）
Source Port（來源連接埠） 選填；若不需要回應可設為 0 16 bits（0–65535）
Destination Port（目的連接埠） 必填；識別目的服務（如 53 = DNS） 16 bits（0–65535）
Length（長度） UDP Header + Data 總長度（bytes），最小值為 8 16 bits（最大 65,535 bytes）
Checksum（校驗和） IPv4 中選填；IPv6 中強制。設為 0 表示不使用 16 bits
Data（資料 Payload） 應用層資料，保留訊息邊界（不同於 TCP 的位元組串流） 0–65,507 bytes（65,535 − 8 UDP Header − 20 IP Header）

UDP 的 Checksum 在 IPv4 中是選填的（設為 0 代表不使用）， 在 IPv6 中是強制的（因 IPv6 Header 本身無 Checksum）。 即便停用，底層乙太網路的 FCS 仍會偵測傳輸錯誤， 因此在可靠的有線網路中停用能輕微提升效能。

Section 03

TCP vs UDP 深度比較

TCP — 傳輸控制協定
連線連線導向，三向交握（1 RTT 開銷）
可靠性保證交付：ACK + 逾時重傳
順序序號確保資料按序交付
流量控制rwnd 接收視窗防止接收方溢出
壅塞控制cwnd，Slow Start / CUBIC / BBR
Header20–60 bytes（含選項）
傳輸模式位元組串流（無訊息邊界）
廣播/多播不支援
適用HTTP/S、SSH、FTP、電子郵件
UDP — 使用者資料報協定
連線無連線，直接傳送（0 RTT 開銷）
可靠性不保證，由應用層決定是否實作
順序不保證，封包可能亂序抵達
流量控制無，傳送速率完全由應用控制
壅塞控制無，可能對網路造成衝擊
Header固定 8 bytes
傳輸模式資料報（保留訊息邊界）
廣播/多播支援
適用DNS、DHCP、VoIP、遊戲、QUIC

「訊息邊界」的關鍵差異

TCP 是位元組串流：發送 send("Hello") 再 send("World")， 接收方可能讀到 "HelloWorld" 或分多次接收，邊界消失，需應用層自訂分隔（如 HTTP 的 Content-Length）。

UDP 是資料報：每次 sendto() 對應一次 recvfrom()，訊息邊界完整保留。 但若資料報超過路徑 MTU，IP 層分割後任一分片遺失就丟棄整個資料報， 所以 UDP 應用通常控制訊息 ≤ 1472 bytes（Ethernet MTU 1500 − IP 20 − UDP 8）。

Section 05

QUIC：UDP 上的現代傳輸層

QUIC（RFC 9000）最初由 Google 開發，現為 HTTP/3 的傳輸層基礎。 它展示了「在 UDP 上自建傳輸層」能達到的極致， 解決了 TCP 長久以來無法在不更動作業系統核心的情況下改善的問題。

QUIC vs TCP + TLS 1.3 關鍵差異
面向
TCP + TLS 1.3
QUIC（UDP 基礎）
連線建立
1 RTT TCP + 1 RTT TLS = 2 RTT
1 RTT 初次；0-RTT 再次連線
HOL Blocking
有：單一 TCP 連線任一封包遺失阻塞所有串流
無：多串流（Stream）各自獨立，互不阻塞
連線遷移
換網路（Wi-Fi → 4G）須重建 TCP 連線
Connection ID 機制，換 IP/Port 仍保持連線
加密範圍
只加密 Payload；TCP Header 明文可被操縱
內建 TLS 1.3，Header 也加密
壅塞控制
受 OS 核心限制，難以客製化
使用者空間實作，可插拔演算法

0-RTT 的安全考量

0-RTT 允許在握手完成前傳送資料，但有重播攻擊（Replay Attack）的風險：攻擊者可截取並重複傳送 0-RTT 資料。因此 0-RTT 只適合冪等操作（如 HTTP GET），非冪等操作（如 HTTP POST 付款）應使用 1-RTT。

三種傳遞模式示意
單播（Unicast）：一對一
  傳送方 ──────────────────────────> 接收方 A

廣播（Broadcast）：一對所有（同子網路）
  傳送方 ──> A（含不需要的主機）
         ──> B（含不需要的主機）
         ──> C（含不需要的主機）

  傳送方 ──> A（已訂閱）
         ──> C（已訂閱）

  傳送方 ──> 最近的伺服器節點（BGP 路由選擇）

傳輸層TCP 或 UDP僅 UDP僅 UDP

IPv4 多播位址

IPv4 多播使用 Class D 位址：224.0.0.0 至 239.255.255.255（二進位前四位 1110）。

範圍用途
224.0.0.0/24本地鏈路（Link-local）：TTL=1，路由器不轉發，用於路由協定鄰居發現
224.0.1.0 – 238.x.x.x全球多播：可跨網際網路路由（需 ISP 支援）
239.0.0.0/8管理性範圍（Administratively Scoped）：類似私有位址，用於組織內部多播

多播 MAC 位址映射

乙太網路上的多播 IP 映射到 MAC 位址規則： 前綴固定為 01:00:5e，後 23 bits 取自多播 IP 後 23 bits。 因捨棄一個位元，會有 32:1 的映射衝突（如 224.0.0.5 與 225.0.0.5 對應同一 MAC）， 需 IGMP Snooping 在交換器層面消除不必要的轉發。

224.0.0.5（OSPF All Routers）→ MAC 映射
多播 IP：  224.0.0.5
二進位：  1110 0000.0000 0000.0000 0000.0000 0101
                         取後 23 bits（丟棄第 25 bit）
                              ↓
多播 MAC：01:00:5e : 00 : 00 : 05
          固定前綴    後三組 = IP 後 23 bits
Section 08

IGMP：多播群組管理協定

IGMP（Internet Group Management Protocol）是主機與路由器之間的多播訂閱機制。 主機透過 IGMP 加入/離開多播群組，路由器根據此資訊決定是否轉發多播流量到該子網路。 IGMP 直接封裝在 IP 中（Protocol = 2）。

版本RFC主要特性
IGMPv1RFC 1112（1989）基本加入機制；無主動離開，等路由器逾時；定期 General Query
IGMPv2RFC 2236（1997）新增主動 Leave Group，縮短離開到停播的延遲；Group-Specific Query
IGMPv3RFC 3376（2002）Source-Specific Multicast（SSM）：指定只接收特定來源的多播流
IGMPv2 加入與離開群組流程
主機 A（訂閱 224.1.2.3 IPTV）            路由器
    │                                      │
    │── IGMP Membership Report ────────────>│  「我要加入 224.1.2.3」
    │                                      │  路由器記錄並開始轉發
    │
    │<── IGMP General Query（定期）─────────│  「誰還在線上？」
    │── IGMP Membership Report ────────────>│  （隨機延遲回應，防止爆炸）
    │
    │── IGMP Leave Group ─────────────────>│  「我要離開了」→ 224.0.0.2
    │<── Group-Specific Query ──────────────│  「還有其他訂閱者嗎？」
    │  （無人回應）
    │                                      │  路由器停止轉發 224.1.2.3 到此介面

IGMP Snooping

交換器（Layer 2）預設把多播訊框發給所有介面，造成未訂閱主機收到大量流量。 IGMP Snooping 讓交換器偷聽 IGMP 訊息， 學習哪些介面有訂閱者，只把多播轉發到相關介面， 是現代企業交換器的標準功能。

Section 09

多播路由協定

IGMP 只管理主機與直連路由器的訂閱關係； 讓多播跨越多個路由器需要多播路由協定， 構建從來源到所有訂閱者的分發樹。

協定特性適用場景
PIM-SM（Sparse Mode） 以集合點（RP）為核心建立共享樹（Shared Tree），再優化為最短路徑樹（SPT） 最廣泛使用；訂閱者分散的大型網路
PIM-DM（Dense Mode） 先洪泛到所有路由器，再剪枝不需要的分支 訂閱者密集的小型網路；大型網路效率差
PIM-SSM（Source Specific） 接收者指定來源 IP，不需要 RP，直接建立最短路徑樹 IPTV、金融行情等單一來源多接收者場景
DVMRP 距離向量多播路由（已逐漸淘汰） 早期網際網路 MBone 多播骨幹

為何公共網際網路多播普及率低？

ISP 不願在骨幹路由器維護多播狀態（開銷高）、商業模式不清晰、CDN 的普及讓用單播模擬多播的成本已大幅下降。實際 IPTV 多播通常限制在 ISP 的 MPLS 骨幹內，不跨越公共網際網路。

Section 10

UDP 安全考量

UDP 的無連線與無狀態特性帶來比 TCP 更大的安全風險，最嚴重的是放大攻擊（Amplification Attack）。

UDP 放大攻擊

攻擊者偽造受害者 IP 作為來源，向 UDP 服務發送小請求，服務回傳比請求大得多的回應給受害者，形成流量倍增的 DDoS。

攻擊類型放大倍數原理
DNS 放大28–54×查詢 ANY 紀錄，回應比查詢大數十倍
NTP 放大（monlist）約 556×NTP monlist 回傳最近 600 個客戶端清單
SSDP 放大30–40×家用 UPnP 設備的 SSDP 回應遠大於探測封包
Memcached 放大高達 51,000×2018 年 GitHub 的 1.35 Tbps DDoS 源自此攻擊

防護措施

措施說明
BCP 38 Anti-SpoofingISP 在邊界過濾來源 IP 非其客戶網段的封包，從根本上消除來源偽造
最小化暴露面防火牆只開放必要 UDP 服務埠，停用不必要服務（如 NTP monlist）
速率限制（RRL）對入站 UDP 做速率限制；DNS Response Rate Limiting 防 DNS 放大
DTLS 加密Datagram TLS（RFC 6347）為 UDP 提供加密與認證，用於 WebRTC、QUIC
來源驗證應用層 HMAC 驗證；DNS 啟用 DNSSEC 防回應偽造

完全封鎖 UDP 的代價

有些防火牆政策選擇封鎖所有 UDP 流量，但這會讓 DNS、DHCP、NTP、QUIC/HTTP/3 全部失效，是過度限制的做法。正確策略是精準的服務白名單 + 速率限制，而非一刀切封鎖。

Day 8 重點回顧

今天你學到了什麼

  • UDP 僅 8 bytes Header，無連線、無重傳、無排序——以速度換可靠性
  • UDP 保留訊息邊界（Datagram）；TCP 是位元組串流，邊界消失
  • IPv4 的 UDP Checksum 選填；IPv6 中強制（IPv6 Header 無 Checksum）
  • DNS、DHCP、VoIP、遊戲、SNMP、NTP 等選擇 UDP，各因低延遲或廣播需求
  • QUIC 在 UDP 上自建傳輸層，解決 TCP 的 HOL Blocking 和多 RTT 握手問題
  • 0-RTT 可能有重播攻擊風險，只應用於冪等操作（如 HTTP GET）
  • 單播 vs 廣播 vs 多播：多播只送給有訂閱者的介面，頻寬效率最高
  • IPv4 多播 Class D（224.0.0.0/4）；224.0.0.0/24 為本地鏈路，路由器不轉發
  • IGMP 管理主機與路由器的訂閱關係；IGMP Snooping 讓交換器智慧轉發多播
  • UDP 放大攻擊利用 IP 偽造 + 大回應放大流量；BCP 38 Anti-Spoofing 是根本防護
