Section 01

ARP 的存在意義

在 Day 3 我們學到封包旅程中有一個關鍵步驟： 路由器在轉送封包前必須用 ARP 取得下一跳的 MAC 位址。 但為什麼必須這樣做？IP 位址不夠用嗎？

根本原因在於網路架構的分層設計。 IP 位址（Layer 3）是邏輯位址， 用於跨越整個網際網路的端對端識別，全域唯一。 MAC 位址（Layer 2）是實體位址， 燒錄在網路卡上，只在同一個區域網路內有意義。

乙太網路交換器只懂 MAC 位址——它根本不看 IP。 當封包需要在同一個 LAN 內從 A 送到 B 時， 乙太網路訊框的目的地欄位填的是 MAC 位址，而不是 IP 位址。 問題是：你知道 B 的 IP（例如從 DNS 查詢到）， 但不知道 B 的 MAC，這時就需要 ARP 來查詢。

核心矛盾

IP 提供全域定址（任何地方）， MAC 提供本地傳輸（同一個 LAN 內）。 兩者的分工讓網路架構既靈活又高效， 但也製造了一個間隙： 已知 IP 卻不知道 MAC 時怎麼辦？ ARP（Address Resolution Protocol）正是填補這個間隙的協定。

ARP 定義於 RFC 826（1982 年），是一個非常簡單但至今仍在廣泛使用的協定。 它屬於 TCP/IP 的網路存取層，或按 OSI 分類， 介於 Layer 2（資料鏈結層）與 Layer 3（網路層）之間的橋接協定。

Section 02

ARP 封包結構

ARP 封包直接封裝在乙太網路訊框中， EtherType 欄位值為 0x0806（不同於 IPv4 的 0x0800）。 封包結構固定為 28 bytes（IPv4 over Ethernet 情況下）。

ARP 封包欄位（IPv4 over Ethernet，共 28 bytes）
Hardware Type（硬體類型） 0x0001（Ethernet） 16 bits
Protocol Type（協定類型） 0x0800（IPv4） 16 bits
HW Addr Len 6（bytes） 8 bits
Proto Addr Len 4（bytes） 8 bits
Operation（操作碼） 1 = ARP Request　2 = ARP Reply 16 bits
Sender Hardware Address（傳送方 MAC） 例：AA:BB:CC:DD:EE:FF 48 bits（6 bytes）
Sender Protocol Address（傳送方 IP） 例：192.168.1.10 32 bits（4 bytes）
Target Hardware Address（目標 MAC） Request 時填 00:00:00:00:00:00（未知） 48 bits（6 bytes）
Target Protocol Address（目標 IP） 例：192.168.1.1（要查詢的 IP） 32 bits（4 bytes）

ARP 封包的設計非常通用：Hardware Type 欄位可以代表任何鏈路層技術， Protocol Type 可以代表任何網路層協定。 理論上 ARP 可以用於 IPv4 over Token Ring、 IPv4 over FDDI 等各種組合，但實際上 99% 的使用場景都是 Ethernet over IPv4。

Section 03

ARP 完整運作流程

假設主機 A（192.168.1.10 / AA:AA:AA:AA:AA:AA） 想要傳送資料給主機 B（192.168.1.20）， 但不知道 B 的 MAC 位址。完整流程如下：

ARP 解析全流程（步驟一到六）
主機 A                           主機 B（192.168.1.20）       其他主機 C、D、E
192.168.1.10                     MAC: BB:BB:BB:BB:BB:BB        （各有不同 IP/MAC）
MAC: AA:AA:AA:AA:AA:AA
│                                        │                           │
│  Step 1：查 ARP Cache                  │                           │
│  cache: 192.168.1.20 → ？（沒有）      │                           │
│                                        │                           │
│  Step 2：建立 ARP Request              │                           │
│  Src MAC: AA:AA:AA:AA:AA:AA            │                           │
│  Src IP:  192.168.1.10                 │                           │
│  Dst MAC: FF:FF:FF:FF:FF:FF（廣播）    │                           │
│  Dst IP:  192.168.1.20                 │                           │
│                                        │                           │
│── ARP Request（廣播）─────────────────>│                          >│
│   「192.168.1.20 是誰？告訴我你的 MAC」│     C、D、E 收到後靜默     │
│                                        │                           │
│                        Step 3：B 收到 ARP Request
│                        B 確認：目標 IP = 192.168.1.20 = 自己！
│                        順便把 A 的 IP→MAC 記入自己的 ARP Cache
│                        建立 ARP Reply（單播）
│                                        │
│<── ARP Reply（單播）──────────────────│
│    Src MAC: BB:BB:BB:BB:BB:BB          │

│    Dst MAC: AA:AA:AA:AA:AA:AA          │
│    Dst IP:  192.168.1.10               │
│    「我是 192.168.1.20，我的 MAC 是 BB:BB:BB:BB:BB:BB」
│                                        │
│  Step 4：A 更新 ARP Cache             │
│  192.168.1.20 → BB:BB:BB:BB:BB:BB    │
│                                        │
│  Step 5：A 開始傳送資料                │
│  乙太網路訊框 Dst MAC = BB:BB:BB:BB:BB:BB
│──── 資料（IP Packet）─────────────────>│

ARP Cache（ARP 快取）

每次 ARP 查詢都需要廣播，頻繁查詢會增加網路負擔。 因此作業系統會將查詢結果暫存在 ARP Cache 中， 下次傳送時直接查表，無需重新廣播。

ARP Cache 有老化（Aging）機制， 條目在一定時間後自動刪除（Windows 預設 2 分鐘，Linux 預設 60 秒）， 以避免 IP 位址變更後使用舊的 MAC 位址導致通訊失敗。

查看 ARP Cache 指令

  網際網路網址          實體位址              類型
  192.168.1.1          aa-bb-cc-dd-ee-ff     動態
  192.168.1.20         bb-bb-bb-bb-bb-bb     動態
  224.0.0.22           01-00-5e-00-00-16     靜態（多播）

# Linux
ip neigh show
192.168.1.1  dev eth0 lladdr aa:bb:cc:dd:ee:ff  REACHABLE
192.168.1.20 dev eth0 lladdr bb:bb:bb:bb:bb:bb  STALE

# macOS

gateway (192.168.1.1) at aa:bb:cc:dd:ee:ff  on en0  [ethernet]
? (192.168.1.20) at bb:bb:bb:bb:bb:bb  on en0  [ethernet]

ARP 與跨路由器通訊

一個重要觀念：ARP 不會跨越路由器。 廣播封包在路由器這裡被攔截，不會轉發到其他子網路。 當主機 A 想要與不同子網路的主機 X 通訊時， A 只需要用 ARP 查到預設閘道（路由器）的 MAC 位址， 然後把 IP Packet 封裝進目的 MAC = 路由器 MAC 的乙太網路訊框中送出。 路由器收到後，解封裝至 IP 層，再重新封裝並繼續轉發。 這就是為什麼「IP 位址端對端不變，MAC 位址每跳都換」。

Section 04

Gratuitous ARP 與 Proxy ARP

Gratuitous ARP（免費 ARP）

Gratuitous ARP 是一種特殊的 ARP， 其中傳送方 IP = 目標 IP， 也就是主機向網路廣播「我的 IP 是 X，我的 MAC 是 Y」， 但不是在回應任何人的查詢。這個「主動公告」有三個主要用途：

用途 說明
IP 衝突偵測 主機開機時廣播自己的 IP，若收到回應表示有 IP 衝突
MAC 位址更新 網路卡更換後廣播新 MAC，讓 LAN 內所有設備更新 ARP Cache
高可用性切換 VRRP/HSRP 等備援協定在主機切換時，用 Gratuitous ARP 通知所有設備更新閘道 MAC

Proxy ARP

Proxy ARP 讓路由器代表遠端主機回應 ARP 查詢。 當主機 A 用 ARP 查詢不在同一子網路的 IP B 時， 路由器識別出「B 在我能到達的另一側」， 便回應「B 的 MAC 就是我的 MAC」， 讓 A 把封包送給路由器，再由路由器轉發給 B。

Proxy ARP 常用於無法設定預設閘道的老舊設備或特殊網路架構， 但它會使設備誤以為所有 IP 都在同一個 LAN 中， 增加廣播流量，並造成路由行為不透明，現代網路設計中通常會停用。

Section 05

ARP 安全威脅

ARP 最嚴重的設計缺陷是完全沒有認證機制。 任何設備都可以發送 ARP Reply，而接收方會無條件相信並更新 ARP Cache。 這個漏洞催生了多種攻擊手法。

ARP Spoofing / ARP Poisoning（ARP 欺騙）

攻擊者向主機 A 發送偽造的 ARP Reply： 「192.168.1.1（閘道）的 MAC 是 EE:EE:EE:EE:EE:EE（攻擊者的 MAC）」。 同時向閘道發送偽造的 ARP Reply： 「192.168.1.10（主機 A）的 MAC 是 EE:EE:EE:EE:EE:EE（攻擊者的 MAC）」。

結果：主機 A 到閘道的所有流量都流向攻擊者， 閘道到主機 A 的流量也流向攻擊者。 攻擊者轉發流量（讓雙方感覺一切正常），同時暗中監聽—— 這就是中間人攻擊（Man-in-the-Middle Attack）。

ARP Spoofing 攻擊示意
正常流量路徑：
  主機 A ─────────────────────────────> 閘道 → 網際網路

ARP Spoofing 後：
  主機 A ──> 攻擊者（以為是閘道）──> 閘道 → 網際網路
                    │
                    └──── 監聽/竄改/截取所有明文流量

偽造的 ARP Reply（攻擊者發送）：
  → 向 A 宣告：閘道 192.168.1.1 的 MAC = EE:EE:EE:EE:EE:EE（攻擊者）
  → 向閘道宣告：主機 A 192.168.1.10 的 MAC = EE:EE:EE:EE:EE:EE（攻擊者）

防護措施

防護技術 說明 部署層次
Dynamic ARP Inspection（DAI） 交換器驗證 ARP 封包的 IP/MAC 對應是否符合 DHCP Snooping 資料庫，拒絕可疑 ARP Layer 2 交換器
靜態 ARP 條目 手動在重要設備（如閘道）建立靜態 ARP 綁定，不可被動態 ARP 覆蓋 主機端
加密傳輸（TLS/VPN） 即使遭到 ARP Spoofing，加密流量也無法被讀取或篡改 應用層
ARP 監控工具 使用 arpwatch、XArp 等工具監控 ARP Cache 異常變動並發出警報 監控層
Section 06

IPv6 的 NDP 取代 ARP

IPv6 設計之初就意識到 ARP 的廣播效率低與安全缺陷問題， 因此以NDP（Neighbor Discovery Protocol，鄰居發現協定，RFC 4861） 取代 ARP 的所有功能，並整合進 ICMPv6 中（類型 133–137）。

NDP 訊息 ICMPv6 類型 對應 IPv4 功能
Router Solicitation（RS） Type 133 主機開機時向路由器請求網路資訊（相當於 DHCP Discover 的部分功能）
Router Advertisement（RA） Type 134 路由器定期廣播網路前綴、閘道資訊，用於 SLAAC 自動配置
Neighbor Solicitation（NS） Type 135 等同 ARP Request，查詢 IPv6 位址對應的 MAC 位址
Neighbor Advertisement（NA） Type 136 等同 ARP Reply，回應 MAC 位址；也用於 Gratuitous ARP 的場景
Redirect Type 137 等同 ICMP Redirect，告知主機有更好的路由路徑

NDP 最大的改進是使用多播而非廣播。 NS 訊息發送到請求節點多播位址（Solicited-Node Multicast Address）， 由每個目標 IP 的後 24 位元衍生，格式為 ff02::1:ffXX:XXXX。 只有擁有該 IP 的設備會加入對應的多播群組，大幅減少不相關設備的處理負擔。

SEcure Neighbor Discovery（SEND）

RFC 3971 定義了 SEND 協定，透過加密數位簽章驗證 NDP 訊息的合法性， 防止 NDP Spoofing 攻擊（類似 ARP Spoofing）。 不過 SEND 部署複雜，實際採用率不高； 大多數環境仍依賴 DAI 的 IPv6 版本（ND Inspection）作為防護。

Section 07

ICMP 協定結構

ICMP（Internet Control Message Protocol）定義於 RFC 792（1981 年）， 是 IP 協定的「控制訊息系統」。它不用於傳輸使用者資料， 而是用於回報錯誤狀況（封包無法到達、TTL 超時、路由不存在） 以及提供網路診斷功能（ping、traceroute）。

ICMP 封包直接封裝在 IP 封包中，IP Header 的 Protocol 欄位值為 1。 ICMPv6 則使用 Protocol 值 58，用於 IPv6 網路。

ICMP 封包通用結構
Type（類型） 訊息大類，如 0=Echo Reply 8 bits
Code（代碼） 細分類，如 Type 3 的原因 8 bits
Checksum 覆蓋 ICMP Header + Data 16 bits
Rest of Header（依 Type/Code 而異） Echo：Identifier（16b）+ Sequence Number（16b） 32 bits
Data（資料段，依 Type 而異） 錯誤訊息：含原始 IP Header + 前 8 bytes 資料；Echo：使用者填充資料 variable

ICMP 是無狀態的——它不建立連線， 每個 ICMP 訊息都是獨立的。ICMP 錯誤訊息只在收到有問題的封包時產生， 且只回傳給原始封包的傳送方，永遠不會觸發新的 ICMP 錯誤（防止 ICMP 風暴）。

Section 08

ICMP 訊息類型全覽

ICMP 訊息分為錯誤訊息（Error Messages） 與查詢訊息（Query/Informational Messages）兩大類。 以下是最重要的類型：

Type 0
Code: 0
Echo Reply
回應 Type 8 的 ping 請求，包含相同的 Identifier 與 Sequence Number
Type 3
Code: 0–15
Destination Unreachable
封包無法到達目的地。Code 0=網路不可達、1=主機不可達、3=連接埠不可達、4=需要分割但 DF 旗標設起
Type 5
Code: 0–3
Redirect
路由器通知傳送方有更好的路徑，請更新路由。Code 0=網路、1=主機、2=ToS+網路、3=ToS+主機
Type 8
Code: 0
Echo Request
ping 指令發出的請求封包。包含 Identifier（識別連線）與 Sequence Number（識別第幾個 ping）
Type 9
Code: 0
Router Advertisement
路由器廣播自己的存在與優先權（與 IRDP 協定相關，IPv4 的路由器發現）
Type 11
Code: 0–1
Time Exceeded
Code 0：TTL 超時（traceroute 的核心機制）。Code 1：分割封包重組超時
Type 12
Code: 0–2
Parameter Problem
IP Header 中發現非法欄位。Code 指向有問題欄位的位元組偏移量
Type 13/14
Timestamp Request/Reply
Timestamp
測量時間同步差異，但實際使用率極低，多數防火牆預設封鎖

ICMP Type 3 Code 3 的實際意義

當你連線到一個伺服器的某個 UDP 連接埠但該連接埠沒有程式在監聽時， 作業系統會回傳 ICMP Type 3 Code 3（Port Unreachable）。 這是 nmap 做 UDP 埠掃描的核心原理之一： 沒有回應可能是有服務在跑；收到 Type 3 Code 3 確認該埠沒有服務。

Section 09

ping 的運作原理

ping 是最基本、最常用的網路診斷工具， 利用 ICMP Echo Request（Type 8）與 Echo Reply（Type 0）測試： 目標主機是否可達、RTT（Round-Trip Time，往返延遲）是多少、 以及是否有封包遺失。

ping 封包交換過程
傳送方（你的電腦）                            目標主機（8.8.8.8）
        │                                           │
        │── ICMP Echo Request ─────────────────────>│
        │   Type=8, Code=0                          │
        │   Identifier=1234（識別此 ping 程序）      │
        │   Sequence=1（第一個封包）                 │
        │   Data: 填充位元組（用於計算 RTT）         │
        │                              t1（送出時間） │
        │                                           │
        │<── ICMP Echo Reply ────────────────────────│
        │   Type=0, Code=0                          │
        │   Identifier=1234（與 Request 相同）       │
        │   Sequence=1（對應第一個封包）             │
        │                              t2（收到時間） │
        │                                           │
        RTT = t2 - t1
        ICMP Sequence=1: 64 bytes from 8.8.8.8: icmp_seq=1 ttl=55 time=12.3 ms

解讀 ping 的輸出

ping 8.8.8.8 典型輸出解析
PING 8.8.8.8 (8.8.8.8): 56 data bytes
64 bytes from 8.8.8.8: icmp_seq=0 ttl=116 time=12.381 ms
64 bytes from 8.8.8.8: icmp_seq=1 ttl=116 time=11.952 ms
64 bytes from 8.8.8.8: icmp_seq=2 ttl=116 time=12.104 ms

--- 8.8.8.8 ping statistics ---
3 packets transmitted, 3 packets received, 0.0% packet loss
round-trip min/avg/max/stddev = 11.952/12.146/12.381/0.179 ms

解析：
  64 bytes    = Echo Reply 封包大小（56 data + 8 ICMP header）
  icmp_seq    = 封包序號（用於偵測遺失與亂序）
  ttl=116     = 回應封包到達時剩餘的 TTL 值（可推算路徑長度）
  time        = RTT，從送出到收到回應的毫秒數
  stddev      = 標準差，代表網路抖動（Jitter）大小

常見 ping 故障診斷

現象 可能原因 下一步
Request timeout 目標不可達、防火牆封鎖 ICMP、對方關閉 ping 回應 先 ping 閘道，確認 Layer 3 連線；再用 traceroute 定位斷點
Destination Host Unreachable 本機沒有到目標的路由，或閘道無法路由 確認路由表（ip route）與閘道設定
TTL expired in transit TTL 耗盡（路由迴圈或路徑過長） 用 traceroute 觀察路由是否有迴圈
高 RTT 或大 stddev 網路壅塞、Wi-Fi 干擾、長距離傳輸 測試有線 vs 無線差異；用 mtr 持續觀察各跳延遲
封包遺失 鏈路品質差、交換器錯誤、壅塞丟包 加大測試次數（ping -c 100）；觀察遺失模式
Section 10

traceroute 的運作原理

traceroute（Windows 上為 tracert）能顯示封包從你的電腦到目標主機 途中經過的每一個路由器（跳）及其延遲。 它的核心機制非常巧妙：利用 IP 的 TTL 欄位，讓每一跳路由器都「暴露自己」。

TTL 遞增技巧

traceroute 依序發送 TTL = 1, 2, 3, 4... 的探測封包。 每個封包在 TTL 歸零的那個路由器被丟棄， 並觸發ICMP Type 11 Time Exceeded回傳， 該回傳包含路由器的 IP 位址，由此得知第幾跳的路由器是誰。

traceroute TTL 遞增機制（目標：8.8.8.8）
你的電腦                 路由器 R1            路由器 R2          8.8.8.8
    │                       │                    │                 │
    │ TTL=1 → 封包 ─────────>│                    │                 │
    │         R1 TTL=1 → 0  │                    │                 │
    │<─── ICMP Time Exceeded（R1 的 IP）──────────│                 │
    │                       │                    │                 │
    │ TTL=2 → 封包 ─────────>│─── TTL=1 ──────────>│                 │
    │                       │    R2 TTL=1 → 0    │                 │
    │<─── ICMP Time Exceeded（R2 的 IP）──────────────────────────── │
    │                       │                    │                 │
    │ TTL=3 → 封包 ─────────>│─── TTL=2 ──────────>│── TTL=1 ────────>│
    │                       │                    │  目標收到：回傳 │
    │<─── Echo Reply 或 Port Unreachable（8.8.8.8）──────────────────│
    │                       │                    │                 │
  → traceroute 完成：已探知所有路徑上的路由器

每個 TTL 值通常發送三個探測封包（預設），並顯示三次 RTT， 用於評估該跳的延遲穩定性。 * 表示該跳的路由器封鎖了 ICMP 或探測封包超時， 但這不代表封包實際無法通過，只是路由器選擇不回應。

mtr：動態版 traceroute

mtr（Matt's Traceroute）結合 ping 與 traceroute， 持續傳送探測封包並即時顯示每跳的丟包率與平均延遲。 比單次 traceroute 更能觀察間歇性網路問題。 Linux 安裝：apt install mtr；macOS：brew install mtr。

Section 11

ICMP 在攻擊中的角色

ICMP 的設計初衷是診斷，但攻擊者也找到了利用它的方式。 了解這些攻擊有助於理解為什麼許多防火牆預設限制 ICMP 流量。

Ping of Death

早期（1990 年代）攻擊手法：傳送超大（> 65,535 bytes）的 ICMP 封包， 利用 IP 分割機制讓封包在接收端重組時超出緩衝區，導致系統崩潰。 現代作業系統已修補此漏洞，但仍具歷史意義。

ICMP Flood（Ping Flood）

大量發送 ICMP Echo Request，讓目標 CPU 和頻寬忙於處理與回應， 造成拒絕服務（DoS）。高速的 ICMP Flood 可輕易讓低頻寬目標離線。 防禦方式：限制 ICMP 處理速率（rate limiting）或在邊界防火牆封鎖外部 ICMP。

Smurf Attack

攻擊者偽造受害者的 IP 位址作為來源， 向大型網路的廣播位址（如 10.0.0.255）發送 ICMP Echo Request。 網路內所有主機都回應 Echo Reply，但全部回傳給被偽造 IP 的受害者， 形成流量放大（Amplification）攻擊。 現代路由器預設不轉發廣播封包，此攻擊已幾乎消失。

ICMP Redirect 攻擊

攻擊者偽造 ICMP Redirect（Type 5）訊息， 誘導受害主機更新路由至攻擊者控制的設備， 實現流量劫持。現代作業系統預設忽略外部 ICMP Redirect 訊息， 但在容器環境或特殊核心設定下仍需注意。

封鎖 ICMP 的兩面性

完全封鎖 ICMP 會讓網路排錯極為困難： ping 失效、traceroute 不完整、MTU Path Discovery 依賴的 ICMP Type 3 Code 4（Fragmentation Needed）也會失效， 導致 TCP 連線的 MTU 問題難以偵測。 建議的策略是：允許有意義的 ICMP 類型（0、3、8、11）， 在邊界對外部 ICMP 進行速率限制，而非完全封鎖。

Day 6 重點回顧

今天你學到了什麼

  • ARP 解決「知道 IP 但不知道 MAC」的問題，透過廣播 Request 與單播 Reply 完成
  • ARP Cache 快取結果並老化（60–120 秒），避免頻繁廣播
  • ARP Request 廣播不跨路由器；跨子網路通訊只需 ARP 查閘道 MAC
  • Gratuitous ARP 用於 IP 衝突偵測、MAC 更新通知、HA 切換
  • ARP 無認證機制，ARP Spoofing 可實現 MitM；DAI、靜態 ARP、TLS 是防護手段
  • IPv6 的 NDP 取代 ARP，使用多播而非廣播，整合進 ICMPv6（Type 135/136）
  • ICMP 封裝在 IP 中（Protocol=1），用 Type+Code 描述各種錯誤與查詢
  • ping 用 Type 8（Request）/ Type 0（Reply），測 RTT 與封包遺失
  • traceroute 利用 TTL 遞增，令每跳觸發 ICMP Type 11 Time Exceeded 暴露路由器 IP
  • Unix 用 UDP 探測、Windows 用 ICMP；mtr 提供持續動態監控
