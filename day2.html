Phase 01 — 網路基礎概念 — Day 02 / 16

OSI 七層模型
網路通訊的通用語言

OSI 模型是理解一切網路協定的根基。它將複雜的通訊過程拆解為七個可獨立討論的層次， 讓不同廠商的設備得以相互溝通，也讓工程師在排錯時能精準定位問題所在的層次。

Section 01

為何需要分層模型

想像你要製造一輛汽車：引擎工程師不需要懂車漆配色， 外觀設計師不需要懂燃油噴射原理。透過清晰的模組分工， 各領域專家能獨立作業，最後組裝成一個完整系統。 電腦網路也面臨同樣的複雜度挑戰。

1970 年代，各大電腦廠商（IBM、DEC、Honeywell）各自建立私有的網路協定， 不同廠商的設備完全無法互通。IBM 的 SNA、DEC 的 DECnet 就像不同語言的人試圖溝通，毫無效果。

ISO（國際標準化組織）從 1977 年開始研究標準化方案， 於 1984 年正式發布 OSI 參考模型（ISO 7498）。 這個模型的目標是：只要每一層遵循標準介面，任何廠商的實作都能互通。

分層設計的三大優勢

抽象化：每層只專注自己的任務，不需關心其他層的實作細節。 互操作性：符合同一層標準的產品可以相互替換。 可維護性：修改某一層的實作不影響其他層，降低變更風險。

有一個重要認知：OSI 是參考模型（Reference Model）， 不是實作規格。它描述「應該做什麼」，不規定「如何做」。 現實中網路通訊主要依賴 TCP/IP 協定族，而非嚴格遵循 OSI 的七層， 但 OSI 的框架依然是分析、討論、教學網路的最佳工具。

Section 03

各層深入解析

Layer 1 — 實體層（Physical Layer）

實體層關注最底層的問題：如何把 0 和 1 在物理介質上傳送出去。 它定義了電壓代表什麼（高電壓 = 1，低電壓 = 0）， 每個位元持續多少時間（位元率），接頭的幾何形狀， 以及訊號的調變方式。實體層不理解「資料的意義」，只管「訊號的傳遞」。

常見的實體層技術包含：雙絞線（Cat5e/Cat6/Cat7）、同軸電纜、 光纖（多模/單模）、Wi-Fi 無線電波、藍牙、NFC。 Hub 和 Repeater 是運作在實體層的設備，它們只放大或複製訊號， 完全不解析訊號的內容。

實體層規格範例

IEEE 802.3ab（Gigabit Ethernet over Cat5e）規定：使用 4 對雙絞線、 125 MHz 時鐘、PAM-5 調變、最大傳輸距離 100 公尺、RJ-45 接頭。 這些全是實體層關注的細節。

Layer 2 — 資料鏈結層（Data Link Layer）

資料鏈結層負責在同一個網路區段（Segment）內的點對點傳輸。 它將位元流組織成訊框（Frame），並加上來源與目的的 MAC 位址（Media Access Control Address）。 MAC 位址是 48 位元（6 個位元組）的硬體識別碼， 燒錄在網路卡（NIC）中，格式如 AA:BB:CC:DD:EE:FF。

資料鏈結層分為兩個子層： LLC（Logical Link Control）提供流量控制與錯誤通知； MAC（Media Access Control）控制誰有權在共享媒介上傳輸， 乙太網路使用 CSMA/CD（碰撞偵測）機制，Wi-Fi 使用 CSMA/CA（碰撞避免）。

Switch 是資料鏈結層的代表設備。它維護一張 MAC 位址表（CAM Table），記錄每個 MAC 位址對應哪個連接埠， 從而實現精準轉送，避免廣播風暴。

Layer 3 — 網路層（Network Layer）

網路層解決了「如何在不同網路之間傳送封包」的問題。 它引入邏輯位址（IP 位址）的概念， 與 MAC 位址不同，IP 位址是根據網路拓樸動態分配的， 可以改變，也可以透過路由找到通往任何地點的路徑。

核心功能包括：路由（Routing）——決定封包走哪條路； 分割（Fragmentation）——當封包大於鏈路 MTU 時切割； TTL（Time to Live）——防止封包在網路中無限迴圈。 Router 是網路層的代表設備，IP 是這層最核心的協定。

Layer 4 — 傳輸層（Transport Layer）

傳輸層是端對端（End-to-End）通訊的保障者。 它解決的問題是：在 IP 只保證「盡力傳送」的基礎上， 如何確保應用程式收到完整、有序、無重複的資料？

傳輸層引入了連接埠（Port）的概念， 允許同一台主機上的多個應用程式同時通訊。 TCP 提供可靠的連線導向服務，有流量控制與壅塞控制； UDP 提供快速但不可靠的無連線服務。 Day 7 和 Day 8 會深入討論兩者。

Layer 5 — 會話層（Session Layer）

會話層管理兩台主機間的對話（Session）的生命週期， 包括建立、維護和終止。它負責同步（Synchronization）—— 在長時間傳輸中插入檢查點（Checkpoint），讓傳輸中斷後可以從斷點恢復， 而非從頭重傳。

在實際的 TCP/IP 協定族中，會話層的功能被整合到應用層協定中。 例如 HTTP 的 Cookie 機制、TLS 的 Session ID 都承擔了會話管理的職責。

Layer 6 — 表現層（Presentation Layer）

表現層是「翻譯官」，確保不同系統間資料格式的一致性。 它處理三件事：資料格式轉換（例如 Big Endian vs. Little Endian、 EBCDIC vs. ASCII）、加密與解密、以及壓縮與解壓縮。

現代 TLS/SSL 加密通常被歸類到表現層，雖然其實作橫跨多個層次。 JPEG、PNG 圖片格式、MP3 音訊格式、ZIP 壓縮也都是表現層關注的範疇。

Layer 7 — 應用層（Application Layer）

應用層是最接近終端使用者的層次，提供網路應用程式直接使用的服務介面。 注意：應用層不是指應用程式本身（瀏覽器、郵件客戶端）， 而是這些應用程式與網路溝通的協定介面。

HTTP（網頁）、HTTPS（安全網頁）、FTP（檔案傳輸）、 SMTP（電子郵件傳送）、POP3/IMAP（電子郵件接收）、 DNS（網域名稱解析）、SSH（安全遠端登入）、 SNMP（網路管理）都屬於應用層協定。 這些是後續課程的主要討論對象。

Section 04

封裝與解封裝

OSI 模型最關鍵的運作機制是封裝（Encapsulation）。 當資料從應用層往下傳遞時，每一層都會在資料前面加上自己的 標頭（Header），有些層（如資料鏈結層）還會加上 尾部（Trailer）用於錯誤檢測。

到了接收端，過程完全相反——從實體層往上逐層 解封裝（Decapsulation），每層讀取並移除自己的標頭， 直到應用層收到原始資料。

封裝流程（傳送端，由上至下）
應用層    [          HTTP 資料            ]
          ↓ 加上 TCP Header
傳輸層    [TCP Hdr][      HTTP 資料       ]        ← Segment
          ↓ 加上 IP Header
網路層    [IP Hdr][TCP Hdr][  HTTP 資料  ]          ← Packet
          ↓ 加上 Ethernet Header + Trailer
資料鏈結層[Eth Hdr][IP Hdr][TCP Hdr][Data][CRC]     ← Frame
          ↓ 轉換為位元流
實體層    0101001101010110110100101010110101...      ← Bits
解封裝流程（接收端，由下至上）
實體層    0101001101010110110100101010110101...      ← 接收位元流
          ↓ 重組為訊框
資料鏈結層[Eth Hdr][IP Hdr][TCP Hdr][Data][CRC]     ← 讀取並驗證 Eth Header
          ↓ 移除 Ethernet 層
網路層    [IP Hdr][TCP Hdr][  HTTP 資料  ]           ← 讀取 IP Header，決定路由
          ↓ 移除 IP 層
傳輸層    [TCP Hdr][      HTTP 資料       ]          ← 讀取 TCP Header，排序重組
          ↓ 移除 TCP 層
應用層    [          HTTP 資料            ]          ← 交給 HTTP 應用程式處理

關鍵理解

封裝讓每一層可以「對話」：傳送端的 Layer 4 與接收端的 Layer 4 彷彿在直接通訊， 傳送端的 Layer 3 與接收端的 Layer 3 也是。這種對等層通訊（Peer-to-Peer Layer Communication） 的抽象，是 OSI 模型最優雅的設計思想。

Section 05

PDU：各層的資料單元名稱

每一層對「資料塊」有專屬的稱呼，稱為 PDU（Protocol Data Unit，協定資料單元）。 在討論網路問題時使用正確的術語，能避免很多混淆。

OSI 層 PDU 名稱 說明
Layer 7–5 Message（訊息） 應用層資料，如 HTTP 請求、電子郵件內容
Layer 4 Segment（區段）/ Datagram TCP 使用 Segment；UDP 使用 Datagram
Layer 3 Packet（封包） 含 IP 標頭的資料單元，是路由的基本單位
Layer 2 Frame（訊框） 含 MAC 位址標頭與 CRC 尾部的資料單元
Layer 1 Bit（位元） 最基本的 0/1 訊號單位

常見錯誤用法

日常口語中「封包」（Packet）常被泛指所有層的資料， 但嚴格來說，封包特指 Layer 3（網路層）的 PDU。 資安工具如 Wireshark 會顯示 Frame/Packet/Segment 各層的詳細資訊， 理解 PDU 的分層命名對讀懂 Wireshark 輸出非常重要。

Section 06

層間通訊原則

服務、介面與協定

每一層提供服務（Service）給上層使用， 透過介面（Interface）（SAP，Service Access Point）進行互動， 並使用自己的協定（Protocol）與對等層通訊。 這三個概念是理解層間關係的關鍵：

概念 定義 範例
Service 某層為上層提供的功能承諾 Layer 3 承諾「盡力將封包送達目的 IP」
Interface 相鄰層之間傳遞資料的規定方式 TCP Socket API（應用程式與 Layer 4 的介面）
Protocol 同一層兩端之間通訊的規則 TCP 三向交握規則、HTTP 請求格式規定

連線導向 vs. 無連線服務

各層可以提供兩種基本服務類型： 連線導向服務（Connection-Oriented）在傳輸前先建立邏輯連線， 確保資料有序可靠地送達，傳輸完成後再釋放連線，TCP 是代表例子； 無連線服務（Connectionless）直接傳送， 不建立連線也不保證可靠性，UDP 和 IP 都是無連線的。

可靠服務 vs. 不可靠服務

可靠服務確保每個 PDU 都被收到，出錯時重傳。 這需要接收端的確認（ACK）機制，產生額外開銷。 不可靠服務不提供確認，效率高但資料可能遺失。 「不可靠」在技術上是中性詞，指的是「不保證」， 適用於即時串流、遊戲等對延遲敏感但可以容忍少量遺失的應用。

Section 07

OSI 在現實中的角色

一個重要的認知是：沒有任何主流網路協定嚴格實作了完整的 OSI 七層。 TCP/IP 協定族（Day 3 深入探討）只有四層， 網際網路的實際運作靠的是 TCP/IP，而非 OSI。

OSI 模型 vs. TCP/IP 協定族對照
    OSI 模型（7層）              TCP/IP（4層）
  ┌─────────────────┐         ┌─────────────────────┐
7 │    Application  │         │                     │
  ├─────────────────┤    →    │    Application       │ HTTP, DNS, SMTP
6 │   Presentation  │         │                     │
  ├─────────────────┤         └─────────────────────┘
5 │     Session     │         ┌─────────────────────┐
  ├─────────────────┤    →    │     Transport        │ TCP, UDP
4 │    Transport    │         └─────────────────────┘
  ├─────────────────┤         ┌─────────────────────┐
3 │     Network     │    →    │     Internet         │ IP, ICMP
  ├─────────────────┤         └─────────────────────┘
2 │   Data Link     │         ┌─────────────────────┐
  ├─────────────────┤    →    │   Network Access     │ Ethernet, Wi-Fi
1 │    Physical     │         └─────────────────────┘
  └─────────────────┘

那麼 OSI 模型還有何用？它的價值在於：

共通語言：「這是 Layer 3 的問題」讓所有工程師立刻理解問題範疇， 不論他們使用什麼廠商的設備或協定。 排錯框架：系統化地由下至上（或由上至下）檢查， 避免漫無目的地亂猜。 新協定設計：設計新協定時，OSI 框架提供了清晰的定位指引。

業界實際用法

網路工程師日常說「這是 L2 的問題」就是指資料鏈結層， 「L3 設備」是路由器，「L7 負載平衡器」是能讀取 HTTP 內容並根據 URL 分流的進階設備。 OSI 層號已成為業界通用的縮語系統。

Section 08

用 OSI 模型進行網路排錯

OSI 模型最實用的應用場景之一是系統化排錯。 面對網路問題時，由下至上逐層排除是最有效率的方法， 這稱為「Bottom-Up Troubleshooting」。

排錯層 檢查項目 常用指令 / 工具
Layer 1 線路是否插好、網路燈是否亮、無線訊號強度 實體檢查、燈號、iwconfig
Layer 2 MAC 位址衝突、交換器 MAC 表、VLAN 設定 arp -a、show mac-address-table
Layer 3 IP 位址設定、子網路遮罩、預設閘道、路由表 ping、traceroute、ip route
Layer 4 連接埠是否開放、防火牆規則、TCP 連線狀態 netstat、ss、telnet <ip> <port>
Layer 7 DNS 解析、HTTP 狀態碼、憑證有效性 nslookup、curl -v、瀏覽器開發者工具

實際例子：你的電腦無法開啟某個網站。 先 ping 8.8.8.8——如果通表示 Layer 1–3 正常，問題在更高層； 如果不通，用 ping 192.168.1.1（預設閘道）—— 如果通表示 Layer 1–3 在本地 LAN 內正常，問題在 WAN 或路由； 如果連閘道都 ping 不通，很可能是 Layer 1 或 Layer 2 的問題。

另一種方法：Top-Down

從 Layer 7 往下排錯適合已知「實體層沒問題」的情況。 例如其他網站都能開、只有特定服務有問題， 這時從應用層（HTTP 錯誤碼、DNS 查詢）往下找更有效率。

Day 2 重點回顧

今天你學到了什麼

  • OSI 是 1984 年 ISO 發布的參考模型，解決不同廠商設備無法互通的問題
  • 七層由下至上：Physical、Data Link、Network、Transport、Session、Presentation、Application
  • 實體層傳訊號、資料鏈結層處理 MAC 位址與訊框、網路層負責 IP 路由
  • 傳輸層提供端對端可靠性（TCP）或快速傳輸（UDP）；上三層處理格式、會話與應用
  • 封裝：傳送端每層加上標頭；解封裝：接收端每層讀取並移除標頭
  • 各層 PDU 名稱：Bits → Frame → Packet → Segment → Message
  • TCP/IP 四層是實際運作的協定；OSI 七層是分析討論的框架
  • Bottom-Up 排錯法：從 Layer 1 實體連線開始，逐層往上檢查
