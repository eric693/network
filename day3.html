Section 01

TCP/IP 的起源與設計哲學

TCP/IP 並非由委員會在白板上設計出來的完美理論， 而是在 ARPANET 的實際工程需求中演化而成。 1973 年，Vint Cerf 與 Bob Kahn 著手解決一個核心問題： 如何讓多個使用不同技術的網路（電話線、衛星、封包無線電） 無縫互連？

他們的答案是一套革命性的設計哲學，體現在幾個核心原則中：

端對端原則（End-to-End Principle）：複雜的邏輯應置於網路邊緣 （終端主機），網路核心只做最簡單的事——轉送封包。路由器不需要理解 應用程式在做什麼，它只管把封包送到下一跳。這讓網路極度靈活： 任何新應用都能在不修改網路基礎設施的情況下部署。

盡力傳送（Best Effort Delivery）：IP 層不保證封包一定送達、 不保證順序、不保證不重複。它只「盡力」嘗試。可靠性的保障交給上層（TCP）負責。 這個設計犧牲了保證性，換來了極高的彈性與效率。

無狀態（Stateless）：路由器不記憶任何連線狀態， 每個封包都是獨立的。這讓路由器能以極高速度處理封包， 也讓網路在局部故障時能自我修復——失效節點的封包會自動繞路。

歷史轉折點

1983 年 1 月 1 日，ARPANET 全面切換至 TCP/IP，這天被稱為 「旗幟日（Flag Day）」。這是網際網路歷史上最重要的單一事件之一， 標誌著現代網際網路的正式誕生。

為何只有四層？

TCP/IP 將 OSI 的 Layer 5（會話層）和 Layer 6（表現層）合併進應用層， 因為實務上這兩層的功能往往由應用程式自行處理（如 TLS 加密、JSON 格式化）， 沒有必要在協定層面強制分離。這種務實主義是 TCP/IP 得以廣泛採用的原因之一。

Section 03

網路存取層（Network Access Layer）

網路存取層對應 OSI 的 Layer 1（實體層）與 Layer 2（資料鏈結層）。 它負責在同一實體網路區段內，將封包包裝成訊框（Frame） 並透過物理介質送出。這一層與具體的硬體技術緊密耦合。

乙太網路（Ethernet, IEEE 802.3）

乙太網路是目前最主流的有線 LAN 技術。它使用 CSMA/CD（Carrier Sense Multiple Access with Collision Detection） 機制管理共享媒介的存取：傳送前先「聽」通道是否空閒（Carrier Sense）， 若碰到碰撞（Collision Detection）則退避隨機時間後重試。 不過現代的全雙工交換式乙太網路已幾乎消除碰撞問題。

乙太網路訊框結構包含：前置碼（Preamble）8 bytes 用於同步、 目的 MAC 6 bytes、來源 MAC 6 bytes、 類型/長度（EtherType） 2 bytes（如 0x0800 代表 IPv4、 0x86DD 代表 IPv6）、資料 46–1500 bytes、 FCS（幀校驗序列） 4 bytes 用於錯誤偵測。

ARP：橋接 Layer 2 與 Layer 3

ARP（Address Resolution Protocol）解決一個關鍵問題： 已知目的 IP 位址，如何找到對應的 MAC 位址？ 傳送方會在區域網路廣播一個 ARP Request： 「誰是 192.168.1.1？請告訴我你的 MAC 位址。」 擁有該 IP 的設備回應 ARP Reply，傳送方將結果快取在 ARP Cache 中。

ARP 解析流程
主機 A (192.168.1.10)         主機 B (192.168.1.1)
        │                              │
        │── ARP Request（廣播）────────>│  "誰有 192.168.1.1？"
        │   Dst MAC: FF:FF:FF:FF:FF:FF  │
        │                              │
        │<── ARP Reply（單播）──────────│  "我是！我的 MAC = AA:BB:CC:DD:EE:FF"
        │                              │
  ARP Cache 更新：192.168.1.1 → AA:BB:CC:DD:EE:FF

ARP 欺騙（ARP Spoofing）

ARP 沒有認證機制，攻擊者可以發送偽造的 ARP Reply， 將閘道器的 IP 對應到自己的 MAC， 使區域網路內的流量都經過攻擊者的機器——這就是 ARP 欺騙攻擊（Man-in-the-Middle）。 Day 13 的網路安全課程會進一步討論防護措施。

Section 04

網際網路層與 IP 協定

網際網路層是 TCP/IP 協定族的核心，IP（Internet Protocol） 是這一層最重要的協定。IP 的職責是：給每個封包標記來源與目的的邏輯位址（IP 位址）， 並讓路由器根據這個位址決定封包的傳遞路徑。

IPv4 標頭結構

IPv4 標頭最小 20 bytes（無選項時），包含路由所需的所有資訊。 理解標頭各欄位，是理解路由、分割、TTL 等機制的前提。

IPv4 Header（每列 32 bits）
Version 4 bits
IHL 4 bits
DSCP 6 bits
ECN 2 bits
Total Length 16 bits
Identification 16 bits
Flags 3 bits
Fragment Offset 13 bits
TTL 8 bits
Protocol 8 bits
Header Checksum 16 bits
Source IP Address 32 bits
Destination IP Address 32 bits
Options (若有) + Padding variable
欄位 大小 說明
Version 4 bits IPv4 = 4，IPv6 = 6
IHL 4 bits Header 長度，單位為 4 bytes；無選項時值為 5（= 20 bytes）
TTL 8 bits 每經過一個路由器遞減 1，歸零時丟棄封包，防止無限迴圈。Windows 預設 128，Linux 預設 64
Protocol 8 bits 上層協定代碼：TCP = 6，UDP = 17，ICMP = 1，OSPF = 89
Src / Dst IP 各 32 bits 來源與目的 IPv4 位址，是路由決策的依據
Flags / Fragment Offset 3 + 13 bits 控制封包分割（Fragmentation）。DF（Don't Fragment）旗標設起時路由器不得分割此封包
Checksum 16 bits 只校驗 IP Header（不含 Payload）。每跳都要重新計算，因 TTL 每跳遞減

IPv4 vs. IPv6 的主要差異

IPv4 使用 32 位元位址，提供約 43 億個位址，已於 2011 年分配耗盡（IANA 層級）。 IPv6 使用 128 位元位址，理論容量為 3.4 × 10³⁸ 個，幾乎是無限的。 除了位址空間，IPv6 還帶來其他改進：

特性 IPv4 IPv6
位址長度 32 bits（4 bytes） 128 bits（16 bytes）
表示方式 點十進位：192.168.1.1 冒號十六進位：2001:db8::1
Header 大小 20–60 bytes（可變） 固定 40 bytes（擴展標頭另計）
Header Checksum 有（每跳計算） 無（由 Layer 2 / 4 負責）
分割（Fragmentation） 路由器可分割 只有來源主機可分割（路由器不分割）
廣播 支援廣播位址 無廣播，改用多播（Multicast）
NAT 幾乎必須使用 設計上不需要 NAT
IPsec 選用 設計內建（雖非強制）

ICMP：網路的診斷工具

ICMP（Internet Control Message Protocol） 是 IP 的「輔助協定」，用於傳遞錯誤訊息與診斷資訊。 它沒有連接埠概念，直接封裝在 IP 封包中（Protocol = 1）。

ping 指令使用 ICMP Echo Request（Type 8）與 Echo Reply（Type 0）來測試可達性與 RTT（往返時間）。 traceroute 則利用逐步增加 TTL 值， 讓每跳路由器回傳 ICMP Time Exceeded（Type 11）訊息， 以此探測封包路徑上的每一個中繼節點。

Section 05

傳輸層：TCP 與 UDP

傳輸層在 IP 提供的「盡力傳送」基礎上，提供兩種截然不同的服務。 選擇 TCP 還是 UDP，是每個網路應用設計的核心決策。

TCP — 傳輸控制協定

連線導向：傳輸前需三向交握建立連線

可靠傳輸：每個 Segment 都需 ACK 確認，逾時重傳

有序傳遞：序號（Sequence Number）確保資料按順序組裝

流量控制：滑動視窗防止接收方被淹沒

壅塞控制：偵測網路壅塞時主動降速

適用：HTTP、FTP、SSH、電子郵件（正確性比速度重要）

UDP — 使用者資料報協定

無連線：直接傳送，不需建立連線

不可靠：不確認、不重傳、不保證順序

低延遲：無握手開銷，適合即時應用

輕量：Header 只有 8 bytes（TCP 是 20 bytes）

可廣播：支援廣播與多播傳輸

適用：DNS、視訊串流、線上遊戲、VoIP、DHCP

TCP 三向交握（Three-Way Handshake）

TCP 連線的建立需要三個步驟，確保雙方都準備好且序號已同步：

TCP 連線建立與終止
    Client                          Server
      │                                │
      │─── SYN（seq=x）───────────────>│  客戶端：我想建立連線，我的序號是 x
      │                                │
      │<── SYN-ACK（seq=y, ack=x+1）──│  伺服器：收到！我的序號是 y，期待你的 x+1
      │                                │
      │─── ACK（ack=y+1）─────────────>│  客戶端：確認！期待你的 y+1
      │                                │
      │         ← 資料傳輸 →          │
      │                                │
      │─── FIN ───────────────────────>│  關閉：客戶端不再傳送
      │<── ACK ────────────────────────│
      │<── FIN ────────────────────────│  伺服器也準備關閉
      │─── ACK ───────────────────────>│
      │                                │  連線完全關閉（四向揮手）

SYN Flood 攻擊

攻擊者大量發送 SYN 封包但不完成三向交握， 使伺服器的半連線佇列（SYN Queue）塞滿而無法接受正常連線。 這是最常見的 DoS 攻擊手法之一，防禦方式包含 SYN Cookie 技術。

Section 07

連接埠號碼（Port Numbers）

連接埠（Port）是傳輸層的多工機制。 一台主機上可能同時執行 Web 伺服器、SSH 伺服器、資料庫等多個服務， IP 位址只能識別主機，連接埠號碼才能識別主機上的特定服務或應用程式。 連接埠是 16 位元整數，範圍 0–65535。

IANA 將連接埠號碼分為三個範圍： Well-Known Ports（0–1023）由 IANA 統一分配給標準服務， 在 Unix 系統上通常需要 root 權限才能監聽； Registered Ports（1024–49151）供特定應用程式登記使用； Dynamic / Ephemeral Ports（49152–65535）由作業系統臨時分配給用戶端連線。

Socket 的概念

一個 TCP 連線由四元組唯一識別： （來源 IP, 來源 Port, 目的 IP, 目的 Port）。 這個四元組稱為 Socket Pair。 同一台伺服器的 443 port 可以同時服務數萬個連線， 因為每個連線的來源 IP 或來源 Port 都不同，四元組各自唯一。

Section 08

封包在網際網路中的完整旅程

把前幾節的知識整合，讓我們追蹤一個 HTTP 請求從你的瀏覽器 到 Web 伺服器的完整路徑，觀察每一層如何協作。

一個 HTTP GET 請求的完整封裝與傳遞流程
你的電腦（192.168.1.10）                        Web 伺服器（93.184.216.34）
━━━━━━━━━━━━━━━━━━━━━━━━━━                      ━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[應用層]  瀏覽器產生 HTTP GET /index.html
          ↓
[傳輸層]  加上 TCP Header
          Src Port: 54321（臨時）
          Dst Port: 80（HTTP）
          Seq: 1001
          ↓
[網路層]  加上 IP Header
          Src IP: 192.168.1.10（你的 IP）
          Dst IP: 93.184.216.34（伺服器 IP）
          TTL: 64
          ↓
[網路存取層] 加上 Ethernet Header
          Src MAC: 你的 NIC MAC
          Dst MAC: 閘道器的 MAC（透過 ARP 取得）
          ↓
[實體層]  轉為電訊號，送上網線

──── 閘道器（你家路由器）────────────────────────────────────────────

  接收訊框 → 解封裝到 IP 層
  查詢路由表 → 下一跳：ISP 路由器 1.2.3.4
  重新封裝：更新 Dst MAC = ISP 路由器的 MAC
  TTL: 64 → 63
  送出

──── 中途路由器（重複上述，TTL 每跳 -1）─────────────────────────────

──── 伺服器端 ───────────────────────────────────────────────────────

[實體層]  接收電訊號
          ↓
[網路存取層] 讀取 Ethernet Frame，確認 Dst MAC = 自己的 MAC
          ↓
[網路層]  讀取 IP Header，確認 Dst IP = 93.184.216.34（是我！）
          ↓
[傳輸層]  讀取 TCP Header，Dst Port = 80 → 交給 HTTP 服務程序
          確認 Seq，發送 ACK
          ↓
[應用層]  HTTP 伺服器處理請求，回傳 200 OK + HTML

注意一個關鍵細節：在整個傳輸過程中， IP 位址（Src/Dst IP）始終不變，是端對端的全程標識； 但MAC 位址（Src/Dst MAC）在每一跳都會更換， 只在相鄰的兩個設備之間有意義。 這正是 IP（全域邏輯位址）與 MAC（本地實體位址）分工的精髓。

Day 3 重點回顧

今天你學到了什麼

  • TCP/IP 的三大設計哲學：端對端原則、盡力傳送、無狀態路由
  • TCP/IP 四層：Network Access → Internet → Transport → Application
  • ARP 在同一 LAN 內將 IP 解析為 MAC 位址，無認證機制易受欺騙

  • 封包旅程：IP 位址全程不變；MAC 位址每跳更換，兩者分工明確
