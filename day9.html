Section 01

DNS 的誕生與設計目標

1983 年以前，整個網際網路（當時稱 ARPANET）的主機名稱解析靠一個叫做 HOSTS.TXT 的純文字檔案——Stanford Research Institute（SRI） 維護這份清單，所有主機定期下載更新。這個方式在只有幾百台主機時運作良好， 但當主機數量急速增長，問題接踵而來：更新頻率跟不上、 名稱衝突越來越多、單點故障風險極高。

Paul Mockapetris 在 1983 年提出 DNS（Domain Name System）， 定義於 RFC 1034 和 RFC 1035，設計目標有三： 分散式（Distributed）——沒有單一中央資料庫； 階層式（Hierarchical）——以樹狀結構組織命名空間； 快取（Cached）——避免重複查詢，提升效能。

DNS 的規模

今日 DNS 是世界上最大的分散式資料庫之一， 每天處理超過數兆次查詢。 根伺服器系統（Root Server System）由全球 13 個邏輯根伺服器（A 到 M） 組成，實際上透過任播（Anycast）技術在全球部署超過 1,700 個節點， 任何一個根伺服器的「毀滅」都不會讓 DNS 停擺。

Section 02

DNS 階層架構

DNS 命名空間是一棵倒置的樹，頂端是根（Root，以「.」表示）， 向下依次是頂級域（TLD）、 二級域（Second-Level Domain）、 子域（Subdomain），最終到達主機名稱。 完整的域名從右讀到左，最右端是根，最左端是最具體的主機。

DNS 命名空間樹狀結構
                            .（根，Root）
                           /     |      \
             ┌────────────┐  ┌───┐  ┌───────────────┐
             com           org    net      tw（國碼 TLD）
            / | \           |               /     \
         google apple amazon wikipedia    com.tw  edu.tw
           |                                |        |
          www                           example   ntu.edu.tw
                                           |           |
                                        mail.example  www

  完整域名（FQDN）：
    www.google.com.         ← 注意尾部的「.」代表根，通常省略
    mail.example.com.tw.
    www.ntu.edu.tw.

  TLD 類型：
    通用 TLD（gTLD）：.com  .org  .net  .edu  .gov  .mil
    國碼 TLD（ccTLD）：.tw  .jp  .uk  .de  .cn  .us
    新 TLD（New gTLD）：.app  .dev  .io  .ai  .shop（2012 年後開放）

DNS 伺服器的四種角色

角色說明例子
根伺服器（Root Server） 知道所有 TLD 的權威伺服器位址，是查詢的起點。全球 13 個邏輯根（A–M），Anycast 擴展至 1,700+ 節點 a.root-servers.net (198.41.0.4)
TLD 伺服器（TLD Server） 管理特定 TLD（如 .com）下所有域名的委派資訊，知道哪個 NS 是 example.com 的權威伺服器 a.gtld-servers.net（管理 .com）
權威伺服器（Authoritative Server） 儲存特定區域（Zone）的實際 DNS 記錄，是最終答案的來源。回應帶有 AA（Authoritative Answer）旗標 ns1.google.com（管理 google.com）
遞迴解析器（Recursive Resolver） 替客戶端執行完整的遞迴查詢流程，快取結果。通常由 ISP 或公共 DNS 服務提供 8.8.8.8（Google）、1.1.1.1（Cloudflare）、168.95.1.1（HiNet）
Section 03

DNS 解析完整流程

以查詢 www.example.com 的 IP 為例， 假設本地 DNS Cache 與解析器 Cache 都沒有記錄， 完整的遞迴解析流程如下：

www.example.com 解析流程（完整 8 步）
1
瀏覽器 / 作業系統快取查詢
先查瀏覽器自身的 DNS Cache；若無，查作業系統的 DNS Cache（Windows 可用 ipconfig /displaydns 查看）。若命中則直接返回，流程結束。
2
查詢遞迴解析器（Recursive Resolver）
Cache 未命中，作業系統向設定的 DNS 伺服器（如 8.8.8.8）發出 UDP 查詢。解析器若有快取則直接回應；若無則開始遞迴查詢。
3
解析器 → 根伺服器
解析器向根伺服器詢問「誰管 .com？」根伺服器回傳 .com TLD 伺服器的 NS 記錄（非遞迴答案，僅告知下一步去哪問）。
4
解析器 → .com TLD 伺服器
解析器再詢問 .com TLD 伺服器「誰管 example.com？」TLD 伺服器回傳 example.com 的 NS 記錄（如 ns1.example.com）。
5
解析器 → example.com 權威伺服器
解析器向 ns1.example.com 詢問「www.example.com 的 A 記錄是什麼？」權威伺服器回傳最終答案（IP 位址）。回應帶 AA 旗標。
6
解析器快取結果
解析器將答案依 TTL 快取，下次相同查詢直接回應，不再遞迴。
7
解析器回傳給客戶端
解析器將 IP 位址回傳給作業系統；作業系統快取後交給瀏覽器。
8
瀏覽器建立 TCP 連線
拿到 IP 後，瀏覽器才開始建立 TCP/TLS 連線，發出 HTTP 請求。DNS 解析本身對使用者是透明的，通常在 10–50ms 內完成。
遞迴查詢 vs 迭代查詢
遞迴查詢（Recursive Query）：客戶端要求解析器完整解決
  客戶端 ──────────────────> 遞迴解析器（「幫我查完」）
  客戶端 <────────────────── 解析器（「答案是 X.X.X.X」）

迭代查詢（Iterative Query）：解析器每步都只得到「下一步去哪問」
  解析器 ──> 根伺服器      → 「去問 a.gtld-servers.net」
  解析器 ──> TLD 伺服器    → 「去問 ns1.example.com」
  解析器 ──> 權威伺服器    → 「答案是 93.184.216.34」

實際運作：
  客戶端 ←──遞迴──→ 解析器（客戶端的角度：一問一答）
  解析器 ←──迭代──→ 各層伺服器（解析器的角度：逐步詢問）
Section 04

DNS 訊息格式

DNS 查詢與回應使用相同的訊息格式， 分為 Header、Question、 Answer、Authority、 Additional 五個區段。 每個訊息大小通常在 512 bytes 以內（UDP 傳輸）， 超過則改用 TCP 或啟用 EDNS0（最大 4096 bytes）。

DNS 訊息 Header（12 bytes）
Transaction ID 查詢識別碼，回應必須與查詢相同（用於匹配問答對） 16 bits
Flags（旗標） QR（查詢/回應）、Opcode、AA、TC、RD、RA、RCODE 等 16 bits
QDCOUNT Question 區段的問題數量（通常為 1） 16 bits
ANCOUNT Answer 區段的 RR 數量 16 bits
NSCOUNT Authority 區段的 NS RR 數量 16 bits
ARCOUNT Additional 區段的 RR 數量 16 bits

重要 Flags 位元說明

旗標全名說明
QRQuery/Response0 = 查詢，1 = 回應
AAAuthoritative Answer1 = 此回應來自權威伺服器（非快取）
TCTruncated1 = 回應超過 512 bytes 被截斷，客戶端應改用 TCP 重試
RDRecursion Desired1 = 客戶端希望解析器執行遞迴查詢
RARecursion Available1 = 伺服器支援遞迴查詢
RCODEResponse Code0=NOERROR、1=FORMERR、2=SERVFAIL、3=NXDOMAIN（域名不存在）、5=REFUSED

NXDOMAIN（RCODE 3）的意義

當查詢的域名根本不存在時，權威伺服器回傳 NXDOMAIN（Non-Existent Domain）。 這與「域名存在但沒有該類型記錄（NOERROR with empty answer）」是不同的情況。 部分 ISP 的 DNS 會把 NXDOMAIN 劫持為指向廣告頁面， 這是公共 DNS（8.8.8.8、1.1.1.1）受歡迎的原因之一。

Section 05

資源記錄（Resource Record）類型

DNS 儲存的資料以資源記錄（RR, Resource Record）為單位， 每筆記錄包含：名稱（Name）、類型（Type）、類別（Class，幾乎都是 IN = Internet）、 TTL（存活時間）、資料長度（RDLENGTH）和實際資料（RDATA）。

A
Address Record（IPv4）
將域名對應到 IPv4 位址。最基本、最常用的記錄類型。
www.example.com. 300 IN A 93.184.216.34
AAAA
Address Record（IPv6）
將域名對應到 IPv6 位址。名稱「四個 A」因為 IPv6 是 IPv4 的四倍長。
www.example.com. 300 IN AAAA 2606:2800:220:1:248:1893:25c8:1946
CNAME
Canonical Name（別名）
將一個域名指向另一個域名（別名）。解析器會繼續查詢 CNAME 指向的域名直到找到 A/AAAA 記錄。
www.example.com. 3600 IN CNAME lb.example.com.
MX
Mail Exchange（郵件交換）
指定負責接收某域名電子郵件的伺服器。含優先權值（數字越小優先）。
example.com. 3600 IN MX 10 mail.example.com.
NS
Name Server（名稱伺服器）
指定某區域的權威 DNS 伺服器。至少需要兩筆（主、備）以確保高可用性。
example.com. 86400 IN NS ns1.example.com.
SOA
Start of Authority（授權起始）
每個區域必有一筆，記錄主要名稱伺服器、管理員信箱、序號、刷新時間等區域參數。
example.com. IN SOA ns1 admin 2024010101 3600 900 604800 300
TXT
Text Record（文字記錄）
任意文字資料。廣泛用於 SPF（防郵件偽造）、DKIM（郵件簽名）、DMARC、網站驗證等。
example.com. IN TXT "v=spf1 include:_spf.google.com ~all"
PTR
Pointer Record（反查記錄）
IP 位址反查域名（Reverse DNS）。存在特殊的 in-addr.arpa 區域中。
34.216.184.93.in-addr.arpa. IN PTR www.example.com.
SRV
Service Record（服務記錄）
指定特定服務的主機、連接埠與優先權。用於 SIP、XMPP、Kubernetes 等服務發現。
_sip._tcp.example.com. IN SRV 10 60 5060 sipserver.example.com.
CAA
Certification Authority Authorization
指定哪些 CA 可以為此域名簽發 SSL/TLS 憑證，防止錯誤簽發。
example.com. IN CAA 0 issue "letsencrypt.org"
DNSKEY
DNS Public Key
DNSSEC 使用，儲存區域的公鑰，用於驗證 RRSIG 數位簽章。
example.com. IN DNSKEY 257 3 8 AwEAAb...
HTTPS
HTTPS Service Binding
較新的記錄類型（RFC 9460），提供 HTTPS 連線的相關參數（如 ALPN、ECH 公鑰）。
example.com. IN HTTPS 1 . alpn="h3,h2"

CNAME 的限制

CNAME 不能與其他記錄共存於同一個名稱上。 最典型的問題是：根域名（Apex Domain，如 example.com，不帶 www） 必須有 SOA 和 NS 記錄，因此不能設定 CNAME。 CloudFlare 等 DNS 服務提供商用「CNAME Flattening」（偽裝成 A 記錄回應）解決此限制。

Section 06

DNS 快取與 TTL

每筆 DNS 記錄都有一個 TTL（Time To Live）值， 單位是秒，表示這筆記錄可以被快取的最長時間。 在 TTL 到期之前，解析器不需要再向權威伺服器查詢， 直接從快取回應，大幅減少 DNS 查詢流量。

TTL 範圍典型用途優缺點
0–60 秒 負載均衡、主動健康檢查（如 Route 53 Health Check） IP 變更即時生效；但每次都要查詢，延遲較高
300 秒（5 分鐘） 需要快速切換的服務（如 DDoS 緩解、CDN 切換） 準備遷移前臨時降低 TTL 的常用值
3600 秒（1 小時） 一般 Web 服務的常用預設值 平衡快取效能與變更靈活性
86400 秒（1 天） 穩定不常變動的記錄（如 NS、MX） 快取命中率高；IP 變更後舊快取存活長達 24 小時

DNS 傳播延遲（Propagation Delay）

當你修改 DNS 記錄時，並非所有使用者立刻看到新值。 全球各地的遞迴解析器可能快取了舊的 TTL 值， 必須等舊記錄過期後才會重新查詢。 這個過渡期稱為 DNS 傳播延遲， 長度等於舊記錄的 TTL 值。 這就是為什麼在計畫重大 IP 變更前，應提前幾天將 TTL 降低到 300 秒以下， 以縮短傳播延遲。

TTL 快取流程示意
  t=0：第一次查詢 www.example.com（TTL=3600）
       解析器 ──遞迴──> 根 → TLD → 權威 → 得到 IP：93.184.216.34
       解析器快取：93.184.216.34，過期時間 = t + 3600

  t=100：第二次查詢
       解析器直接從快取回應，TTL 剩餘值 = 3600 - 100 = 3500
       （客戶端收到的 TTL 是剩餘值，不是原始值）

  t=3600：TTL 到期
       快取失效，下次查詢重新遞迴

  修改 IP 前的最佳實踐：
    Day -3：將 TTL 從 86400 降到 300
    Day  0：修改 A 記錄 IP 位址
    Day +1：確認生效後，視需要調回較高 TTL
Section 07

DNS 區域與權威伺服器

區域（Zone）是 DNS 命名空間的管理單位， 是一個組織或個人可以控制的域名範圍。 區域的所有記錄儲存在區域檔案（Zone File）中， 由主要名稱伺服器（Primary NS）維護， 並定期同步給次要名稱伺服器（Secondary NS）以確保高可用性。

區域傳輸（Zone Transfer）

當主要 NS 的記錄更新時，次要 NS 透過區域傳輸同步最新資料：

類型說明
AXFR（全量傳輸） 次要 NS 向主要 NS 請求整個區域的所有記錄。透過 TCP Port 53 傳輸（資料量大）
IXFR（增量傳輸） 只傳輸自上次同步後變更的記錄，效率更高。比對 SOA 記錄的序號（Serial）判斷是否有更新

區域傳輸的安全風險

若 DNS 伺服器對任何人開放 AXFR 查詢（而非只允許次要 NS）， 攻擊者可以下載整個區域的所有記錄， 獲取組織內部所有主機名稱與 IP 位址， 這是信息蒐集（Reconnaissance）階段的強力工具。 應在防火牆與 DNS 設定中限制 AXFR 只允許授權的次要 NS IP 存取。

典型區域檔案結構（BIND 格式）
$ORIGIN example.com.
$TTL 3600
; SOA 記錄（每個區域必須有且只有一筆）
@    IN    SOA    ns1.example.com. admin.example.com. (
                  2024010101  ; Serial（YYYYMMDDNN，每次修改遞增）
                  3600        ; Refresh（次要 NS 多久查詢一次更新）
                  900         ; Retry（Refresh 失敗後重試間隔）
                  604800      ; Expire（多久後次要 NS 停止回應）
                  300 )       ; Minimum TTL（負快取 TTL）

; NS 記錄（至少兩筆）
@    IN    NS     ns1.example.com.
@    IN    NS     ns2.example.com.

; A 記錄
@    IN    A      93.184.216.34    ; example.com 本身
www  IN    A      93.184.216.34
mail IN    A      93.184.216.35

; MX 記錄
@    IN    MX     10 mail.example.com.
@    IN    MX     20 mail2.example.com.   ; 備援

; CNAME 記錄
ftp  IN    CNAME  www.example.com.

; TXT 記錄（SPF）
@    IN    TXT    "v=spf1 mx -all"
Section 08

進階 DNS：反查、委派、萬用字元

反向 DNS 查詢（Reverse DNS / rDNS）

正向查詢是「域名 → IP」；反向查詢是「IP → 域名」，使用特殊的 in-addr.arpa 域（IPv4） 或 ip6.arpa 域（IPv6）。 IPv4 位址反轉後加上 .in-addr.arpa： 93.184.216.34 的反查域名是 34.216.184.93.in-addr.arpa， 對應的 PTR 記錄指向 www.example.com。 反向 DNS 常用於郵件伺服器驗證（防止垃圾郵件）和網路日誌的主機識別。

委派（Delegation）

DNS 的分散式管理透過委派實現： 父區域透過 NS 記錄將某個子域的管理權移交給另一組名稱伺服器。 例如 .com TLD 伺服器委派 example.com 給 ns1.example.com， example.com 又可以委派 sub.example.com 給另一組伺服器， 形成完整的分散管理體系。

伴隨 NS 委派，父區域通常也提供黏合記錄（Glue Record）： 當委派的 NS 主機本身就在被委派的區域內時（如 ns1.example.com 管理 example.com）， 若沒有黏合記錄，解析器會陷入「要問 ns1.example.com 在哪， 但 ns1.example.com 的 IP 要問 ns1.example.com」的循環。 黏合記錄直接在父區域提供這些 NS 的 A 記錄，打破循環。

萬用字元記錄（Wildcard Record）

使用 * 作為最左側標籤， 可以匹配任何未明確定義的子域名：

萬用字元記錄的應用
*.example.com.  IN  A  203.0.113.1

效果：
  abc.example.com    → 203.0.113.1  ✓
  xyz.example.com    → 203.0.113.1  ✓
  www.example.com    → 優先使用明確的 A 記錄（若存在），萬用字元不覆蓋明確記錄

常見用途：
  - 泛域名 SSL 憑證（*.example.com）搭配萬用字元 A/CNAME
  - SaaS 服務讓客戶使用自訂子域名（*.app.saas.com → 統一 IP）
  - 捕獲拼字錯誤的子域名到特定頁面
Section 09

DNS 安全威脅

DNS 快取中毒（DNS Cache Poisoning / Spoofing）

攻擊者在解析器向權威伺服器查詢時，偽造回應搶先抵達， 將惡意的 IP 植入解析器的快取中。 之後所有查詢該域名的使用者都會被導向攻擊者的伺服器， 直到 TTL 到期。

2008 年，研究員 Dan Kaminsky 發現了一個嚴重的快取中毒漏洞： 攻擊者可以在很短時間內以高速猜測 Transaction ID（16 bits，只有 65,536 種可能）， 成功率遠高於預期。此事件促使全球 DNS 解析器緊急實施來源連接埠隨機化（Source Port Randomization）， 將猜測空間從 65,536 擴大到 65,536 × 65,536 ≈ 40 億種組合。

DNS 劫持（DNS Hijacking）

攻擊者直接篡改使用者設備上的 DNS 設定（如透過惡意軟體修改 /etc/resolv.conf 或 Windows 登錄）、入侵路由器修改 DHCP 推送的 DNS 伺服器， 或入侵域名登記商帳號修改 NS 記錄， 讓所有查詢都由攻擊者控制的 DNS 伺服器回應。

DNS 放大攻擊（DNS Amplification DDoS）

已在 Day 8（UDP 安全）詳細介紹。攻擊者偽造受害者 IP， 向 open resolver 發出 ANY 查詢，回應放大 28–54 倍打向受害者。 防禦：關閉 open resolver、啟用 Response Rate Limiting（RRL）。

DNS 隧道（DNS Tunneling）

將任意資料（C2 指令、資料外洩內容）編碼進 DNS 查詢的子域名 （如 aGVsbG8=.attacker.com）和 TXT 回應中， 繞過只允許 DNS 流量的防火牆，建立隱蔽的命令與控制通道。 即使在嚴格管控的網路環境，DNS 通常也被允許通過。 偵測方式：監控異常長的域名、高頻率查詢、大量 TXT 查詢等異常行為。

子域名接管（Subdomain Takeover）

組織的 CNAME 記錄指向第三方服務（如 GitHub Pages、Heroku、S3）， 但若該第三方服務的帳號被刪除或取消， CNAME 目標不再存在。 攻擊者可以在第三方服務上建立同名資源， 接管這個子域名，在上面建立釣魚頁面或竊取 Cookie。 防護：定期審計所有 CNAME 記錄，移除不再使用的指向。

Section 10

DNSSEC 與加密 DNS

DNSSEC：DNS 安全擴展

DNSSEC（DNS Security Extensions，RFC 4033–4035） 透過數位簽章解決 DNS 快取中毒問題。 每個區域的記錄都由私鑰簽名，解析器可以用公鑰驗證回應是否被篡改。 DNSSEC 建立了從根到葉的信任鏈（Chain of Trust）：

DNSSEC 信任鏈
  根區域 (.) 的 DNSKEY（硬編碼在解析器中的信任錨）
      │
      │ 簽名 .com 的 DS 記錄（委派簽名者記錄）
      ↓
  .com 區域的 DNSKEY
      │
      │ 簽名 example.com 的 DS 記錄
      ↓
  example.com 區域的 DNSKEY
      │
      │ RRSIG（Resource Record Signature，每筆記錄的簽名）
      ↓
  www.example.com 的 A 記錄 ← 解析器驗證簽章，確認未被篡改

  新增記錄：DS（委派簽名者）、DNSKEY（區域公鑰）、RRSIG（簽章）、NSEC/NSEC3（證明域名不存在）

DNSSEC 的缺點：部署複雜、Zone Enumeration 問題（NSEC 記錄暴露所有子域名，NSEC3 部分緩解）、 簽名增加回應體積（可能超過 UDP 限制，觸發 TCP fallback）。 全球 DNSSEC 部署率仍然偏低，但 .gov、.se、.br 等 TLD 已強制要求。

加密 DNS：DoT 與 DoH

傳統 DNS 使用明文 UDP/TCP Port 53，ISP 和中間人可以監看所有查詢， 實施 DNS 劫持或監控使用者的瀏覽行為。 加密 DNS 解決了這個隱私問題：

協定傳輸連接埠特點
DNS over TLS（DoT） TCP + TLS 853 明確區分加密 DNS 流量（可被網路管理員識別並管控），適合企業環境
DNS over HTTPS（DoH） HTTPS（TCP/QUIC） 443 DNS 查詢混入一般 HTTPS 流量，難以被封鎖或識別。Firefox、Chrome 預設啟用
DNS over QUIC（DoQ） QUIC（UDP） 853 最新標準（RFC 9250），結合 DoT 的隔離性與 QUIC 的低延遲，逐漸普及中

DoH 的爭議

DoH 雖然保護使用者隱私，但它把 DNS 控制權從 ISP 和企業 IT 轉移給 DoH 提供者（如 Google、Cloudflare）。 在企業環境中，DoH 會繞過企業的 DNS 過濾（Content Filtering），造成安全管理困難。 部分企業使用 HTTPS 檢查（SSL Inspection）或 MDM 策略強制指定企業 DoH 伺服器。

Day 9 重點回顧

今天你學到了什麼

  • DNS 取代 HOSTS.TXT，設計原則：分散式、階層式、快取
  • DNS 階層：根（.）→ TLD（.com/.tw）→ 二級域（example.com）→ 主機名稱
  • 四種角色：根伺服器、TLD 伺服器、權威伺服器、遞迴解析器
  • 解析流程：瀏覽器快取 → OS 快取 → 解析器 → 遞迴查詢根/TLD/權威
  • 客戶端對解析器遞迴；解析器對各層伺服器迭代
  • RR 類型：A（IPv4）、AAAA（IPv6）、CNAME（別名）、MX（郵件）、NS、SOA、TXT、PTR（反查）
  • TTL 決定快取時間；修改 IP 前應提前降低 TTL 縮短傳播延遲
  • AXFR/IXFR 區域傳輸需限制 IP，否則洩漏所有記錄
  • DNS 威脅：快取中毒（Dan Kaminsky 漏洞）、劫持、放大攻擊、隧道、子域名接管
  • DNSSEC 透過數位簽章建立信任鏈；DoT/DoH/DoQ 為 DNS 查詢加密保護隱私
