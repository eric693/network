Section 01

TCP 的設計目標

TCP（Transmission Control Protocol）定義於 RFC 793（1981 年）， 是 TCP/IP 協定族傳輸層最核心的協定。 它的設計目標可以用四個詞概括： 可靠（Reliable）、有序（Ordered）、 無重複（No Duplicate）、位元組串流（Byte Stream）。

TCP 在 IP 層的「盡力傳送」（封包可能遺失、亂序、重複）之上， 透過一系列精密機制實現對應用程式看起來像在直接讀寫一條 可靠管道的抽象。應用程式呼叫 send() 與 recv()， 不需要關心底層的封包切割、重傳、排序——這些全由 TCP 處理。

問題 IP 的行為 TCP 的解決機制
封包遺失 直接丟棄，不通知 ACK 確認 + 逾時重傳（Retransmission）
封包亂序 依抵達順序交付 序號（Sequence Number）排序重組
封包重複 可能重複交付 序號去重，丟棄已接收的重複封包
接收方過慢 不限速，可能淹沒接收方 接收視窗（rwnd）流量控制
網路壅塞 不感知壅塞，繼續傳送 壅塞視窗（cwnd）壅塞控制

TCP 的代價

可靠性不是免費的。TCP 的握手需要至少 1 個 RTT 才能開始傳輸資料， ACK 確認機制增加了往返延遲，壅塞控制在丟包時主動降速。 這些機制讓 TCP 不適合對延遲極度敏感的應用（如 DNS 查詢、線上遊戲）， 這類應用會選擇 UDP（Day 8）。

Section 02

TCP 標頭結構

TCP 標頭最小 20 bytes（無選項時）， 最大 60 bytes（含選項欄位）。 每個欄位都服務於特定的 TCP 機制，理解標頭是理解整個 TCP 的基礎。

TCP Header（每列 32 bits = 4 bytes）
Source Port（來源連接埠） 傳送方的連接埠號 16 bits
Destination Port（目的連接埠） 接收方的連接埠號（如 80、443） 16 bits
Sequence Number（序號） 此 Segment 第一個位元組在整個位元組串流中的位置編號 32 bits
Acknowledgment Number（確認號） 期待接收的下一個位元組序號（表示此號之前的都已收到） 32 bits
Data Offset Header 長度（4 byte 為單位） 4 bits
Reserved 保留（全為 0） 4 bits
Control Flags（控制旗標） CWR、ECE、URG、ACK、PSH、RST、SYN、FIN 8 bits
Window Size（視窗大小） 接收方還能接收的位元組數（rwnd），用於流量控制 16 bits（最大 65,535 bytes，可用 Window Scale 選項擴展）
Checksum 覆蓋 TCP Header + Data + IP 假標頭（Pseudo Header） 16 bits
Urgent Pointer 當 URG 旗標設起時，指向緊急資料的偏移量 16 bits
Options（選項） MSS、Window Scale、SACK、Timestamps、NOP 等選項 0–40 bytes（需 4 bytes 對齊）

重要 TCP Options

選項 全名 用途
MSS Maximum Segment Size 在 SYN 中宣告自己能接收的最大 Segment 大小，通常為 MTU - 40（1460 bytes for Ethernet）
Window Scale RFC 7323 將 Window Size 左移 N 位元，允許視窗最大達 1 GB（解決高頻寬長距離網路的效率問題）
SACK Selective Acknowledgment 允許接收方告知傳送方哪些不連續的區塊已收到，讓重傳只補缺漏的部分，而非全部重傳
Timestamps RFC 7323 精確測量 RTT（回傳時間戳記的 echo），以及 PAWS（防止舊封包混入新連線）
Section 03

控制旗標（Control Flags）

TCP 標頭中的 8 個控制旗標（各 1 bit）是 TCP 狀態機的驅動力， 控制連線的建立、維持、傳輸模式與終止。

SYN
Synchronize
連線建立請求；同步序號。只在三向交握的前兩個封包中設起
ACK
Acknowledgment
確認號欄位有效。除第一個 SYN 外，所有封包幾乎都設起此旗標
FIN
Finish
通知對方「我沒有更多資料要傳送了」，啟動四向揮手關閉流程
RST
Reset
立即重置連線，不等待四向揮手。連到不存在的埠、錯誤狀態時使用
PSH
Push
要求接收方立即將此資料交付給應用程式，不要在緩衝區中等待
URG
Urgent
緊急指標欄位有效，此 Segment 含需優先處理的緊急資料（如 SSH Ctrl+C）
ECE
ECN-Echo
ECN（Explicit Congestion Notification）機制：通知傳送方網路正在壅塞
CWR
Congestion Window Reduced
ECN 機制：確認已收到 ECE 通知並已降低傳送速率
Section 04

三向交握與四向揮手

三向交握（Three-Way Handshake）

TCP 連線建立需要三個步驟，確保雙方都同意連線， 並完成初始序號（ISN, Initial Sequence Number）的同步。 ISN 是隨機選擇的（防止舊連線的封包被誤認為新連線的資料）， 現代系統使用時間+隨機數演算法生成。 

    Client                                    Server
    CLOSED                                    LISTEN
      │                                         │
      │──── SYN ────────────────────────────────>│  Client: 我想建立連線
      │     Flags: SYN                           │  seq=x（Client 的 ISN）
      │     seq=x                                │
      │                                  SYN_RCVD│
      │<─── SYN-ACK ─────────────────────────────│  Server: 好的，我也準備好
      │     Flags: SYN, ACK                      │  seq=y（Server 的 ISN）
      │     seq=y, ack=x+1                       │  ack=x+1（期待 Client 的 x+1）
      │                                          │
      │──── ACK ────────────────────────────────>│  Client: 確認，開始傳資料
      │     Flags: ACK                           │
      │     seq=x+1, ack=y+1                     │
 ESTABLISHED                              ESTABLISHED
      │                                         │
      │════════════ 資料傳輸中 ════════════════════│

為什麼需要三步，而非兩步？

兩步驟握手（只有 SYN + SYN-ACK）只讓 Server 知道 Client 能傳送， 但 Client 不知道 Server 能傳送給自己。 第三步的 ACK 讓 Server 確認「我的 SYN-ACK 被收到了，雙向通訊確立」。 少一步就無法確保雙向連線品質。

四向揮手（Four-Way Handshake / Connection Termination）

由於 TCP 是全雙工的，關閉連線時兩個方向各自獨立關閉， 所以需要四個步驟。這讓「半關閉（Half-Close）」成為可能： 一方已不再傳送資料（發出 FIN），但仍可繼續接收對方的資料。 

    Client (主動關閉方)              Server (被動關閉方)
    ESTABLISHED                      ESTABLISHED
      │                                   │
      │──── FIN ─────────────────────────>│  Client: 我傳完了
      │     seq=u                          │
      │                             CLOSE_WAIT
      │<─── ACK ──────────────────────────│  Server: 收到，等我傳完
      │     ack=u+1                        │
      │                                   │  Server 可繼續傳送剩餘資料
      │<─── FIN ──────────────────────────│  Server: 我也傳完了
      │     seq=v                          │
      │                              LAST_ACK
      │──── ACK ─────────────────────────>│  Client: 確認
      │     ack=v+1                        │
      │                                   CLOSED
      │ (等待 2*MSL 後)                   │
    CLOSED                                │

  MSL = Maximum Segment Lifetime（封包最大存活時間，通常 60 秒）
  TIME_WAIT 狀態持續 2*MSL = 120 秒，確保網路中殘留的舊封包消散

TIME_WAIT 狀態的存在意義

主動關閉方在送出最後一個 ACK 後進入 TIME_WAIT 狀態，持續 2×MSL（通常 120 秒）。 這確保兩件事：第一，若最後的 ACK 遺失，Server 重傳的 FIN 能被正確回應； 第二，網路中所有屬於舊連線的封包在此期間消散， 不會混入使用相同四元組的新連線。 高並發伺服器上 TIME_WAIT 累積過多可能耗盡連接埠， 解法包含 SO_REUSEADDR 和調整 tcp_tw_reuse 核心參數。

Section 05

序號與確認號機制

TCP 以位元組為單位編號，而不是以封包為單位。 連線建立後，雙方各自維護一個 32 位元的序號計數器， 記錄已發送的位元組偏移量。

序號與確認號的運作（以傳送 3 個 Segment 為例）
傳送方                                接收方
  │                                    │
  │── Segment 1（seq=1, len=500）──────>│  含位元組 1–500
  │── Segment 2（seq=501, len=500）────>│  含位元組 501–1000
  │── Segment 3（seq=1001, len=500）───>│  含位元組 1001–1500
  │                                    │
  │<── ACK（ack=1501）─────────────────│  「我已收到 1500 bytes，期待 1501」
  │                                    │
  ▲ 累積確認（Cumulative ACK）：ack=N 表示 N-1 及之前的所有位元組都已收到

若 Segment 2 遺失：
  │── Segment 1（seq=1）───────────────>│
  │── Segment 2（seq=501）── 遺失 ──X  │
  │── Segment 3（seq=1001）────────────>│  收到但亂序
  │                                    │
  │<── ACK（ack=501）──────────────────│  「期待 501，之後的都先暫存」
  │<── ACK（ack=501）──────────────────│  重複 ACK（Duplicate ACK）
  │<── ACK（ack=501）──────────────────│  三次重複 ACK → 觸發快速重傳
  │── Segment 2（seq=501）重傳 ────────>│
  │<── ACK（ack=1501）─────────────────│  收到後一次確認所有

累積確認（Cumulative ACK）是 TCP 的基本確認模式： ACK 號表示「我已收到此號之前的所有位元組」。 若中間有封包遺失，接收方會持續對最後連續收到的序號發出重複 ACK， 傳送方收到三次重複 ACK 時觸發快速重傳（Fast Retransmit）， 不等逾時計時器到期就立刻補傳遺失的封包。

搭配 SACK（Selective Acknowledgment）選項時， 接收方可以在 ACK 中附帶「我已收到的不連續區塊資訊」， 例如告訴傳送方「我收到了 1–500 和 1001–1500，只缺 501–1000」， 讓重傳只補缺口，大幅提升效率。

Section 06

滑動視窗與流量控制

流量控制（Flow Control）解決「傳送方太快、接收方緩衝區被淹沒」的問題。 機制是接收方在每個 ACK 中告知傳送方自己的 接收視窗（rwnd, Receive Window）—— 接收緩衝區還有多少空間。傳送方不能讓「已傳送但未確認的資料量」超過 rwnd。

視窗「滑動」的意思是：每當接收方發出新的 ACK， 視窗左邊界向右移動（確認的資料移出視窗）， 同時視窗右邊界根據新的 rwnd 值調整。 傳送方持續填滿可傳送的視窗，形成「流水線（Pipeline）」效果， 使得傳輸效率遠高於每次等待 ACK 再傳下一個的「停等（Stop-and-Wait）」模式。

零視窗與視窗探測

當接收方緩衝區滿了，它會宣告 rwnd = 0， 稱為零視窗（Zero Window）。 傳送方此時必須停止傳送資料。 但如果接收方緩衝區騰出後忘記發 ACK 通知，雙方就會進入僵局。 為避免此情況，傳送方會定期發送視窗探測封包（Window Probe）， 詢問接收方的視窗大小是否有更新。

Silly Window Syndrome（愚蠢視窗症候群）

接收方每騰出 1 byte 就廣播 rwnd=1， 導致傳送方發出含大量標頭但只有 1 byte 資料的超小封包， 嚴重浪費頻寬。解法是 Clark 演算法（接收方延遲更新視窗直到緩衝區有足夠空間） 和 Nagle 演算法（傳送方累積小資料直到上一個封包被確認或湊滿 MSS 再傳）。

Section 07

壅塞控制（Congestion Control）

流量控制防止接收方被淹沒， 但不能防止網路本身被過量流量壅塞。 壅塞控制（Congestion Control）讓 TCP 感知網路壅塞並主動降速， 避免所有主機同時高速傳送而癱瘓路由器。

TCP 維護一個壅塞視窗（cwnd, Congestion Window）， 實際傳送量受 min(rwnd, cwnd) 限制。 cwnd 根據網路狀況動態調整，由四個演算法相互配合：

Slow Start
cwnd 初始值 1 MSS
每收到一個 ACK，cwnd += 1 MSS，實現指數成長。 每個 RTT 視窗加倍，快速探測可用頻寬， 直到達到 ssthresh（慢啟動門限）為止。
Congestion Avoidance
cwnd 達到 ssthresh 後
每個 RTT cwnd += 1 MSS（線性成長）， 比 Slow Start 保守，避免急速衝破瓶頸頻寬。 持續增長直到偵測到丟包（視窗壅塞信號）。
Fast Retransmit
3 次重複 ACK
收到三個重複 ACK 代表輕微丟包（網路未完全壅塞）。 立即重傳遺失的 Segment，不等計時器。 ssthresh = cwnd / 2，然後進入 Fast Recovery。
Fast Recovery
Reno / CUBIC 演算法
快速重傳後不回到 Slow Start， cwnd 設為新 ssthresh + 3 MSS，繼續線性增長， 直到收到對重傳封包的 ACK 才恢復正常傳輸。

Section 08

重傳機制

TCP 保證可靠傳輸的核心是：若封包沒有收到 ACK，就重傳。 但「等多久不收到 ACK 才重傳」是個精細的問題——太短浪費頻寬，太長降低效能。

RTO：重傳逾時計時器

RTO（Retransmission Timeout）是觸發重傳的計時器。 它的值由RTT 估算動態計算，使用 Jacobson/Karels 演算法：

RTO 計算公式（RFC 6298）
每次收到 ACK 更新 RTT 估算：

  SRTT = (1 - α) × SRTT + α × RTTsample      α = 0.125（1/8）
  RTTVAR = (1 - β) × RTTVAR + β × |SRTT - RTTsample|   β = 0.25（1/4）
  RTO = SRTT + 4 × RTTVAR

  SRTT：平滑 RTT（Smoothed RTT）
  RTTVAR：RTT 變異量（RTT Variance）

  RTO 下限：1 秒（RFC 6298）
  RTO 上限：通常 60–120 秒

重傳後 RTO 指數退避（Exponential Backoff）：
  第 1 次重傳後 RTO × 2
  第 2 次重傳後 RTO × 4
  第 3 次重傳後 RTO × 8
  ... 直到達到上限

快速重傳 vs. 逾時重傳

機制 觸發條件 cwnd 變化 特點
Fast Retransmit 連續收到 3 個重複 ACK ssthresh = cwnd/2；cwnd = ssthresh（不歸 1） 速度快，cwnd 不必從 1 重新 Slow Start
Timeout Retransmit RTO 計時器到期，沒有收到 ACK ssthresh = cwnd/2；cwnd = 1（歸一，重新 Slow Start） 最嚴重，表示完全沒有 ACK 回來，重傳並大幅降速
Section 09

TCP 狀態機

TCP 連線在任何時刻都處於一個明確的狀態。 理解狀態機能幫助你讀懂 netstat/ss 的輸出， 以及診斷各種連線問題。

CLOSED
沒有連線，初始狀態
LISTEN
伺服器端等待連線，已呼叫 listen()
SYN_SENT
客戶端已送出 SYN，等待 SYN-ACK
SYN_RECEIVED
伺服器收到 SYN 並回送 SYN-ACK，等待最後的 ACK
ESTABLISHED
連線建立，雙向資料傳輸中
FIN_WAIT_1
主動關閉方已送出 FIN，等待 ACK
FIN_WAIT_2
收到 ACK，等待對方的 FIN
CLOSE_WAIT
被動關閉方收到 FIN，等待應用程式關閉
CLOSING
雙方同時發出 FIN，等待對方 ACK（罕見）
LAST_ACK
被動關閉方已送出 FIN，等待最後一個 ACK
TIME_WAIT
主動關閉方等待 2×MSL（120 秒）確保對方收到最後 ACK
CLOSED
2×MSL 超時後完全關閉，回到初始狀態

  LISTEN 0/128    = Recv-Q/backlog，目前等待 accept() 的連線數/最大值
  ESTABLISHED     = 活躍連線，Send-Q=0 表示無積壓資料
  TIME-WAIT       = 剛關閉的連線，等待 2MSL
  CLOSE-WAIT      = 對方已 FIN 但應用程式尚未 close()，Recv-Q=1 有資料待讀取
                    CLOSE-WAIT 累積過多通常表示應用程式 bug（未關閉連線）
Section 10

TCP 效能最佳化

TCP Fast Open（TFO）

標準 TCP 建立連線需要一個 RTT 的三向交握才能傳送第一筆資料， 對短連線（如 DNS over TCP、HTTP 短請求）影響顯著。 TCP Fast Open（RFC 7413）允許客戶端在 SYN 封包中夾帶資料， 讓伺服器在回傳 SYN-ACK 的同時就開始處理請求， 節省一個 RTT 的延遲（後續連線）。

Keep-Alive

TCP Keep-Alive 是一個偵測閒置連線是否仍然存活的機制。 啟用後，若連線一段時間（預設 7200 秒）沒有資料往來， TCP 會定期發送探測封包（Keep-Alive Probe）。 若對方沒有回應，連線被判定死亡並關閉。 這對 NAT 環境尤其重要，因為 NAT 的映射表會在一段時間後自動老化刪除閒置連線。

Nagle 演算法與 TCP_NODELAY

Nagle 演算法預設啟用，它讓 TCP 累積小資料直到 「上一個未確認 Segment 被 ACK」或「資料量達到 MSS」才送出， 避免大量小封包浪費頻寬。 但對於互動式應用（如 SSH 鍵入、遊戲指令、gRPC 雙向串流）， Nagle 演算法引入的延遲是不可接受的， 應使用 TCP_NODELAY socket 選項停用它。

MTU Path Discovery 與 Path MTU

封包在網路路徑上經過的每個鏈路都有最大傳輸單元（MTU）限制。 若封包超過某個鏈路的 MTU，且 DF（Don't Fragment）旗標設起， 路由器會丟棄封包並回傳 ICMP Type 3 Code 4（Fragmentation Needed）。 TCP 利用這個機制自動探測整條路徑的最小 MTU， 稱為Path MTU Discovery（PMTUD，RFC 1191）， 並將 MSS 設為 PMTU - 40，避免分割帶來的效能損耗。

常用 TCP Socket 選項 作用
TCP_NODELAY 停用 Nagle 演算法，立即傳送小資料，適用互動式應用
SO_KEEPALIVE 啟用 Keep-Alive 探測，偵測死亡連線
SO_REUSEADDR 允許在 TIME_WAIT 期間重用同一 port，避免 port 被長時間占用
TCP_FASTOPEN 啟用 TCP Fast Open，第二次連線可在 SYN 中夾帶資料
SO_RCVBUF / SO_SNDBUF 手動調整接收/傳送緩衝區大小，影響 rwnd 和傳輸效率
TCP_CORK 暫時阻塞傳送，累積資料後一次送出（與 Nagle 類似但由應用控制）
Day 7 重點回顧

今天你學到了什麼

  • TCP 在 IP 不可靠的基礎上提供可靠、有序、無重複的位元組串流
  • TCP Header 20 bytes：Seq/Ack Number（排序+確認）、Window（流量控制）、Flags（狀態控制）
  • MSS、Window Scale、SACK、Timestamps 是四個最重要的 TCP Options
  • 三向交握（SYN → SYN-ACK → ACK）建立連線；四向揮手（FIN/ACK × 2）關閉連線
  • TIME_WAIT 持續 2×MSL，確保最後 ACK 到達以及舊封包消散
  • 滑動視窗：傳送方可以連續送出 cwnd 大小的資料，不必等每個 ACK
  • 流量控制（rwnd）：防止接收方緩衝區溢出；壅塞控制（cwnd）：防止網路壅塞
  • Slow Start 指數成長 → Congestion Avoidance 線性成長 → 丟包時降速
  • 3 次重複 ACK 觸發 Fast Retransmit（cwnd 減半）；RTO 超時觸發更嚴重的 Slow Start
  • CLOSE_WAIT 大量累積通常是應用程式 bug；TIME_WAIT 過多可用 SO_REUSEADDR 緩解
