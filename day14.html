雲端網路架構
在別人的電腦上建造自己的網路

雲端平台抽象掉了實體基礎設施，但網路的核心概念從未改變—— 改變的是如何透過 API 和控制台快速構建出等效的架構。 本課以 AWS 為主軸對照 Azure 和 GCP，帶你理解 VPC、子網路、 安全群組、負載均衡、CDN 和混合雲連線的設計邏輯。

Section 01

雲端網路的基本模型

雲端供應商的資料中心分佈在全球多個區域（Region），每個 Region 由多個可用區（Availability Zone，AZ）組成。這個物理分層直接對應到網路架構設計的高可用策略。

Region / AZ / 資料中心三層物理架構
  Region：ap-northeast-1（東京）
  ┌──────────────────────────────────────────────────────────────┐
  │  AZ-A（1a）            AZ-B（1b）           AZ-C（1c）        │
  │  ┌────────────────┐   ┌────────────────┐   ┌──────────────┐ │
  │  │ DC 1a-1, 1a-2  │   │ DC 1b-1, 1b-2  │   │ DC 1c-1      │ │
  │  └────────────────┘   └────────────────┘   └──────────────┘ │
  │        ↕ AZ 間低延遲高頻寬私有光纖互連（< 1ms）               │
  └──────────────────────────────────────────────────────────────┘
       ↕ 跨 Region 骨幹私有網路
  Region：ap-southeast-1（新加坡）

  設計原則：
  • 高可用：至少跨兩個 AZ 部署（電力/冷卻/網路各自獨立）
  • 災難復原：跨 Region（延遲約 70ms 東京到新加坡）
  • 效能優先：單一 AZ（最低延遲，但無冗餘）

Section 02

VPC：虛擬私有雲

VPC（Virtual Private Cloud）是雲端網路的基本隔離單位，讓你在雲端擁有一個邏輯上獨立的網路環境，完全隔離於其他客戶的 VPC。

典型三層 VPC 架構（Web / App / DB 分層，跨兩個 AZ）
  VPC：10.0.0.0/16（65,536 個 IP）
  ┌─────────────────────────────────────────────────────────────┐
  │  公有子網路（含 IGW 路由）                                    │
  │  10.0.1.0/24（AZ-A）          10.0.2.0/24（AZ-B）            │
  │  ALB / Bastion / NAT GW       Internet Gateway              │
  ├─────────────────────────────────────────────────────────────┤
  │  私有子網路—應用層                                            │
  │  10.0.11.0/24（AZ-A）         10.0.12.0/24（AZ-B）           │
  │  EC2 App Server / ECS Pod     EC2 App Server / ECS Pod      │
  ├─────────────────────────────────────────────────────────────┤
  │  私有子網路—資料層（隔離，無出站預設路由）                    │
  │  10.0.21.0/24（AZ-A）         10.0.22.0/24（AZ-B）           │
  │  RDS Primary / ElastiCache    RDS Standby / Cache Replica   │
  └─────────────────────────────────────────────────────────────┘

GCP VPC 的特殊性：AWS / Azure 的 VPC 是 Regional（每個 Region 獨立），而 GCP 的 VPC 是 Global——一個 VPC 跨越所有 Region，子網路是區域性的，但防火牆規則和路由全球統一管理，多 Region 架構更簡單但爆炸半徑更大。

IP 位址規劃的重要性

VPC 的 CIDR 一旦建立無法修改主要 CIDR。不同環境應使用不重疊 CIDR（Prod: 10.1.0.0/16、Staging: 10.2.0.0/16），否則日後無法建立 VPC Peering。

Section 03

子網路、路由表與閘道

子網路類型路由表特徵典型用途
公有子網路0.0.0.0/0 → igw-xxxxx，實例分配公有 IPALB、Bastion Host、NAT Gateway
私有子網路0.0.0.0/0 → nat-xxxxx（出站透過 NAT）App Server、微服務、快取
隔離子網路無 0.0.0.0/0，完全無法存取網際網路高敏感資料庫、HSM
VPC 閘道元件與流量路徑
  ① 外部 → VPC（公開服務）：
     Internet → Internet Gateway → ALB（公有）→ EC2（私有）

  ② VPC → 外部（私有子網路出站）：
     EC2（私有）→ NAT Gateway（公有）→ IGW → Internet
     （來源 IP 被 NAT 為 NAT GW 的 EIP）

  ③ VPC → AWS 服務（不出公網）：
     EC2 → VPC Endpoint（Gateway/Interface）→ S3 / DynamoDB
     （走 AWS 骨幹，不計流量費，安全）

  ④ VPC → 企業內網（混合雲）：
     EC2 → Virtual Private Gateway → Direct Connect / VPN → 企業

  閘道說明：
  Internet Gateway    VPC 出入公網，無費用，高可用
  NAT Gateway         私有子網路出站 NAT，按時 + 流量計費
  VPC Endpoint        私有存取 AWS 服務（S3/DynamoDB 免費）
  Transit Gateway     Hub-and-Spoke，連接多 VPC 和混合雲

NAT Gateway 費用陷阱

NAT GW 約 $0.045/小時（~$32/月）+ $0.045/GB 處理費。私有子網路大量存取 S3 費用快速累積。解決：為 S3 / DynamoDB 建立 VPC Gateway Endpoint（免費），每月可節省數百美元。

安全群組與網路 ACL

特性
安全群組（Security Group）
網路 ACL（NACL）
套用層級
EC2 實例（ENI 層級）
子網路層級
狀態
有狀態（Stateful）：允許入站後，對應回應自動允許
無狀態（Stateless）：入站和出站規則各自獨立設定
預設行為
預設拒絕所有（Default Deny）
預設 VPC NACL 允許全部；自訂 NACL 預設拒絕
規則評估
所有規則評估，取最寬允許（無順序）
按規則編號由小到大，首個匹配生效（有序）
來源指定
可用另一個 SG ID 作來源（SG 引用）
只能用 IP CIDR
Security Group 最佳實踐：SG 引用取代硬編碼 IP
  ✅ 推薦：SG 引用（動態可維護）
  ALB-SG  → App-SG:8080（只有 ALB 可存取 App）
  App-SG  → DB-SG:5432 （只有 App 可存取 DB）

  優點：App Server IP 改變 → 不需改 DB-SG 規則
        新增 App Server 套用 App-SG 即自動獲得 DB 存取

  ❌ 避免：硬編碼 IP CIDR
  DB-SG 入站：允許 10.0.11.0/24:5432（維護困難）
Section 05

雲端負載均衡

類型工作層核心能力適用場景
ALB（Application LB）L7（HTTP/HTTPS/gRPC）路徑/主機名稱/Header 路由；WAF 整合；Lambda 目標；WebSocketWeb 應用、微服務路由
NLB（Network LB）L4（TCP/UDP/TLS）超低延遲（<1ms）；靜態 EIP；保留來源 IP；TLS Passthrough即時遊戲、金融交易、需靜態 IP
GLB（Gateway LB）L3（IP）透明重導向第三方 NFV（GENEVE 封裝）整合 Palo Alto / Fortinet VM
ALB 路徑路由：單一 LB 服務多個微服務
  api.example.com → ALB
  ├── /api/users/*  → User-Service Target Group（EC2 x3）
  ├── /api/orders/* → Order-Service Target Group（ECS x4）
  ├── /api/pay/*    → Payment-Service Target Group
  └── 預設規則      → Frontend Target Group

  特性：
  • 各目標群組獨立健康檢查，不健康實例自動移除
  • Auto Scaling Group 依流量自動增減實例
  • 可整合 WAF 規則在 LB 層攔截惡意請求
Section 06

CDN 與邊緣網路

CDN（Content Delivery Network）將內容快取在全球分佈的邊緣節點（PoP），讓使用者從地理位置最近的節點取得內容，大幅降低延遲並卸載原始伺服器流量。

CDN 請求流程：快取命中 vs 快取未命中
  快取命中（Cache Hit）：
  台灣用戶 → DNS 解析到台灣 PoP → PoP 有快取 → 直接回應 ~5ms

  快取未命中（Cache Miss）：
  台灣用戶 → 台灣 PoP → 無快取 → 回源美國 Origin（~150ms）
  → PoP 快取內容（依 Cache-Control TTL）→ 回應用戶
  → 後續請求 ~5ms（快取命中）

  快取設定最佳實踐：
  靜態資源（含 hash）：Cache-Control: max-age=31536000, immutable
  HTML / API 動態：    Cache-Control: no-cache
  Vary: Accept-Encoding（gzip 與原始版本分開快取）
CloudFront 功能說明
Lambda@Edge在邊緣節點執行程式碼，修改請求/回應、A/B Testing、地理阻擋、認證驗證
Origin Shield在 PoP 與 Origin 之間增加中間快取層，減少回源請求，保護 Origin
Signed URL/Cookie為私有內容生成帶時效性的簽名存取 URL（付費影片、授權下載）
WAF 整合在邊緣節點執行 WAF 規則，攔截惡意流量在抵達 Origin 之前

Anycast vs GeoDNS

GeoDNS（AWS CloudFront）：依 DNS 查詢者 IP 位置回傳最近 PoP IP，簡單但精度受 DNS 快取影響。Anycast（Cloudflare / GCP）：所有 PoP 宣告相同 IP，BGP 自動路由到最近 PoP，精度更高且 DDoS 流量可分散到全球 PoP。

Section 07

混合雲連線：VPN 與 Direct Connect

AWS 混合雲連線方式對比
  企業資料中心
       │
       ├── ① Site-to-Site VPN（IPsec 加密隧道）
       │       頻寬：最高 ~1.25 Gbps
       │       延遲：受公網抖動影響
       │       建立時間：分鐘級
       │       費用：低（VPN 小時費 + 流量費）
       │       加密：IPsec 內建
       │
       └── ② Direct Connect（實體專線，非公網）
               頻寬：1 / 10 / 100 Gbps
               延遲：確定性，低抖動
               建立時間：數週到數月
               費用：高（電路月租 + 連接費）
               加密：需自行加密（MACsec 可選）

  最佳實踐：Direct Connect 主路徑 + Site-to-Site VPN 備援

Transit Gateway（TGW）是 AWS 的中央 Hub，採用 Hub-and-Spoke 模式：所有 VPC 和 VPN/Direct Connect 都連到 TGW，流量統一路由。100 個 VPC 只需 100 條連線（vs Peering 全網狀需要 4,950 條）。TGW 支援跨 Region Peering 和多帳號共用（RAM）。

Section 08

多雲與 VPC Peering

VPC Peering 不可傳遞性 vs Transit Gateway 可傳遞性
  VPC Peering（不可傳遞）：
  VPC-A ←─Peering─→ VPC-B ←─Peering─→ VPC-C
  問題：VPC-A 無法透過 VPC-B 存取 VPC-C
  解決：A 和 C 之間另建第三條 Peering

  Transit Gateway（可傳遞）：
            ┌──────────────────┐
            │  Transit Gateway │
            └─┬──────┬──────┬──┘
           VPC-A  VPC-B  VPC-C
  A 可透過 TGW 存取 C，路由由 TGW 路由表控制
多雲互連方案說明
公網 IPsec VPNAWS VGW ↔ Azure VPN GW 建立 IPsec 隧道。成本低，延遲高，頻寬有限
IXP 互連在中立共置 DC（如台灣 TWIX）同時接入多家雲端的 Direct Connect / ExpressRoute，低延遲
雲端網路平台Aviatrix、Megaport、Equinix Fabric 提供統一多雲控制平面
應用層 OverlayTailscale / WireGuard / Cloudflare Tunnel 建立跨雲 Overlay，不依賴雲端 VPN
Section 09

雲端網路成本與優化

費用項目計費方式優化策略
資料傳出（Egress）網際網路出站 ~$0.09/GB（前 10TB）；跨 Region ~$0.02/GB；跨 AZ ~$0.01/GBCloudFront 快取降低回源；VPC Endpoint 存取 S3 避免 NAT
NAT Gateway~$0.045/小時 + $0.045/GBS3 / DynamoDB 改用 VPC Gateway Endpoint（免費）；同 AZ 部署
ALB / NLBALB $0.008/LCU/小時；NLB $0.006/NLCU/小時多服務共用一個 ALB（路徑路由）；移除閒置 LB
Direct Connect連接埠費 + 資料傳輸費選合適速率；搭配 S3 直接存取節省回源費

跨 AZ 流量費用是容易被忽略的成本

跨 AZ 傳輸雙向各 $0.01/GB，大型應用的資料庫讀寫和快取同步每月可累積數 TB。優化：同 AZ 部署相互依賴的服務（App Server + Cache 在同一 AZ）；使用本地快取減少跨 AZ 查詢。

AWS PrivateLink 讓你透過私有 IP 存取 AWS 服務，流量完全留在 AWS 骨幹。Gateway Endpoint（免費）支援 S3 和 DynamoDB；Interface Endpoint（PrivateLink，按小時計費）支援其他 AWS 服務，在子網路中建立 ENI 以私有 IP 呈現。

Day 14 重點回顧

今天你學到了什麼

  • Region → AZ → 資料中心三層架構；高可用設計至少跨兩個 AZ（電力/冷卻/網路各自獨立）
  • VPC CIDR 一旦設定無法修改；不同環境使用不重疊 CIDR 以利未來 Peering
  • GCP VPC 是全球性（Global）；AWS / Azure VPC 是區域性（Regional）
  • 公有子網路 = 路由表有 IGW；私有子網路 = 路由表有 NAT GW；隔離子網路 = 無預設路由
  • S3 / DynamoDB 應建立 VPC Gateway Endpoint（免費）取代 NAT Gateway，節省費用
  • Security Group 有狀態（Stateful），套用於實例；NACL 無狀態（Stateless），套用於子網路
  • Security Group 推薦使用 SG 引用取代硬編碼 IP CIDR，動態且可維護
  • ALB（L7 路徑路由）、NLB（L4 靜態 IP 超低延遲）、GLB（NFV 流量重導）各有適用場景
  • CDN Cache Hit ~5ms；靜態資源設 max-age=31536000 + immutable；Anycast（Cloudflare/GCP）vs GeoDNS（AWS）
  • Direct Connect（確定性延遲，數月建立）vs VPN（公網，分鐘建立）；建議雙路備援
  • VPC Peering 不可傳遞；Transit Gateway 提供可傳遞 Hub-and-Spoke，100 VPC 只需 100 條連線
