Section 01

CIA 三元組

資訊安全的一切目標都可以歸納為三個核心屬性， 業界稱為 CIA 三元組（CIA Triad）。 任何安全決策——從協定選擇到存取控制設計—— 都在思考如何同時維護這三項屬性，或在它們之間做出合理的取捨。

C
機密性（Confidentiality）
確保資訊只有授權的人能存取。防止未授權的讀取、攔截或揭露。
← 對應：加密、存取控制、VPN、TLS
I
完整性（Integrity）
確保資訊在儲存與傳輸過程中未被未授權修改。任何竄改都能被偵測。
← 對應：雜湊、數位簽章、MAC、HTTPS
A
可用性（Availability）
確保授權用戶在需要時能存取資訊與服務。防止拒絕服務攻擊與系統中斷。
← 對應：DDoS 防護、冗餘、負載均衡、備份

真實的安全設計往往需要在三者之間做取捨。 例如，強加密增強了機密性，但若金鑰遺失則危及可用性； 多重備份提高了可用性，但增加了機密性洩漏的攻擊面。 安全架構師的工作就是在業務需求的框架下找到最佳平衡點。

延伸：AAA 框架

CIA 描述的是「保護什麼」，而 AAA 框架描述的是「如何管控存取」： Authentication（認證）——你是誰？ Authorization（授權）——你能做什麼？ Accounting（稽核）——你做了什麼？ RADIUS、TACACS+ 等協定正是實作 AAA 的標準工具。

Section 02

密碼學基礎

密碼學是網路安全的數學基石。 理解核心概念有助於正確評估與選擇安全機制， 而不只是盲目套用「只要加密就安全」的錯誤觀念。

加密的基本概念

術語說明
明文（Plaintext）原始、未加密的資料
密文（Ciphertext）加密後的資料，對攻擊者無法直接讀取
加密（Encryption）用金鑰將明文轉換為密文的過程
解密（Decryption）用金鑰將密文還原為明文的過程
金鑰（Key）控制加解密過程的秘密參數。演算法公開，安全性依賴金鑰的保密性（Kerckhoffs 原則）
密碼套件（Cipher Suite）一組演算法的組合，如 TLS 1.3 的 TLS_AES_256_GCM_SHA384

Kerckhoffs 原則

現代密碼學的核心原則：系統的安全性不應依賴演算法的保密， 而應依賴金鑰的保密。演算法應該是公開可審計的。 以「我的演算法是秘密的，所以是安全的」為基礎的「保密即安全 （Security through Obscurity）」是一種危險的謬誤—— 一旦演算法洩漏，整個系統就毫無防護。 RSA、AES、ChaCha20 都是完全公開的演算法，其安全性依賴金鑰長度的計算複雜度。

隨機性的重要性

密碼學安全的隨機數生成（CSPRNG，Cryptographically Secure Pseudo-Random Number Generator）是所有密碼系統的命脈。 若隨機數生成器可預測，攻擊者可以猜出金鑰、IV（初始向量）或 nonce。 歷史上最著名的案例是 2008 年 Debian OpenSSL 漏洞： 一個「優化」補丁意外移除了隨機性的主要熵來源， 導致所有生成的 RSA/DSA 金鑰實際上只有 32,767 種可能值， 整個受影響的 Debian 系統的 SSH 金鑰幾乎在數小時內就能被暴力破解。

Section 03

對稱與非對稱加密

對稱加密（Symmetric Encryption）
金鑰加解密使用同一把金鑰
速度極快，適合大量資料加密
問題金鑰分發難題：如何安全地把金鑰給對方？
演算法AES-128/256、ChaCha20、3DES（已棄用）
模式AES-GCM（認證加密）、AES-CBC（需搭配 MAC）
用途TLS 傳輸資料、磁碟加密、VPN 資料通道
非對稱加密（Asymmetric Encryption）
金鑰公鑰（Public Key）加密，私鑰（Private Key）解密
速度慢（比對稱加密慢 100–1000 倍）
優點解決金鑰分發：公鑰可公開，私鑰自己保管
演算法RSA（金鑰交換、簽名）、ECDSA（簽名）、ECDH/X25519（金鑰協商）
安全基礎RSA：大數分解難題；ECC：橢圓曲線離散對數難題
用途TLS 握手金鑰交換、數位簽章、SSH 認證

混合加密：兩者的最佳組合

現實中幾乎所有安全協定都使用混合加密（Hybrid Encryption）： 用非對稱加密解決金鑰分發問題，用對稱加密處理實際的大量資料加密。

混合加密流程（TLS 握手的本質）
  1. 金鑰協商（非對稱，ECDH）：
     客戶端 & 伺服器各自生成臨時 ECDH 金鑰對
     交換公鑰 → 雙方各自計算出相同的共享密鑰（Shared Secret）
     （攻擊者攔截公鑰但無法計算共享密鑰——ECDH 的數學特性保證）

  2. 金鑰衍生（KDF）：
     Shared Secret + 隨機數 → 用 HKDF 衍生出對稱金鑰（Session Key）

  3. 對稱加密傳輸：
     所有後續資料都用 Session Key 的 AES-GCM 加密
     速度快、計算開銷低

  結果：非對稱的安全 + 對稱的效能

前向保密（Perfect Forward Secrecy, PFS）

傳統 RSA 金鑰交換中，若伺服器的私鑰未來被洩漏， 攻擊者可以解密所有過去攔截的流量。 前向保密（使用 ECDHE 而非靜態 RSA）確保每次連線都生成新的臨時金鑰， 即使私鑰未來被洩漏，過去的通訊仍然安全。 TLS 1.3 強制要求使用具備 PFS 的金鑰交換演算法（ECDHE 或 DHE）， 移除了所有不支援 PFS 的密碼套件。

Section 04

雜湊函數與 MAC

密碼學雜湊函數

雜湊函數（Cryptographic Hash Function）將任意長度的輸入 映射到固定長度的輸出（摘要／Digest），具備三個關鍵特性：

特性說明
單向性（One-Way）從摘要無法反推原始輸入（計算不可逆）
碰撞抵抗（Collision Resistance）找不到兩個不同的輸入產生相同的輸出（難以偽造）
雪崩效應（Avalanche Effect）輸入改變一個位元，輸出的約 50% 位元會改變（無規律）
演算法輸出長度狀態用途
MD5128 bits已破解（碰撞已知）僅用於非安全場景（如校驗和），不可用於安全用途
SHA-1160 bits已棄用（2017 年 SHAttered 攻擊成功）舊系統遺留，新系統禁止使用
SHA-256256 bits安全，廣泛使用TLS 憑證、Git commit hash、HTTPS 完整性驗證
SHA-3（Keccak）224/256/384/512 bits安全，後備選擇設計完全不同於 SHA-2，防止 SHA-2 被攻破時的替代
BLAKE3256 bits（可調）現代、極快高效能場景的雜湊，比 SHA-256 快 5–10 倍

密碼儲存：為什麼不能只用雜湊？

若資料庫儲存 SHA-256("password123")， 攻擊者可以預先計算所有常見密碼的雜湊值（彩虹表攻擊，Rainbow Table）。 正確的密碼儲存需要：

加鹽（Salting）：為每個密碼附加隨機的唯一鹽值再雜湊， 讓彩虹表完全失效（每個密碼的雜湊都不同）。 慢速雜湊（Slow Hash）：使用專為密碼設計的演算法 bcrypt、scrypt、Argon2（2015 年密碼雜湊競賽冠軍）， 這些演算法刻意設計成計算成本高，使暴力破解需要耗費大量時間與記憶體。 永遠不要用 SHA-256 直接儲存密碼，即使加鹽也不夠——它太快了。

MAC：帶金鑰的完整性驗證

純雜湊只能驗證完整性，但任何人都能重新計算雜湊值並偽造訊息。 MAC（Message Authentication Code）用共享的密鑰計算， 只有擁有金鑰的人才能驗證——同時提供完整性和來源認證。

最常用的是 HMAC（Hash-based MAC）： HMAC(key, message) = H((key ⊕ opad) ∥ H((key ⊕ ipad) ∥ message))。 更現代的是 AEAD（Authenticated Encryption with Associated Data）， 如 AES-GCM 和 ChaCha20-Poly1305，在加密的同時自動生成認證標籤（Auth Tag）， 一步完成加密和完整性驗證。

Section 05

防火牆的演進

防火牆是網路安全的第一道邊界防線， 但「防火牆」這個詞涵蓋了從最簡單的封包過濾到應用層深度檢測的多個世代技術。

世代技術運作層特點與侷限
第一代（1988） 封包過濾（Packet Filtering） L3/L4 依 IP、Port、Protocol 的 ACL 規則過濾。無狀態，無法辨識應用語意。容易被偽造 IP 或分割封包繞過
第二代（1990） 狀態檢測（Stateful Inspection） L3/L4 追蹤 TCP 連線狀態（SYN/ESTABLISHED），自動允許已建立連線的回應封包，大幅降低規則複雜度。仍無法理解應用層內容
第三代（1994） 應用層代理（Application Proxy） L7 針對特定協定（HTTP、FTP、DNS）做深度解析。安全性高，但效能低、延遲高、每個協定需獨立代理
第四代（2000s） 下一代防火牆（NGFW） L3–L7 整合 IPS、SSL 解密、應用識別、使用者識別、URL 過濾於一體。Palo Alto、Fortinet 代表產品
現代 雲端 WAF / SASE L7+ Web Application Firewall 防護 OWASP Top 10；SASE 將安全功能移至雲端邊緣，適應零信任架構
三區架構（DMZ）：企業網路的標準分區設計
  網際網路
      │
  ┌───┴───────────────────────────────────────────┐
  │  外部防火牆（External Firewall）                │
  │  規則：只允許 TCP 80/443 入站到 DMZ             │
  └───┬───────────────────────────────────────────┘
      │
  ┌───┴──────────────────────────────────────────────────┐
  │  非軍事區（DMZ, Demilitarized Zone）                  │
  │  ┌──────────┐  ┌──────────┐  ┌──────────┐           │
  │  │ Web Server│  │Mail Server│  │DNS Server │          │

Section 06

入侵偵測與防禦（IDS/IPS）

防火牆基於規則允許或拒絕流量，但無法辨識已通過規則的惡意行為。 IDS/IPS 則持續監控流量模式，尋找已知攻擊特徵或異常行為。

系統全名功能部署模式
IDS Intrusion Detection System 偵測並告警，不主動阻擋。被動監控，適合不能影響流量的場景 旁路（Out-of-Band），接收流量副本（SPAN Port）
IPS Intrusion Prevention System 偵測並主動阻擋惡意流量。可以丟棄封包、重置連線 串聯（Inline），流量必須經過它
NIDS Network-based IDS 監控網路流量，偵測網路層攻擊 核心交換器的 SPAN/TAP 介面
HIDS Host-based IDS 監控單一主機的系統呼叫、日誌、檔案變更，偵測主機層攻擊 代理程式安裝在每台主機上（如 OSSEC、Wazuh）

偵測方法

方法原理優缺點
特徵比對（Signature-based） 比對已知攻擊的特徵碼（如 Snort Rules），類似防毒軟體的病毒碼 精準度高、誤報少；無法偵測未知（Zero-day）攻擊；特徵庫需持續更新
異常偵測（Anomaly-based） 建立正常行為基準線（Baseline），偵測偏離基準的異常行為 可偵測未知攻擊；誤報率高（新業務部署常被誤判）；需較長的學習期
行為分析（Behavioral） 使用機器學習分析流量模式、用戶行為（UEBA），識別可疑行為序列 現代 SIEM/EDR 的核心；可偵測 APT；計算資源需求高
Section 07

VPN 原理與協定

VPN（Virtual Private Network）在公共網路上建立加密隧道， 讓遠端設備彷彿在私有網路內部一樣通訊。 VPN 同時提供機密性（加密）、完整性（MAC）和認證（憑證或預共享金鑰）。

VPN 隧道封裝示意
原始封包（內部網路）：
  ┌─────────────────────────────────────┐
  │ IP Header │ TCP Header │  HTTP Data  │
  │ 10.0.0.5  │  Port 80   │  明文內容   │
  └─────────────────────────────────────┘

經過 VPN 隧道封裝後（在公共網際網路傳輸）：
  ┌──────────────────────────────────────────────────────────┐
  │ 外部 IP Header │ VPN Header │       加密 Payload          │
  │ 公網 IP 位址   │ (ESP/TLS)  │ ┌────────────────────────┐ │
  │                │            │ │ 原始 IP │TCP│HTTP(加密)  │ │
  │                │            │ └────────────────────────┘ │
  └──────────────────────────────────────────────────────────┘

中間人只能看到：「從公網 IP A 到公網 IP B 的加密資料」
無法得知：內部 IP、連接埠、應用層內容

主要 VPN 協定比較

協定傳輸特點適用場景
IPsec（IKEv2） IP Protocol 50/51（ESP/AH），UDP 500/4500 作業系統原生支援；兩種模式：傳輸模式（加密 Payload）和隧道模式（加密整個封包）；行動設備切換網路時連線穩定（MOBIKE） 企業 Site-to-Site VPN、行動設備 Remote Access
OpenVPN UDP（預設）或 TCP，Port 可自訂（常用 1194/443） 基於 TLS，開源，跨平台；設定靈活但複雜；用 TCP 443 可穿越嚴格防火牆 遠端存取、隱私保護
WireGuard UDP，Port 51820（預設） 程式碼僅 4,000 行（OpenVPN 約 100,000 行），審計容易；效能卓越；ChaCha20-Poly1305 加密；已納入 Linux 核心 5.6+ 現代首選，個人 VPN、雲端連線
SSL/TLS VPN HTTPS（TCP 443） 透過瀏覽器或輕量用戶端存取；只需 443 port，幾乎不被防火牆封鎖；不需安裝 VPN 客戶端（無客戶端模式） 零信任網路存取（ZTNA）、企業 Remote Access

Split Tunneling（分割隧道）

VPN 預設把所有流量導入加密隧道（Full Tunnel）， 包含存取 YouTube 等公共服務，增加 VPN 伺服器負擔。 Split Tunneling 只把需要的流量（如存取公司內網） 路由進 VPN，其餘流量直接走本地網路， 在效能與安全性之間取得平衡。 反制方案：企業可強制 Full Tunnel，防止資料從本地連線洩漏。

Section 08

網路攻擊殺傷鏈

Cyber Kill Chain（網路攻擊殺傷鏈）由 Lockheed Martin 於 2011 年提出， 描述了進階持續性威脅（APT）的典型攻擊階段。 理解攻擊鏈有助於在每個階段設計有效的偵測與阻斷機制—— 攻擊者必須完成所有階段才能成功，防禦者只需在任一階段阻斷就能瓦解攻擊。

Cyber Kill Chain — 七個攻擊階段
1
偵查（Reconnaissance）
蒐集目標資訊：WHOIS 查詢、DNS 枚舉、LinkedIn 職員資訊、Shodan 掃描、釣魚目標蒐集。防禦：限制公開資訊、AXFR 限制
2
武器化（Weaponization）
製作武器：將漏洞利用程式（Exploit）打包進可傳遞的載體（如惡意 PDF、Office 文件、含後門的安裝包）。防禦：難以直接阻止，依賴後續偵測
3
傳遞（Delivery）
將武器送達目標：魚叉式釣魚郵件（Spear Phishing）、惡意連結、供應鏈攻擊、USB 植入。防禦：郵件過濾、沙箱分析、員工安全意識培訓
4
觸發（Exploitation）
漏洞被觸發執行：使用者開啟惡意文件、瀏覽器漏洞利用（Drive-by Download）、零日漏洞。防禦：及時修補（Patching）、最小化攻擊面、應用白名單
5
安裝（Installation）
植入持久化機制：安裝 RAT（遠端存取木馬）、後門、修改開機設定、建立排程任務或服務。防禦：EDR、檔案完整性監控、特權存取管理
6
命令控制（C2）
建立與攻擊者的通訊通道：DNS 隧道、HTTPS C2（混入正常流量）、社交媒體隱蔽通道。防禦：DNS 過濾、異常流量分析、出站流量白名單
7
目標行動（Actions）
達成最終目標：資料外洩（Exfiltration）、勒索軟體加密、破壞系統、橫向移動（Lateral Movement）。防禦：資料外洩防護（DLP）、最小權限、網路分段
Section 09

常見網路層攻擊手法

DDoS 分散式阻斷服務攻擊可用性威脅

容積型（Volumetric）：以大量流量淹沒頻寬（UDP Flood、ICMP Flood、DNS/NTP 放大）。2024 年最大記錄達 5.6 Tbps（Cloudflare 緩解）。

協定型（Protocol）：利用協定狀態機耗盡伺服器資源（SYN Flood 耗盡半開連線佇列）。防禦：SYN Cookie——伺服器不分配資源，僅在收到最後 ACK 時才建立連線狀態。

應用層型（Application Layer / L7）：以少量看似合法的請求耗盡應用程式資源（Slowloris 保持 HTTP 連線但不完成、HTTP GET Flood 查詢耗費 CPU 的 API）。防禦：WAF、速率限制、CAPTCHA。

中間人攻擊（MitM）機密性 / 完整性威脅

攻擊者將自己插入通訊雙方之間，可攔截、讀取或修改所有通訊內容。常見手法：

ARP Spoofing（已在 Day 6 詳述）：毒化 ARP Cache 攔截 LAN 流量。

HTTPS Downgrade / SSL Stripping：攔截 HTTP→HTTPS 重導，讓客戶端以為在用 HTTP，中間人讀取明文。防禦：HSTS + Preload。

BGP Hijacking（已在 Day 5 詳述）：劫持路由使流量繞道攻擊者。防禦：RPKI。

Evil Twin Wi-Fi：架設與合法 AP 同名的假熱點，讓設備自動連入。防禦：強制 VPN、憑證驗證、802.1X 認證。

Port Scan 與服務枚舉偵查工具

TCP Connect Scan：完成完整三向交握後關閉連線（慢、留有日誌）。SYN Scan（半開掃描）：只送 SYN，收到 SYN-ACK 即發 RST，不完成握手（快、不易被應用層偵測，nmap 預設模式）。UDP Scan：發送 UDP 封包，接收 ICMP Port Unreachable 判斷埠是否關閉。

防禦：防火牆預設拒絕原則（Default-Deny）、Port Knocking、入侵偵測系統偵測掃描行為。

橫向移動（Lateral Movement）攻擊鏈延伸

攻擊者在成功入侵一台主機後，利用該主機的憑證、信任關係或共同服務，在內部網路中擴散到更多高價值目標。常見技術：Pass-the-Hash（重用 NTLM 雜湊）、Pass-the-Ticket（Kerberos 票據重用）、SMB 利用、SSH 金鑰轉發。

防禦：網路微分割（Micro-Segmentation）、最小權限（Least Privilege）、特權存取工作站（PAW）、定期輪換憑證。

Section 10

縱深防禦與零信任

縱深防禦（Defense in Depth）

縱深防禦的核心思想是：任何單一安全控制措施都可能失效， 因此應在多個層面建立獨立的防禦機制， 讓攻擊者必須突破多道防線才能達成目標。

零信任架構（Zero Trust Architecture）

傳統安全模型假設「內網是安全的」（城堡加護城河）， 一旦通過邊界就信任所有流量。 零信任（Zero Trust）的核心原則是： 「永不信任，始終驗證（Never Trust, Always Verify）」。 無論請求來自內網還是外網，都必須通過嚴格的認證和授權。

原則傳統邊界安全零信任架構
信任基礎位置（在內網 = 信任）身份（每次請求都必須驗證）
網路邊界明確的內外網邊界無邊界，每個資源都是獨立保護的
存取控制進入內網後較少限制最小權限，只授予完成任務所需的最少存取
通訊加密內網流量通常不加密所有流量加密，包含東西向（East-West）流量
假設入侵只來自外部假設已被入侵（Assume Breach），最小化爆炸半徑

零信任的實踐工具包括：身份提供者（IdP）+ 多因素認證（MFA）、 微分割（Micro-Segmentation）限制東西向流量、 設備合規性檢查（Device Posture）確認裝置未被感染、 以及持續的存取行為分析偵測異常。 Google 的 BeyondCorp 和 NIST SP 800-207 是零信任的主要參考框架。

Day 11 重點回顧

今天你學到了什麼

  • CIA 三元組：機密性（加密）、完整性（雜湊/MAC）、可用性（冗餘/DDoS 防護）
  • Kerckhoffs 原則：安全性靠金鑰，不靠演算法保密；Security through Obscurity 是謬誤
  • 對稱加密（AES-GCM）速度快；非對稱加密（RSA/ECDH）解決金鑰分發；實際用混合加密
  • 前向保密（PFS）：ECDHE 每次生成臨時金鑰，私鑰洩漏不影響過去通訊
  • SHA-256 安全可用；MD5/SHA-1 已破解；密碼儲存須用 bcrypt/Argon2（加鹽 + 慢速）
  • AEAD（AES-GCM / ChaCha20-Poly1305）同時完成加密和認證
  • 防火牆從封包過濾進化到 NGFW；DMZ 三區架構提供縱深防禦
  • IDS 偵測告警；IPS 主動阻擋；特徵比對 vs 異常偵測各有適用場景
  • WireGuard 是現代 VPN 首選（4,000 行程式碼）；IPsec 企業標準；Split Tunneling 分流
  • Kill Chain 7 階段：偵查→武器化→傳遞→觸發→安裝→C2→目標行動，每段可介入阻斷
  • 零信任：「永不信任，始終驗證」取代「內網預設安全」的舊模型
