HTTP 與 HTTPS
網頁世界的語言與信封

HTTP 是現代網際網路的核心應用層協定，從 1991 年的單行指令到今日的 HTTP/3， 每次演進都在解決上一版本的根本瓶頸。HTTPS 在 HTTP 之上加上 TLS 的信封， 讓竊聽與篡改成為徒勞。本課從請求/回應格式到 TLS 握手， 帶你完整理解每一次網頁載入背後的精密協作。

Section 01

HTTP 的演進歷程

HTTP（HyperText Transfer Protocol）由 Tim Berners-Lee 在 CERN 提出， 與 HTML 和 URL 共同構成了全球資訊網（WWW）的三大基石。 從 1991 年只有一行指令的 HTTP/0.9，到今日整合在 QUIC 上的 HTTP/3， 每一次版本演進都是對效能瓶頸的直接回應。

HTTP 版本演進
HTTP/0.9
1991 — 單行協定
只有 GET 方法，無 Header，無狀態碼。請求一行：GET /index.html，回應直接是 HTML 內容。每次請求後立即關閉連線。
HTTP/1.0
1996（RFC 1945）— 加入 Header 與狀態碼
引入 Header、狀態碼、POST/HEAD 方法、Content-Type。但每個請求仍需一個獨立 TCP 連線，開銷極大（1 RTT 握手 + 傳輸 + 關閉）。
HTTP/1.1
1997（RFC 2616）→ 2014（RFC 7230–7235）— 持久連線
引入 Keep-Alive 持久連線、Host 標頭（虛擬主機）、Chunked Transfer Encoding、管線化（Pipelining，效果有限）。至今仍廣泛使用。核心問題：隊頭阻塞（HOL Blocking）。
HTTP/2
2015（RFC 7540）— 多路復用
二進位框架層、多路復用（Multiplexing）、標頭壓縮（HPACK）、伺服器推送（Server Push）、流優先級。解決了應用層 HOL Blocking，但 TCP 層的 HOL Blocking 依然存在。
HTTP/3
2022（RFC 9114）— 基於 QUIC
捨棄 TCP，改用 QUIC（UDP）作為傳輸層。徹底解決 HOL Blocking、0-RTT 快速重連、連線遷移（Connection Migration）。2024 年已有約 30% 的網站流量使用 HTTP/3。
Section 02

HTTP 請求與回應格式

HTTP/1.1 使用純文字格式（Human-readable）， 由起始行、Header 欄位、空行（CRLF）和可選的 Body 組成。 理解此格式是掌握所有 Web 技術的基礎。

HTTP/1.1 請求（Request）範例
起始行（Request Line）
POST /api/login HTTP/1.1
請求標頭（Request Headers）
Host: api.example.com
Content-Type: application/json
Content-Length: 42
Accept: application/json
Authorization: Bearer eyJhbGciOiJSUzI1NiJ9...
User-Agent: Mozilla/5.0 (Macintosh; Intel Mac OS X)
Connection: keep-alive
空行（CRLF，分隔 Header 與 Body）
（空行）
請求本體（Request Body，僅部分方法有）
{"username": "alice", "password": "s3cr3t"}
HTTP/1.1 回應（Response）範例
狀態行（Status Line）
HTTP/1.1 200 OK
回應標頭（Response Headers）
Content-Type: application/json; charset=utf-8
Content-Length: 128
Cache-Control: no-store
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict
X-Content-Type-Options: nosniff
Strict-Transport-Security: max-age=31536000; includeSubDomains
Date: Mon, 02 Mar 2026 12:00:00 GMT
空行
（空行）
回應本體（Response Body）
{"status": "ok", "token": "eyJhbGciOiJSUzI1NiJ9...", "expires_in": 3600}

URL 各組成部分

  ──┬──   ────┬────  ───────┬──────  ──┬─ ───┬─── ───────┬───────  ──┬──
    │         │              │          │     │            │           │
  scheme  userinfo        host        port  path         query      fragment
  (協定)  (憑證，少用)   (主機名稱)  (連接埠) (路徑)      (查詢字串)  (片段，不傳給伺服器)

Section 03

HTTP 方法（Verbs）

HTTP 方法定義了對資源執行的操作語意。 設計良好的 REST API 應遵循方法的安全性（Safe，不修改資源） 與冪等性（Idempotent，多次執行結果相同）原則。

GET
安全 ✓  冪等 ✓
取得資源。請求不應有 Body。回應可被快取。最常用的方法，不應有副作用。
POST
安全 ✗  冪等 ✗
建立資源或觸發操作。每次呼叫可能產生不同結果（如建立新訂單）。回應不快取。
PUT
安全 ✗  冪等 ✓
完整取代資源。多次執行結果相同（第 2 次 PUT 與第 1 次效果一致）。需提供完整資源表示。
PATCH
安全 ✗  冪等 △
部分更新資源，只傳修改的欄位。冪等性取決於實作（差量更新通常非冪等）。
DELETE
安全 ✗  冪等 ✓
刪除資源。第一次執行刪除，後續再刪除已不存在的資源仍應回傳 204/404，結果一致。
HEAD
安全 ✓  冪等 ✓
與 GET 相同，但伺服器只回傳 Header，不回傳 Body。用於檢查資源是否存在或取得 Metadata。
OPTIONS
安全 ✓  冪等 ✓
查詢伺服器對某 URL 支援哪些方法。CORS 預檢請求（Preflight）使用此方法。
CONNECT
安全 ✗  冪等 ✗
建立 HTTP 隧道（Tunnel）。HTTPS Proxy 用此方法讓客戶端與目標伺服器建立 TLS 連線。
Section 04

HTTP 狀態碼

HTTP 狀態碼是三位數的數字，第一位代表類別。 熟悉常用狀態碼是 Web 開發與故障排除的必備能力。

1xx — 資訊性（Informational）：請求已收到，繼續處理
100
Continue — 客戶端可繼續發送請求 Body（大型 POST 前的確認）
101
Switching Protocols — 協定升級，如從 HTTP/1.1 升級到 WebSocket
2xx — 成功（Success）：請求已成功接收、理解並處理
200
OK — 最常見的成功回應，GET/PUT 回傳資源，POST 回傳結果
201
Created — POST 成功建立新資源，Location 標頭指向新資源 URL
204
No Content — 成功但無回應 Body，常用於 DELETE 或 PATCH 後
206
Partial Content — 回傳部分資源（Range 請求，用於斷點續傳、影片串流）
3xx — 重新導向（Redirection）：需要進一步操作完成請求
301
Moved Permanently — 永久重導，瀏覽器快取並自動改用新 URL。SEO 傳遞
302
Found（舊稱 Moved Temporarily）— 臨時重導，不快取。常被誤用，應考慮 307/308
304
Not Modified — 資源未變更，瀏覽器使用快取版本（配合 If-Modified-Since / ETag）
307
Temporary Redirect — 臨時重導，保留原始 HTTP 方法（302 可能將 POST 改為 GET）
308
Permanent Redirect — 永久重導，保留 HTTP 方法（301 的嚴格版）
4xx — 客戶端錯誤（Client Error）：請求有問題
400
Bad Request — 請求語法錯誤，伺服器無法解析（如 JSON 格式錯誤）
401
Unauthorized — 未認證（需要提供身份憑證），名字有點誤導，實為「未認證」
403
Forbidden — 已認證但無授權，伺服器拒絕執行（與 401 的差異：你是誰 vs 你能幹嘛）
404
Not Found — 資源不存在。也可用於隱藏「資源存在但無授權」（避免資訊洩漏）
405
Method Not Allowed — 該 URL 不支援此 HTTP 方法，回應應包含 Allow 標頭
409
Conflict — 請求與目前資源狀態衝突（如建立重複的使用者名稱）
422
Unprocessable Entity — 語法正確但語意錯誤（如 email 格式不合法），REST API 常用
429
Too Many Requests — 超過速率限制（Rate Limiting），回應通常含 Retry-After 標頭
5xx — 伺服器錯誤（Server Error）：伺服器在處理時出錯
500
Internal Server Error — 伺服器未預期的錯誤，通常是 Bug 或例外未處理
502
Bad Gateway — 代理伺服器從上游收到無效回應（如 Nginx 背後的 App Server 崩潰）
503
Service Unavailable — 伺服器暫時無法處理（過載或維護），通常含 Retry-After
504
Gateway Timeout — 代理伺服器等待上游逾時（上游回應太慢）
Section 05

重要 HTTP 標頭

快取控制標頭

標頭說明
Cache-Controlmax-age=3600（快取 1 小時）、no-cache（每次需重新驗證）、no-store（完全不快取）、public/private、immutable（永久快取）
ETag資源的版本標識（通常是內容的 Hash）。客戶端下次請求帶 If-None-Match: "etag值"，若未變則回 304
Last-Modified資源最後修改時間。客戶端帶 If-Modified-Since，若未變則回 304，比 ETag 精度低
Expires絕對過期時間（HTTP/1.0 遺留）；有 Cache-Control 時此標頭被忽略
Vary告訴快取伺服器依哪些請求標頭區分快取，如 Vary: Accept-Encoding 讓 gzip 與原始版本分開快取

內容協商標頭

標頭說明
Accept客戶端可接受的 Content-Type，如 application/json, text/html;q=0.9, */*;q=0.8（q 值代表優先權）
Accept-Encoding客戶端支援的壓縮演算法，如 gzip, deflate, br（br = Brotli，比 gzip 壓縮率高 20%）
Accept-Language客戶端偏好語言，如 zh-TW, zh;q=0.9, en-US;q=0.8
Content-TypeBody 的媒體類型，如 application/json、multipart/form-data、text/html; charset=utf-8
Content-EncodingBody 使用的壓縮格式，如 gzip，客戶端需解壓後讀取

認證與授權標頭

標頭說明
Authorization攜帶身份憑證。格式：Bearer <JWT token>、Basic <base64(user:pass)>、Digest ...
WWW-Authenticate回應 401 時告知客戶端所需的認證方式，如 WWW-Authenticate: Bearer realm="api"
Cookie客戶端發送儲存的 Cookie。Cookie: session=abc123; user_id=42
Set-Cookie伺服器設定 Cookie。重要屬性：HttpOnly（JS 無法存取）、Secure（僅 HTTPS）、SameSite（CSRF 防護）、Max-Age
Section 06

HTTP/1.1 的連線管理

HTTP/1.0 每個請求都需要建立一個新的 TCP 連線， 三向握手的開銷對頁面載入時間影響極大。 HTTP/1.1 引入的持久連線（Persistent Connection / Keep-Alive） 允許在同一 TCP 連線上串行發送多個請求，大幅減少握手次數。

HTTP/1.0 vs HTTP/1.1 vs HTTP/2 連線模型
HTTP/1.0（每次請求建立新連線）：
  TCP SYN → SYN-ACK → ACK → Request → Response → FIN
  TCP SYN → SYN-ACK → ACK → Request → Response → FIN  ← 每次都握手！
  TCP SYN → SYN-ACK → ACK → Request → Response → FIN

HTTP/1.1（持久連線 + 管線化）：
  TCP SYN → SYN-ACK → ACK
  ├── Request 1 → Response 1
  ├── Request 2 → Response 2  ← 串行，必須等上一個完成
  └── Request 3 → Response 3
  FIN（可重用，減少握手）

  問題：HOL（Head-of-Line）Blocking
  Request 1 卡住 → Request 2、3 全部等待

HTTP/2（多路復用，同一連線）：
  TCP SYN → SYN-ACK → ACK
  ├── Stream 1：Request 1 → Response 1 ─┐
  ├── Stream 3：Request 2 → Response 2 ─┤ 並行傳輸，互不阻塞
  └── Stream 5：Request 3 → Response 3 ─┘
  FIN

HTTP/1.1 引入了管線化（Pipelining）， 允許客戶端在收到前一個回應之前就發出下一個請求， 但回應必須按請求順序返回， 第一個慢請求仍會阻塞後面所有請求——這就是 隊頭阻塞（Head-of-Line Blocking，HOL Blocking）問題。 實際上大多數瀏覽器預設不啟用 Pipelining， 而是以「一個域名開最多 6 個並行 TCP 連線」的方式繞過 HOL Blocking。

域名分片（Domain Sharding）的反模式

在 HTTP/1.1 時代，工程師常把靜態資源分散在多個子域名 （如 static1.example.com、static2.example.com）， 以突破每個域名最多 6 個連線的限制，這種技巧稱為「域名分片」。 但在 HTTP/2 時代，域名分片是反模式—— 它讓多路復用的好處消失，反而增加 DNS 查詢與 TLS 握手的開銷。 現代最佳實踐是合併到同一個域名並啟用 HTTP/2。

Section 07

HTTP/2 的多路復用

HTTP/2 最核心的改進是引入了二進位框架層（Binary Framing Layer）， 將 HTTP 訊息分解為更小的幀（Frame）， 每個幀屬於某個串流（Stream）， 多個串流可以在同一 TCP 連線上交錯傳輸。

HTTP/2 核心概念說明
Frame（幀） HTTP/2 的最小傳輸單位。類型包含 HEADERS、DATA、PRIORITY、RST_STREAM、SETTINGS、PUSH_PROMISE、PING、GOAWAY、WINDOW_UPDATE
Stream（串流） 一個請求/回應對的邏輯通道，有唯一的 Stream ID（客戶端用奇數，伺服器用偶數）。多個 Stream 同時進行即為多路復用
HPACK 標頭壓縮 使用靜態表（61 個常用標頭）、動態表（之前傳過的標頭）和 Huffman 編碼，將重複標頭的體積減少 85–88%。解決 HTTP/1.1 每個請求都重複傳送相同標頭的浪費
Server Push 伺服器在客戶端請求 HTML 之前，主動推送 CSS/JS，減少客戶端等待時間。但實際部署效果有限，HTTP/3 中已不再強調
Stream Priority 客戶端可為串流設定優先級（如 CSS > JS > 圖片），讓伺服器依優先順序配置頻寬

HTTP/2 的殘留問題

HTTP/2 在應用層解決了 HOL Blocking，但底層 TCP 的 HOL Blocking 依然存在： 若一個 TCP 封包遺失，整個 TCP 連線的所有串流都必須等待該封包重傳， 即使其他串流的資料早已到達緩衝區。這在高丟包環境（如行動網路）中 甚至讓 HTTP/2 比多個 HTTP/1.1 連線更慢。HTTP/3 透過換掉 TCP 為 QUIC 解決了這個問題。

Section 08

HTTP/3 與 QUIC

HTTP/3 將傳輸層從 TCP 換成 QUIC（UDP 基礎）， 徹底解決了 TCP HOL Blocking，同時整合 TLS 1.3， 大幅降低連線建立延遲。 （QUIC 的詳細機制已在 Day 8 介紹，此處聚焦 HTTP/3 層面的改進。）

面向HTTP/2 + TLS (TCP)HTTP/3 (QUIC)
傳輸層TCP（作業系統核心控制）QUIC（UDP，使用者空間實作）
連線建立1 RTT TCP + 1 RTT TLS = 2 RTT1 RTT（初次）或 0-RTT（再次）
HOL BlockingTCP 層仍有徹底消除（QUIC Stream 各自獨立）
連線遷移換網路需重建連線Connection ID 維持連線
標頭壓縮HPACKQPACK（HPACK 的 QUIC 適配版，避免 HPACK 的阻塞問題）
辨識方式TCP Port 80/443UDP Port 443（Alt-Svc 標頭告知客戶端可升級）
HTTP/3 協商流程（Alt-Svc 升級）
  客戶端                              伺服器
    │                                  │
    │── HTTPS (HTTP/1.1 或 HTTP/2) ───>│  首次連線，TCP + TLS
    │                                  │
    │<── Response ──────────────────────│
    │    Headers 含：                  │
    │    alt-svc: h3=":443"; ma=86400  │  「我支援 HTTP/3！」
    │                                  │
    │  （下次連線）                    │
    │── QUIC/HTTP/3 到 UDP:443 ─────→│  直接用 HTTP/3，無需 TCP
Section 09

HTTPS 與 TLS 握手

HTTPS = HTTP + TLS（Transport Layer Security）。 TLS 在 TCP（或 QUIC）之上、HTTP 之下， 提供三項核心保障： 加密（Confidentiality）——中間人無法讀取內容； 完整性（Integrity）——篡改會被偵測； 認證（Authentication）——確認伺服器身份是真實的。

TLS 的當前版本是 TLS 1.3（RFC 8446，2018）， 相比 TLS 1.2 大幅簡化了握手（從 2 RTT 降至 1 RTT）， 移除了所有不安全的密碼套件，並支援 0-RTT Session Resumption。 TLS 1.0 和 1.1 已被 IETF 正式棄用（RFC 8996）。

TLS 1.3 握手流程（1 RTT）
1
Client Hello
客戶端發送：支援的 TLS 版本、支援的密碼套件清單（Cipher Suites）、隨機數（Client Random）、Key Share（用於 ECDHE 金鑰交換的公鑰，TLS 1.3 預先傳送以節省一個 RTT）
2
Server Hello + Certificate + Finished
伺服器回應：選定的密碼套件、隨機數（Server Random）、Key Share（伺服器的 ECDHE 公鑰）、數位憑證（含公鑰與 CA 簽名）、Finished（已用協商好的金鑰加密，證明伺服器擁有私鑰）。此時雙方已可計算出對稱金鑰。
3
客戶端驗證憑證
客戶端驗證：憑證鏈（Chain of Trust）是否由受信任的 CA 簽發、憑證未過期、CN/SAN 欄位與域名相符、憑證未被吊銷（OCSP 查詢）。若驗證失敗，顯示「您的連線不是私人連線」警告。
4
Client Finished
客戶端發送 Finished 訊息，確認握手完成。之後所有 HTTP 通訊都用協商好的對稱金鑰（AES-GCM 或 ChaCha20-Poly1305）加密傳輸。
5
加密 HTTP 通訊開始
整個 HTTP 交換（包含 URL、Header、Body）都被加密。中間人只能看到「與 IP X.X.X.X 的 TLS 連線」，無法得知請求的具體內容（SNI 標頭除外，但 ECH 可加密它）。
TLS 1.2 vs TLS 1.3 握手延遲比較
TLS 1.2（2 RTT 握手）：
  Client Hello ──────────────────────→
  ←──────── Server Hello + Cert + Server Hello Done
  Client Key Exchange + Change Cipher + Finished ─→
  ←──────── Change Cipher Spec + Finished
  ←──────── HTTP Response（第 3 RTT 才收到！）

TLS 1.3（1 RTT 握手）：
  Client Hello + Key Share ──────────→
  ←──────── Server Hello + Key Share + Cert + Finished
  Client Finished + HTTP Request ────→
  ←──────── HTTP Response（第 2 RTT 收到！）

TLS 1.3 0-RTT（Session Resumption）：
  Client Hello + Early Data (HTTP Request) ──→  ← 直接帶請求！
  ←──────────────────── HTTP Response（第 1 RTT 收到）
Section 10

憑證體系與 PKI

TLS 的認證機制依賴公鑰基礎建設（PKI，Public Key Infrastructure）。 每個 HTTPS 網站都有一張由憑證授權中心（CA，Certificate Authority） 簽發的數位憑證（X.509 Certificate）， 瀏覽器信任的根 CA 清單（Root Store）預先內建在作業系統或瀏覽器中。

憑證鏈（Certificate Chain）

三層憑證鏈結構
  根 CA（Root CA）
  自簽名，離線保存，極少使用
  例：DigiCert Global Root G2
      │
      │ 簽發
      ↓
  中間 CA（Intermediate CA）
  由根 CA 簽名，線上運作
  例：DigiCert TLS RSA SHA256 2020 CA1
      │
      │ 簽發
      ↓
  終端憑證（End-Entity / Leaf Certificate）
  你的網站憑證，含域名、公鑰、有效期
  例：www.example.com（有效期至 2025-12-31）

  瀏覽器驗證：
    www.example.com 憑證 → 由 DigiCert 中間 CA 簽名 → 由受信任根 CA 簽名 ✓

憑證類型

類型驗證程度取得方式適用場景
DV（Domain Validation）只驗證域名所有權（DNS 或 HTTP 驗證）自動化，數分鐘內完成（Let's Encrypt）個人網站、API、博客
OV（Organization Validation）驗證域名 + 組織合法性（人工審查）數天，需提供公司文件企業網站、B2B 服務
EV（Extended Validation）最嚴格，驗證組織、地址、法律地位數週，費用高銀行、政府、金融機構
萬用字元（Wildcard）覆蓋 *.example.com 所有子域通常為 OV/DV 等級多子域名的服務
多域名（SAN/Multi-Domain）一張憑證涵蓋多個不同域名（SAN 欄位）依主憑證等級管理多個域名的組織

Let's Encrypt 與憑證自動化

Let's Encrypt 是非營利的免費 CA， 透過 ACME 協定（RFC 8555）自動化 DV 憑證的申請與續期， 大幅推動了全球 HTTPS 採用率（2014 年前後全球 HTTPS 比例約 30%， 2024 年已超過 85%）。 配合 Certbot 或 Caddy，憑證申請與每 90 天自動續期全程無人工介入。

HSTS：強制 HTTPS

HSTS（HTTP Strict Transport Security） 透過回應標頭 Strict-Transport-Security: max-age=31536000; includeSubDomains 告訴瀏覽器：在 max-age 期間，此域名及所有子域名只能用 HTTPS 連線， 任何 HTTP 嘗試都自動升級為 HTTPS，防止 SSL Stripping 攻擊。 加入 HSTS Preload 清單（chromium.org/hsts） 讓瀏覽器從第一次訪問就強制 HTTPS，無需先收到標頭。

Section 11

HTTP 安全標頭與常見漏洞

重要安全回應標頭

標頭作用建議值
Content-Security-Policy 指定哪些來源的腳本/樣式/圖片可以載入，防止 XSS 和資源注入攻擊 default-src 'self'; script-src 'self' cdn.example.com
X-Content-Type-Options 防止瀏覽器 MIME 嗅探（把 image/jpeg 執行為 script） nosniff
X-Frame-Options 防止頁面被嵌入 iframe（Clickjacking 防護），CSP 的 frame-ancestors 是更現代的替代 DENY 或 SAMEORIGIN
Referrer-Policy 控制 HTTP Referer 標頭發送的資訊量，防止敏感 URL 洩漏給第三方 strict-origin-when-cross-origin
Permissions-Policy 限制頁面可使用的瀏覽器 API（Camera、Microphone、Geolocation 等） camera=(), microphone=(), geolocation=(self)

CORS：跨來源資源共用

CORS（Cross-Origin Resource Sharing）是瀏覽器的同源政策（Same-Origin Policy）的延伸機制， 允許伺服器明確宣告哪些來源可以跨域存取其資源。 關鍵回應標頭：

CORS 預檢（Preflight）流程
瀏覽器（在 app.example.com）              API（api.other.com）
    │                                         │
    │── OPTIONS /data ───────────────────────>│  預檢請求
    │   Origin: https://app.example.com       │
    │   Access-Control-Request-Method: POST   │
    │   Access-Control-Request-Headers: Authorization
    │                                         │
    │<── 204 No Content ──────────────────────│  伺服器授權
    │    Access-Control-Allow-Origin: https://app.example.com
    │    Access-Control-Allow-Methods: GET, POST, PUT
    │    Access-Control-Allow-Headers: Authorization
    │    Access-Control-Max-Age: 86400        │  快取預檢 24 小時
    │                                         │
    │── POST /data (實際請求) ───────────────>│
    │<── 200 OK ──────────────────────────────│

常見 HTTP 層安全漏洞

漏洞說明與防護
XSS（跨站腳本） 攻擊者注入惡意 JavaScript 到頁面。防護：輸出編碼（HTML Escaping）+ CSP 標頭 + HttpOnly Cookie
CSRF（跨站請求偽造） 誘使已登入用戶點擊惡意連結，觸發非預期操作。防護：SameSite Cookie + CSRF Token + Double Submit Cookie
SQL Injection 惡意 SQL 語句注入 HTTP 參數，操控資料庫。防護：參數化查詢（Prepared Statement）+ 最小化 DB 權限
HTTP Header Injection 在 Set-Cookie / Location 等標頭中注入換行（CRLF），偽造額外標頭或分割回應。防護：嚴格驗證並過濾換行字元
SSL Stripping 中間人將 HTTPS 降級為 HTTP。防護：HSTS + HSTS Preload 清單
Day 10 重點回顧

今天你學到了什麼

  • HTTP 從 0.9 的單行指令演進到 HTTP/3，每版解決上一版的連線效率瓶頸
  • HTTP 訊息由起始行、Header、空行、Body 組成；URL 含 scheme/host/port/path/query/fragment
  • GET/HEAD/OPTIONS 是安全冪等方法；PUT/DELETE 是冪等但非安全；POST 既非安全也非冪等
  • 2xx=成功、3xx=重導（301 永久/307 臨時保留方法）、4xx=客戶端錯誤、5xx=伺服器錯誤
  • HTTP/1.1 持久連線改善握手開銷，但 Pipelining 的 HOL Blocking 問題依然存在
  • HTTP/2 引入二進位幀、多路復用（Stream）、HPACK 壓縮，解決應用層 HOL Blocking
  • HTTP/3 換用 QUIC（UDP），徹底消除 HOL Blocking，0-RTT 重連，Connection ID 遷移
  • HTTPS = HTTP + TLS；TLS 1.3 只需 1 RTT 握手（TLS 1.2 需 2 RTT）
  • 憑證鏈：根 CA → 中間 CA → 終端憑證；Let's Encrypt ACME 協定實現免費自動化 DV 憑證
  • HSTS 防 SSL Stripping；CSP 防 XSS；SameSite Cookie 防 CSRF；CORS 管控跨域存取
