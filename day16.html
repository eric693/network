Section 02

IPv6 的現在與未來

IPv4 只有約 43 億個地址，早在 2011 年 IANA 就已分配完畢。 IPv6 提供 2¹²⁸ ≈ 340 澗個地址——地球上每粒沙子都能分配到數兆個 IP。 然而 IPv6 的部署速度遠比預期緩慢，主要原因是 NAT 延長了 IPv4 的壽命， 以及雙棧（Dual Stack）過渡期的複雜性。

面向現況（2025）
全球 IPv6 採用率Google 統計約 45–50%（已啟用 IPv6 的用戶比例）。台灣約 50%，美國約 55%，印度約 70%（行動網路主導推動）
主要驅動力行動網路（4G/5G 設備預設 IPv6）；雲端平台（AWS/GCP/Azure 原生雙棧）；ISP 強制部署（CGN 成本高昂，直接上 IPv6 更經濟）
IPv6 特有功能SLAAC（無狀態地址自動配置，不需要 DHCP）；強制 IPsec 支援（雖非強制使用）；Flow Label（可用於 QoS 和負載均衡）；無廣播，用 Multicast 替代
NAT64 / DNS64過渡技術：讓 IPv6-only 設備存取 IPv4 服務。合成 IPv6 地址（DNS64 將 A 記錄合成 AAAA），NAT64 在出口做翻譯
剩餘挑戰部分企業中間設備（防火牆、負載均衡器）的 IPv6 支援不完整；企業內部 IPv6 規劃複雜；安全工具對 IPv6 的可見性仍不如 IPv4

IPv6 地址規劃最佳實踐

IPv6 不需要再像 IPv4 一樣「節約」地址。企業應為每個子網路分配 /64， 每個站點分配 /48（65,536 個 /64 子網路）。 全球可路由前綴從 ISP 取得 /32–/48，自行再細分。 不要試圖把 IPv6 當 IPv4 用——慷慨分配地址才是正確姿態。

Section 03

QUIC 與 HTTP/3：傳輸層革命

TCP 設計於 1974 年，植根於作業系統核心——這意味著每次改進都需要全球數十億台設備更新核心， 迭代速度極慢。Google 在 2012 年開始實驗 QUIC， 用一個執行在 UDP 之上的應用層協定取代 TCP+TLS+HTTP/2 的整體， 徹底繞開了核心更新的瓶頸。2021 年 IETF 將 QUIC 標準化（RFC 9000）， HTTP/3（RFC 9114）在 QUIC 之上取代了 HTTP/2。

TCP + TLS + HTTP/2 vs QUIC + HTTP/3 的協定棧對比
  傳統 HTTPS（HTTP/2）                HTTP/3（QUIC）
  ┌─────────────────────┐            ┌──────────────────────────┐
  │      HTTP/2         │            │         HTTP/3            │
  ├─────────────────────┤            ├──────────────────────────┤
  │      TLS 1.3        │            │  QUIC（含內建 TLS 1.3）   │
  ├─────────────────────┤            ├──────────────────────────┤
  │      TCP            │            │         UDP               │
  └─────────────────────┘            └──────────────────────────┘

  連線建立延遲：
  TCP 3-way handshake：  1 RTT（SYN → SYN-ACK → ACK）
  TLS 1.3 handshake：   +1 RTT（合計 2 RTT）
  QUIC 首次連線：         1 RTT（握手合併）
  QUIC 再次連線（0-RTT）：0 RTT（用快取 Session Ticket 直接傳資料）

  隊頭阻塞（Head-of-Line Blocking）：
  HTTP/2 over TCP：TCP 層有 HoL，一個封包遺失停頓所有串流
  HTTP/3 over QUIC：每個串流獨立，一個遺失不影響其他串流

QUIC 的核心優勢

特性說明
連線遷移（Connection Migration）QUIC 連線以 Connection ID 識別，而非 4-tuple（IP+Port）。手機從 Wi-Fi 切換到 5G 時，IP 改變但 QUIC 連線不中斷——對 VPN 和行動應用體驗大幅提升
內建加密（Always Encrypted）QUIC 沒有未加密模式，TLS 1.3 是協定的必要組成部分，比 TCP 上可選加密更安全
多路複用無 HoL 阻塞多個 HTTP/3 請求在不同 QUIC 串流上並行，一個串流遺失不阻塞其他串流（解決了 HTTP/2 在不穩定網路上的主要問題）
用戶空間實作QUIC 在應用層（user space）實作，更新不需要改核心，迭代速度可以像 Web 服務一樣快（Google Chrome、Facebook、Cloudflare 都有自己的 QUIC 實作）
流量控制更細緻QUIC 同時有串流級（Stream-level）和連線級（Connection-level）的流量控制，比 TCP 的單一視窗更靈活

截至 2025 年，Google、Facebook、Cloudflare、Akamai 的流量有超過 30% 走 HTTP/3， 並且快速增長。主要瀏覽器（Chrome、Firefox、Safari、Edge）全面支援 HTTP/3。 主要 CDN 和雲端供應商也都已支援 HTTP/3，對終端用戶透明升級。

Section 04

量子通訊與後量子密碼學

量子計算對現代網路安全的威脅是真實的—— 但威脅並非「量子電腦今天就能破解你的加密」， 而是一個更微妙的時間軸問題：

「先收集，後解密」的威脅（Harvest Now, Decrypt Later）

敵對組織現在就在大規模擷取和儲存加密通訊， 等待未來的量子電腦解密。 這意味著今天傳輸的機密資料（政府文件、商業秘密、個人資料）， 若使用 RSA 或 ECDH 加密，在未來量子電腦成熟時可能被解密。 這就是為什麼後量子密碼學的遷移必須現在開始，而非等量子電腦出現再行動。

Shor 演算法：為什麼 RSA/ECC 會被破解

RSA 的安全性基於大整數分解的計算難度，ECC 基於橢圓曲線離散對數問題。 這兩個問題對傳統電腦是指數級難度，但 Shor 演算法（1994）可以在量子電腦上以多項式時間解決， 從根本上瓦解了它們的安全基礎。 當一台足夠大的量子電腦（需要數千個邏輯量子位元）出現時， 目前所有基於這兩個問題的非對稱加密系統都將不再安全。

NIST 後量子密碼學標準（2024 年正式發布）

演算法用途基礎數學問題現況
ML-KEM（CRYSTALS-Kyber）金鑰封裝（Key Encapsulation，取代 RSA/ECDH 金鑰交換）格密碼學（Lattice）的 Module-LWE 問題FIPS 203，正式標準，TLS 1.3 已開始整合
ML-DSA（CRYSTALS-Dilithium）數位簽章（取代 RSA / ECDSA 簽章）格密碼學 Module-LWE / SISFIPS 204，正式標準
SLH-DSA（SPHINCS+）數位簽章（無狀態雜湊型）雜湊函數安全性（不依賴格問題）FIPS 205，正式標準，適合高保證場景
FN-DSA（FALCON）數位簽章，簽章體積小NTRU 格密碼學即將正式標準化，適合頻寬受限場景

量子金鑰分發（QKD）

QKD（Quantum Key Distribution）是另一條路線： 利用量子力學的測量即干擾原理（海森堡不確定性）， 讓任何竊聽行為都會在通道中留下可偵測的痕跡， 從物理原理上保證金鑰分發的安全性（BB84 協定，1984）。 中國「墨子號」衛星已實現洲際 QKD 實驗。 不過 QKD 只解決金鑰分發問題，仍需配合後量子密碼學， 且目前成本極高、距離有限，短期內難以大規模商用。

Section 05

AI 驅動的網路自動化

傳統網路管理高度依賴人工：工程師登入設備執行 CLI、手動分析日誌、 根據經驗判斷故障原因。隨著網路規模和複雜度的爆炸性增長， 人工操作已經成為瓶頸，AI 和機器學習正在進入每一個網路管理環節。

應用場景說明
意圖驅動網路（Intent-Based Networking，IBN） 網路工程師用自然語言或高階策略描述「意圖」（如「確保視訊會議流量延遲 <20ms」），AI 系統自動翻譯為底層配置並持續驗證合規。Cisco DNA Center、Juniper Apstra 是主要產品
異常偵測與 AIOps 傳統基於規則的告警會產生大量誤報。ML 模型學習正常流量模式，偵測統計上的異常（如某個流量突增 3σ），關聯多個訊號自動判斷根因（Root Cause Analysis），減少 MTTR
預測性維護 分析設備遙測數據（錯誤率、溫度、光功率），預測光模組或線路失效，在故障發生前主動遷移流量和更換硬體
動態流量工程 AI 即時分析全網流量模式，動態調整 BGP 路徑和 MPLS TE 標籤，比人工調整快幾個數量級。Google B4、Facebook Fabric 使用 AI 管理骨幹流量
自動化安全響應（SOAR） Security Orchestration, Automation and Response：AI 分析安全事件並自動執行響應（如自動隔離受感染主機、更新防火牆規則、通知相關人員），將響應時間從小時縮短到秒
LLM 輔助網路診斷 大型語言模型（如 Cisco 的 AI Assistant）理解自然語言問題，分析設備日誌和設定，回答「為什麼 BGP peer 掉了？」並給出修復建議，降低初階工程師的技術門檻

AI 網路的挑戰：可解釋性與安全性

AI 自動修改網路配置帶來新的風險：模型的決策難以解釋（為什麼它選擇這條路由？）， 訓練資料的偏差可能導致生產事故，AI 系統本身也是攻擊面（對抗樣本 Adversarial Examples 可以欺騙流量分類模型）。 人類工程師的監督和審核機制在可預見的未來仍是必要的。

Section 06

網際網路治理與數位主權

技術與地緣政治的交匯正在重塑全球網際網路的架構。 曾經被視為「無國界」的網路，正在沿著國家邊界碎片化。

網際網路碎片化（Splinternet）的趨勢

面向說明
BGP 路由政治化國家級防火牆（中國 GFW）、BGP 路由異常（俄羅斯曾試圖將流量繞到國內節點）、海底電纜的地緣戰略競爭（誰擁有連接哪些大陸的海底電纜）
資料在地化法規GDPR（歐盟）、中國《資料安全法》、印度 DPDP 法案——各國要求公民資料必須儲存在境內，直接影響雲端架構（需要在地 Region）和跨境資料傳輸（SCCs、BCR）
DNS 管控部分國家在國家級 DNS 層封鎖或過濾域名；DNS over HTTPS（DoH）和 DNS over TLS（DoT）加密 DNS 查詢，繞過 ISP 監控，但也讓國家防火牆更難分析流量（雙刃劍）
數位主權基礎設施歐盟 GAIA-X 計畫推動歐洲自主雲端基礎設施；各國政府加速建設本土 CDN 和 IXP（網際網路交換點），降低對美國科技公司基礎設施的依賴

關鍵海底電纜基礎設施

全球約 97% 的國際網路流量透過海底電纜傳輸（衛星只處理極小比例的頻寬）。 目前全球有超過 500 條海底電纜，總長超過 140 萬公里。 主要的威脅包括：船錨切斷（大多數電纜故障的原因）、 自然災害（地震、海底山崩）、以及愈來愈被討論的蓄意破壞。 2022 年北溪天然氣管道爆炸和 2024 年波羅的海電纜切斷事件， 讓水下關鍵基礎設施安全成為各國戰略議題。

Section 07

衛星網路：SpaceX Starlink 與低軌道革命

傳統地球靜止軌道衛星（GEO，距地面 36,000 公里）有一個根本性的問題： 訊號往返延遲約 600ms，對即時應用完全不可用。 低地球軌道（LEO，距地面 200–2,000 公里）衛星可將延遲降至 20–40ms， 接近地面寬頻網路的水準。

衛星網路軌道衛星數量延遲現況
Starlink（SpaceX）LEO（340–570 km）6,000+ 顆（2024），目標 42,000 顆20–40ms全球最大 LEO 衛星星座，已在超過 100 個國家提供服務，下載速率 50–200 Mbps。烏克蘭戰爭中展示了抗干擾韌性
Project Kuiper（Amazon）LEO（590–630 km）目標 3,236 顆~20–35ms2024 年開始發射，計畫與 AWS 深度整合
OneWeb（英國）LEO（1,200 km）648 顆~32ms2023 年完成部署，主要面向企業和政府市場
O3b mPower（SES）MEO（8,000 km）11 顆（2024）~100ms針對海事和企業市場，高頻寬（數 Gbps）

LEO 衛星網路的挑戰

LEO 衛星的高速運動（每 90 分鐘繞地球一圈）意味著需要大量衛星才能提供連續覆蓋， 也意味著訊號需要在多顆衛星之間切換（Handoff）。 Starlink Gen 2 衛星使用星間雷射鏈路（ISL）在衛星之間直接傳輸資料， 跨洲際通訊可能比地面光纖更快（因為光在真空中比在光纖中更快）。 長期挑戰：太空垃圾（Kessler Syndrome 風險）、天文觀測干擾（光污染）、 頻率協調（ITU 規範有限的頻譜資源）。

Section 08

網路設計的不變原則

技術浪潮一波接一波，從 Frame Relay 到 MPLS，從 SDN 到 SASE， 每隔幾年就有「改變一切的新典範」。 然而在這些變化之下，有幾個設計原則幾十年來從未動搖：

原則 01
分層是複雜度的解藥
OSI 模型之所以歷久彌新，是因為分層讓每一層可以獨立演化而不影響其他層。TCP 不需要知道底層是 Wi-Fi 還是光纖；應用程式不需要知道 TLS 的細節。每次技術革命本質上都是在某一層引入更好的抽象。
原則 02
端到端原則（End-to-End Principle）
網路核心應保持簡單和「啞巴」，複雜的功能應在端點（終端設備）實作。這是互聯網能夠爆炸性成長的根本原因——任何新應用都可以部署在端點，不需要修改網路核心。NAT 和深度封包檢測（DPI）是對這個原則的違背，帶來了已知的問題。
原則 03
設計容錯，而非設計無錯
沒有任何硬體或軟體是完美的。好的網路設計假設元件「必然」故障，設計多路徑、冗餘和自動恢復機制，確保單點故障不影響整體服務。「平均無故障時間（MTBF）」沒有「平均恢復時間（MTTR）」更重要。
原則 04
安全是架構，而非功能
「後來加上去」的安全永遠比不上「設計進去」的安全。TLS、IPsec、WPA3 都是在舊協定吃過虧後補強的例子。Zero Trust 架構的核心洞察是：不能假設任何網段是安全的，必須對每個請求進行驗證和最小授權。
原則 05
可觀測性讓不可見變可見
你無法管理你無法量測的東西。從 RFC 792 的 ICMP（1981）到現代的 OpenTelemetry，可觀測性是排查問題和驗證改進的基礎。現代系統的複雜性讓「主動監控」（Proactive Observability）比「事後診斷」（Reactive Debugging）更加重要。
原則 06
簡單勝於複雜
WireGuard 以 4,000 行程式碼做到了 OpenVPN 100,000 行才能做到的事。更少的程式碼意味著更少的 bug、更容易審計、更快的效能。每一個額外的複雜度都是未來的技術債。在能用簡單方案解決的問題上，複雜方案不是進步，是負擔。
Section 09

繼續學習的路徑

十六天是一個開始，不是終點。以下按職涯方向整理了深化路徑：

網路工程師 / 基礎設施
深入路由交換、資料中心設計和 WAN 技術。掌握動手操作能力（GNS3、EVE-NG 模擬器），取得廠商認證。
認證路徑：CCNA → CCNP → CCIE
或 Juniper JNCIA → JNCIP → JNCIE
雲端架構師
深化 AWS/GCP/Azure 的進階網路服務（Transit Gateway、PrivateLink、Network Firewall）。練習 Terraform、CDK 基礎設施即程式碼。
認證路徑：AWS SAA → AWS ANS-C01（進階網路）
或 GCP PCNE / Azure Network Engineer
網路安全工程師
深入滲透測試（網路層）、防火牆策略設計、SIEM 規則開發和事件響應。熟練 Wireshark、Nmap、Burp Suite 等工具。
認證路徑：CompTIA Security+ → CEH / OSCP
或 CISSP（管理層）/ PNPT（實務）
SRE / 平台工程師
深化 Kubernetes 網路（CNI、Service Mesh）、可觀測性（Prometheus/Grafana/Jaeger）和混沌工程。強調自動化和基礎設施即程式碼。
認證路徑：CKA（Kubernetes Admin）→ CKS（Security）
或 CKAD（Application Developer）

資源類型適合對象

最有效的學習方法：動手建造

閱讀和觀看永遠無法取代親手操作。建議的實踐項目： 用 GNS3 或 EVE-NG 模擬 BGP 多 AS 網路； 在 Raspberry Pi 上建立家用 PiHole DNS + WireGuard VPN； 用 Mininet 實驗 SDN OpenFlow 流表； 在 AWS 上建立包含 ALB、VPC Peering 和 Transit Gateway 的多 VPC 架構； 用 Wireshark 擷取並分析 TLS 握手和 QUIC 建立流程。 每一個「我自己做過」的技術，理解深度都勝過一百篇文章。

Day 16 / 16  ·  課程完結

網路是人類有史以來建造的最複雜的機器，
也是讓世界運轉的隱形基礎。

從第一天的電壓訊號到今天的後量子密碼學，你走過了一段從 實體層電纜到衛星星座、從 TCP 三向握手到 5G 網路切片的完整旅程。 這些知識不會因為技術更迭而過時——因為我們學的不只是工具， 而是網路思維：分層抽象、容錯設計、安全前置、可觀測性。

網路工程最迷人之處在於，它是理論與實踐的完美結合。 每一個 RFC 背後都是真實工程師面對真實問題的智慧結晶； 每一次封包捕獲都能讓抽象的協定變得具體可感。 繼續保持好奇心——每當你連上 Wi-Fi、打開網頁、撥出 VoIP 電話， 都有無數層次的工程奇蹟在毫秒間發生。
