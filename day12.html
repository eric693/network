Section 01

系統性診斷方法論

遇到網路問題時，最常見的錯誤是跳過分析直接嘗試解決方案， 結果花了一小時「試」各種方法卻沒有進展。 系統性診斷的核心思想是：先定位問題在哪一層、哪個環節，再針對性修復。

OSI 分層診斷法（由下往上）

分層排查流程：「能上網嗎？」的完整診斷路徑
  L1 — 實體層
    ├── 網路線插好了嗎？燈號是否正常？
    ├── Wi-Fi 訊號強度是否足夠？（iwconfig / Airport Utility）
    └── 確認：實體連線正常
         ↓
  L2 — 資料鏈路層
    ├── ip link show — 介面是否 UP？
    ├── 是否有 IP 位址？（ip addr 或 ipconfig）
    └── ARP 表是否有閘道的 MAC？（arp -a 或 ip neigh）
         ↓
  L3 — 網路層
    ├── ping 閘道（如 192.168.1.1）— 能到閘道嗎？
    ├── ping 8.8.8.8 — 能到網際網路 IP 嗎？
    ├── traceroute 8.8.8.8 — 封包在哪一跳卡住？
    └── ip route show — 路由表是否正確？
         ↓
  L4 — 傳輸層
    ├── telnet host 443 / nc -zv host 443 — 連接埠是否開放？
    ├── ss -tnp — 有沒有建立連線？
    └── netstat -s — 是否有大量 TCP 重傳？
         ↓
  L7 — 應用層
    ├── curl -v https://example.com — HTTP/TLS 是否正常？
    ├── dig example.com — DNS 是否正確解析？
    └── 查看應用程式日誌 — 有無應用層錯誤訊息？

二分法（Binary Search）診斷

面對複雜的路徑問題（如 A 到 Z 之間某跳出問題）， 不要從頭逐一測試，而是先測試中間點。 若中間點通，問題在後半段；若不通，問題在前半段。 持續二分直到定位到具體故障點，效率遠高於線性搜尋。

診斷黃金法則

在開始任何排查之前，先問自己三個問題： （1）這個問題什麼時候開始的？（確認是突然發生還是漸進惡化） （2）這段時間有什麼變更？（更新、設定修改、新設備上線） （3）影響範圍是什麼？（只有你的機器？特定 VLAN？所有用戶？） 這三個問題往往能在 5 分鐘內大幅縮小故障範圍。

Section 02

基礎診斷指令全覽

ping
L3 — ICMP Echo
測試主機可達性與 RTT。可指定封包大小（-s）、次數（-c）、TTL（-t）。
ping -c 10 -s 1400 8.8.8.8
traceroute / tracert
L3 — TTL 遞增探測
顯示封包路徑上每一跳的路由器與延遲。Linux 預設 UDP，Windows 用 ICMP。
traceroute -n 8.8.8.8  |  tracert 8.8.8.8
mtr
L3 — 持續追蹤
結合 ping 和 traceroute，持續更新每跳的封包遺失率與延遲，是排查間歇性路由問題的最佳工具。
mtr --report -n 8.8.8.8
dig / nslookup
L7 — DNS 查詢
查詢 DNS 記錄。+trace 選項從根伺服器開始完整追蹤解析路徑，診斷 DNS 傳播問題。
dig +trace @8.8.8.8 www.example.com A
curl
L7 — HTTP/TLS 測試
測試 HTTP/HTTPS 連線。-v 顯示完整握手過程，-w 輸出各階段計時，-k 忽略憑證錯誤。
curl -v -w "\nTime: %{time_total}s\n" https://example.com
ss / netstat
L4 — 連線狀態
顯示 TCP/UDP socket 狀態。ss 是現代替代品，速度更快。-tnp 顯示 TCP 連線與對應進程。
ss -tnp  |  ss -lnup（監聽的 UDP）
ip / ifconfig
L2/L3 — 介面與路由
查看介面狀態、IP 位址、路由表、ARP 鄰居表。ip 是 ifconfig/route 的現代替代品。
ip addr show  |  ip route show  |  ip neigh
nmap
L3/L4 — 掃描探測
主機探測、埠掃描、服務版本偵測、OS 識別。-sV 偵測服務版本，-sC 執行預設腳本。
nmap -sV -p 1-1000 192.168.1.0/24
tcpdump
L2–L7 — 封包擷取
命令列封包擷取工具。可用 BPF 過濾器精確指定擷取條件，輸出可供 Wireshark 分析。
tcpdump -i eth0 -w cap.pcap 'host 8.8.8.8'
iperf3
L4 — 頻寬測試
精確測量兩點之間的 TCP/UDP 吞吐量、抖動（Jitter）和封包遺失率。需在兩端同時執行。
iperf3 -s（伺服端） |  iperf3 -c <IP>（客戶端）
openssl s_client
L7 — TLS 診斷
建立 TLS 連線並顯示完整憑證鏈資訊。診斷憑證過期、SNI 問題、TLS 版本協商等。
openssl s_client -connect example.com:443 -servername example.com
ethtool
L1/L2 — 實體層診斷
查看網卡速度、雙工模式、連線狀態。可診斷實體層協商問題（如 1Gbps 被協商成 100Mbps）。
ethtool eth0  |  ethtool -S eth0（統計）
curl 計時分解：每個階段耗時多少？Linux / macOS
$ curl -w "
DNS 解析:      %{time_namelookup}s
TCP 連線:      %{time_connect}s
TLS 握手:      %{time_appconnect}s
等待第一 Byte: %{time_starttransfer}s
總計:          %{time_total}s
" -o /dev/null -s https://www.example.com

# 典型輸出：
DNS 解析:      0.012s
TCP 連線:      0.045s
TLS 握手:      0.098s
等待第一 Byte: 0.142s
總計:          0.185s

# 分析：TLS 握手從 TCP 連線後的 0.045s 到 0.098s，耗時 53ms（一次 RTT）
# 這是 TLS 1.3 的正常值；若超過 200ms，需檢查 OCSP 裁決或 DNS 解析
Section 03

封包分析：Wireshark 實戰

Wireshark 是業界標準的 GUI 封包分析工具， 能夠擷取、解碼、過濾和分析所有網路協定的封包。 掌握 Wireshark 的顯示過濾器和統計功能， 是進階網路診斷和安全分析的核心能力。

BPF 擷取過濾器 vs 顯示過濾器

類型語法說明
擷取過濾器（Capture Filter） BPF 語法 在擷取時就過濾，減少 pcap 檔案大小。host 8.8.8.8、port 443、tcp and not port 22
顯示過濾器（Display Filter） Wireshark 語法 擷取後在介面過濾顯示，功能更強大。http.request、tcp.flags.syn == 1、dns.qry.name contains "google"
常用 Wireshark 顯示過濾器備忘錄Wireshark Display Filter
# HTTP/HTTPS
http.request.method == "POST"
http.response.code == 200
ssl.handshake.type == 1        # Client Hello
tls.record.content_type == 21  # TLS Alert（憑證錯誤等）

# TCP 問題診斷
tcp.analysis.retransmission    # 所有重傳封包
tcp.analysis.zero_window       # 零視窗（接收方緩衝區滿）
tcp.analysis.duplicate_ack     # 重複 ACK（潛在丟包）
tcp.flags.reset == 1           # RST 封包（連線異常重置）

# DNS
dns.flags.response == 1 and dns.flags.rcode != 0  # DNS 錯誤回應
dns.qry.type == 28             # AAAA（IPv6 查詢）

# 進階過濾
ip.addr == 192.168.1.0/24      # 某個子網路的所有流量
frame.time_delta > 1           # 相鄰封包時間差超過 1 秒
tcp.stream eq 5                # 只顯示第 5 條 TCP 串流

Wireshark 實用功能

功能位置用途
Follow TCP Stream右鍵選單重建整條 TCP 連線的應用層內容（如完整 HTTP 請求/回應），一鍵查看明文對話
Statistics → I/O GraphStatistics 選單視覺化流量隨時間的變化，識別流量突增或週期性模式
Statistics → TCP Stream GraphsStatistics 選單RTT 圖、吞吐量圖、視窗大小圖，直觀診斷 TCP 效能問題
Statistics → ConversationsStatistics 選單列出所有通訊對端及其流量統計，快速找出最大流量來源
Expert InformationAnalyze → Expert InfoWireshark 自動標示的問題：重傳、零視窗、連線重置等異常
Decryption（TLS 解密）Preferences → RSA Keys 或 SSLKEYLOGFILE若有 Session Key 日誌（Firefox/Chrome 支援），可解密 TLS 流量查看明文
Section 04

網路效能指標

量化網路效能需要幾個核心指標， 理解它們的定義與相互關係， 才能正確解讀監控數據並找出真正的瓶頸。

頻寬（Bandwidth）
bps / Mbps / Gbps
連線的最大資料傳輸速率（理論上限）。1Gbps 乙太網路的頻寬是 1,000 Mbps。注意頻寬 ≠ 速度。
吞吐量（Throughput）
bps（實際值）
實際達到的資料傳輸速率，通常低於頻寬（受協定開銷、丟包、壅塞控制影響）。iperf3 測量的就是吞吐量。
延遲（Latency / RTT）
ms
封包從 A 到 B 再回來的時間（往返）。由傳播延遲、處理延遲、排隊延遲組成。ping 測量 RTT。
抖動（Jitter）
ms
延遲的變異量（不穩定性）。對 VoIP 和視訊會議影響極大，抖動 >30ms 通話品質明顯下降。
封包遺失率（Packet Loss）
%
遺失封包佔總傳送封包的比例。1% 的丟包對 TCP 吞吐量可能造成超過 50% 的下降（壅塞控制降速）。
BDP（頻寬-延遲積）
bytes
BDP = Bandwidth × RTT。代表「飛行中」的最大資料量，決定 TCP 視窗的最優大小。高 BDP 網路需要 Window Scale 選項。

頻寬 vs 延遲：誰是真正的瓶頸？

不同應用對頻寬與延遲的敏感度
  高頻寬需求 ↑
              │  大型檔案下載   4K 影片串流
              │  (頻寬敏感，    (頻寬 + 緩衝
              │   延遲可接受)    延遲較不敏感)
              │
              │  HTTP 網頁載入  即時遊戲
              │  (兩者都重要，  VoIP / 視訊會議
              │   TLS 握手延遲  (低延遲最關鍵，
              │   是主要瓶頸)    頻寬需求低)
              │
  低頻寬需求  └─────────────────────────────→ 高延遲敏感
              低延遲敏感           高延遲敏感

  規律：
    互動式應用（遊戲、VoIP）：延遲 > 頻寬
    大量傳輸（備份、影片）：頻寬 > 延遲
    Web 應用：兩者都重要，但首字節時間（TTFB）通常是關鍵
Section 05

SNMP 與網路設備監控

SNMP（Simple Network Management Protocol） 是監控路由器、交換器、UPS 等網路設備的標準協定， 使用 UDP Port 161（查詢）和 162（Trap 通知）。 大多數網路設備都支援 SNMP，它是傳統 NOC（Network Operations Center）的基礎。

版本安全性認證方式
SNMPv1極低Community String（明文），如 "public"、"private"。已不建議使用
SNMPv2c低仍用明文 Community String，但效能更好。目前最常見，但有安全風險
SNMPv3高使用者認證（MD5/SHA）+ 資料加密（DES/AES）。企業環境應強制使用

MIB 與 OID：SNMP 的資料結構

SNMP 透過OID（Object Identifier）存取設備上的特定資料， 例如 1.3.6.1.2.1.2.2.1.10.1 代表介面 1 的接收位元組數。 MIB（Management Information Base）是 OID 的定義字典， 將數字 OID 映射到人類可讀的名稱（如 ifInOctets）。

SNMP 常用指令Linux / net-snmp
# 查詢設備系統描述
$ snmpget -v2c -c public 192.168.1.1 sysDescr.0

# 遍歷介面統計（Walk）
$ snmpwalk -v2c -c public 192.168.1.1 ifTable

# 常用 OID
sysDescr.0      # 系統描述
ifInOctets      # 介面接收位元組數
ifOutOctets     # 介面傳送位元組數
ifInErrors      # 介面接收錯誤數
cpuUser         # CPU 使用率（廠商自訂 OID）

# 接收 SNMP Trap（設備主動通知）
$ snmptrapd -f -Lo # 前台接收並列印

工具類型特點

Section 06

NetFlow 與流量分析

SNMP 只能告訴你「介面的流量是多少」，但無法告訴你「是誰在佔用頻寬」。 NetFlow（Cisco，RFC 3954）和其開放標準 IPFIX（RFC 7011） 提供流量紀錄（Flow Records）——記錄每一條網路流（五元組：來源 IP/Port、目的 IP/Port、Protocol）的封包數和位元組數。

NetFlow 架構：資料如何從設備到分析平台
  ┌──────────────────────────────────────────────────────┐
  │  路由器 / 交換器（Flow Exporter）                     │
  │  偵測流量 → 聚合成 Flow Records → 定期（如 60 秒）     │
  │  匯出到 Collector                                     │
  └────────────────────┬─────────────────────────────────┘
                       │ UDP（通常 Port 2055 或 9995）
                       ↓
  ┌──────────────────────────────────────────────────────┐
  │  NetFlow Collector（如 ntopng、Elastiflow、Scrutinizer）│
  │  接收、儲存、索引 Flow Records                        │
  └────────────────────┬─────────────────────────────────┘
                       │
                       ↓
  ┌──────────────────────────────────────────────────────┐
  │  分析 / 視覺化介面                                    │
  │  Top Talkers（最大流量來源）                          │
  │  Top Destinations（最多連線目的地）                    │
  │  Protocol Distribution（協定分布）                    │
  │  異常流量偵測（如突然的大量 UDP → 可能是 DDoS）        │
  └──────────────────────────────────────────────────────┘

NetFlow 是容量規劃（Capacity Planning）和安全事件調查的強大工具： 它可以在事後重建「某個 IP 在某段時間的所有對外連線」， 即使沒有完整的封包擷取記錄也能追蹤攻擊源頭。 相比完整封包擷取，NetFlow 的儲存空間需求約為流量的 0.5%， 更適合長期保留。

Section 07

日誌管理與 SIEM

集中式日誌管理

每台設備都有日誌，但分散在各自的本地儲存中， 既難以搜尋跨設備的事件，又有被攻擊者篡改的風險。 集中式日誌管理將所有設備的日誌即時傳送到中央平台。

協定 / 工具說明
Syslog（UDP/TCP 514）Unix/Linux 系統和網路設備的標準日誌協定（RFC 5424）。rsyslog/syslog-ng 可轉發到中央伺服器
Windows Event LogWindows 系統日誌，可透過 WinRM 或 Elastic Agent 轉發到集中平台
ELK StackElasticsearch（儲存搜尋）+ Logstash（解析轉換）+ Kibana（視覺化），開源的日誌分析平台
Loki + GrafanaGrafana Labs 的輕量級日誌聚合方案，不索引日誌內容，儲存成本低

SIEM：安全事件關聯分析

SIEM（Security Information and Event Management） 在集中日誌的基礎上增加關聯分析（Correlation）能力： 將來自防火牆、IDS、VPN、應用程式的事件關聯起來， 偵測單一事件看不出來的攻擊模式。

SIEM 關聯規則範例
  規則：偵測帳號暴力破解後成功登入
  ─────────────────────────────────────────────────────
  條件 1：同一 IP 在 5 分鐘內觸發 ≥ 10 次登入失敗
  AND
  條件 2：同一 IP 在隨後 1 分鐘內有一次登入成功

  → 觸發告警：高嚴重性「帳號暴力破解後成功入侵」
  → 自動動作：鎖定帳號 + 封鎖來源 IP + 通知 SOC

  ─────────────────────────────────────────────────────
  規則：偵測可能的 APT 橫向移動
  ─────────────────────────────────────────────────────
  條件：同一帳號在 30 分鐘內登入 ≥ 5 台不同主機
  AND 這些主機不在同一 VLAN
  AND 時間是非工作時間（22:00–06:00）

  → 觸發告警：中嚴重性「異常橫向移動跡象」

主流 SIEM 平台包含：Splunk（業界標準，功能強大但昂貴）、 Microsoft Sentinel（Azure 雲端 SIEM，整合 M365 生態）、 IBM QRadar（企業級）、 Elastic SIEM（基於 ELK，開源選項）。

Section 08

現代可觀測性：Metrics、Logs、Traces

傳統監控回答「系統是否正常」（Is it up?）， 而可觀測性（Observability）回答「系統為什麼不正常」（Why is it broken?）。 現代分散式系統的複雜度讓傳統監控不再足夠， 可觀測性建立在三個支柱上：

可觀測性三支柱
支柱
定義與特性
主要工具
Metrics（指標）
時間序列的數值資料，如 CPU 使用率、HTTP 錯誤率、P99 延遲。聚合儲存，查詢效率高，適合儀表板與告警。代表：Prometheus 的 Counter、Gauge、Histogram
Prometheus、Datadog、InfluxDB、CloudWatch
Logs（日誌）
帶時間戳的非結構化或半結構化文字事件，提供最豐富的上下文資訊。儲存成本高，但保留了完整的事件細節，適合事後調查
ELK Stack、Loki、Splunk、CloudWatch Logs
Traces（鏈路追蹤）
一個請求在分散式系統中跨多個服務的完整執行路徑。每個 Trace 由多個 Span 組成，每個 Span 代表一個服務操作，含耗時、父子關係。OpenTelemetry 是標準
Jaeger、Zipkin、Tempo、Datadog APM

SLI、SLO、SLA：效能承諾的語言

縮寫全名說明範例
SLI Service Level Indicator 衡量服務品質的具體指標（測量值） 過去 5 分鐘的 HTTP 成功率、P99 延遲
SLO Service Level Objective SLI 的目標值（內部承諾） HTTP 成功率 ≥ 99.9%；P99 延遲 ≤ 200ms
SLA Service Level Agreement 對外承諾的服務等級（違約有懲罰） 月可用性 ≥ 99.9%（每月允許停機 43.8 分鐘）
Error Budget 錯誤預算 SLO 允許「失敗」的額度，用於衡量風險與發布速度 99.9% SLO 剩餘 0.1%（每月 43.8 分鐘）的錯誤預算

可用性的九九法則

99%（兩個九）= 每年停機 87.6 小時； 99.9%（三個九）= 每年停機 8.76 小時； 99.99%（四個九）= 每年停機 52.6 分鐘； 99.999%（五個九）= 每年停機 5.26 分鐘。 每增加一個「九」，工程複雜度和成本通常呈指數增加。

Section 09

常見故障場景排查指南

場景一：「能 ping 通但網頁打不開」

排查步驟Linux / macOS
# 1. 確認 DNS 解析是否正常
$ dig www.google.com A
# 若無回應或 SERVFAIL → DNS 問題

# 2. 用 IP 直接測試
$ curl -v http://142.250.74.68/ -H "Host: www.google.com"
# 若 IP 可通但域名不通 → 確認是 DNS 問題

# 3. 測試 TCP 連線（Port 80/443）
$ nc -zv www.google.com 443
# Connection refused → 防火牆阻擋或服務未啟動

# 4. 完整 HTTP 測試含計時
$ curl -vso /dev/null https://www.google.com 2>&1 | head -50
# 觀察是哪個階段失敗：DNS / TCP / TLS / HTTP

場景二：「網路突然變慢」

排查步驟Linux
# 1. 確認是整體慢還是特定目的地慢
$ mtr --report -c 100 8.8.8.8
# 觀察哪一跳開始出現高延遲或丟包

# 2. 確認本地頻寬是否飽和
$ iftop -n                     # 即時查看介面流量 Top
$ nethogs                      # 按進程查看頻寬使用

# 3. 確認是否有大量 TCP 重傳
$ netstat -s | grep retransmit
# RetransSegs 數字快速增長 → 丟包造成重傳

# 4. 確認網卡錯誤統計
$ ethtool -S eth0 | grep -i error
# rx_errors / tx_errors 增長 → 實體層問題

場景三：「TLS 憑證相關錯誤」

排查步驟Linux / macOS
# 查看完整憑證鏈資訊
$ openssl s_client -connect example.com:443 -servername example.com

# 常見問題診斷：
# 1. 憑證已過期（Verify return code: 10 - certificate has expired）
$ openssl s_client -connect example.com:443 2>/dev/null | openssl x509 -noout -dates

# 2. 主機名稱不符（ERR_CERT_COMMON_NAME_INVALID）
$ openssl s_client -connect example.com:443 2>/dev/null | openssl x509 -noout -text | grep -A1 "Subject Alternative Name"

# 3. 憑證鏈不完整（中間 CA 未配置）
# 輸出的 verify error: num=20（unable to get local issuer certificate）

# 4. 確認 TLS 版本與密碼套件
$ nmap --script ssl-enum-ciphers -p 443 example.com

場景四：「特定客戶端無法連線」

當問題只影響部分客戶端時，排查重點在於找出差異： IP 位址 / 子網路（路由或 ACL 問題）、 DNS 解析結果不同（GeoDNS 或快取）、 MTU 差異（VPN 客戶端的有效 MTU 更小）、 TLS 版本或密碼套件支援（舊設備不支援 TLS 1.3）、 IPv4 vs IPv6（雙棧環境的路由問題）。 對比正常客戶端和問題客戶端的差異是最快的定位方法。

Day 12 重點回顧

今天你學到了什麼

  • 系統性診斷：OSI 分層由下往上逐層確認，先定位問題在哪層再修復
  • 診斷三問：「什麼時候開始？有什麼變更？影響範圍？」
  • 核心工具鏈：ping/traceroute/mtr（L3）、ss（L4）、dig/curl（L7）、tcpdump/Wireshark（全層）
  • curl -w 可拆解 DNS/TCP/TLS/TTFB 各階段耗時，精確定位 HTTPS 慢在哪裡
  • Wireshark Display Filter：tcp.analysis.retransmission、tcp.flags.reset 等快速定位問題
  • 六個效能指標：頻寬、吞吐量、延遲（RTT）、抖動、封包遺失率、BDP
  • SNMPv3 是唯一安全版本；SNMP + Prometheus/Grafana 是現代監控組合
  • NetFlow/IPFIX 記錄流量五元組，用於找出「誰在佔用頻寬」和事後安全調查
  • SIEM 關聯多來源事件，偵測單事件無法識別的攻擊模式（如暴力破解後成功登入）
  • 可觀測性三支柱：Metrics（趨勢告警）、Logs（事後調查）、Traces（分散式鏈路追蹤）
